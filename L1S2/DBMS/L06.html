📋 Topics Covered

Part 1 Relational Database Design using ER-to-Relational Mapping (Steps 1–7)
Part 2 Relational Database Design using EER-to-Relational Mapping (Steps 8–9)
Context: Mapping is part of the Logical Design (Data Model Mapping) phase in the database design process — it converts a conceptual ER/EER schema into a relational schema specific to a DBMS.

Part 1 — Mapping ER Model Constructs to Relations

Step 1

Mapping of Regular (Strong) Entity Types

  • For each regular (strong) entity type E, create a relation R that includes all simple attributes of E.
  • For composite attributes, include only their simple component attributes (not the composite itself).
  • Choose one key attribute of E as the primary key of R.
  • If the chosen key is composite, the set of its simple components together form the primary key of R.
  • If multiple keys exist for E, additional keys become secondary (unique) keys of R.
  • Relations created this way are called entity relations — each tuple represents one entity instance.
Example — COMPANY Database (Regular Entity Relations):
EMPLOYEE
FnameMinitLnameSsnBdateAddressSexSalary
DEPARTMENT
DnameDnumber
PROJECT
PnamePnumberPlocation
Step 2

Mapping of Weak Entity Types

  • For each weak entity type W with owner entity type E, create a relation R with all simple attributes of W.
  • Include the primary key of the owner as a foreign key in R — this handles the identifying relationship.
  • The primary key of R = primary key(s) of owner(s) + partial key of W (if any).
  • If a weak entity E2 has a weak entity E1 as its owner, map E1 first.
Example: DEPENDENT's primary key = {Essn, Dependent_name} because Dependent_name is the partial key.
DEPENDENT
EssnDependent_nameSexBdateRelationship
Step 3

Mapping of Binary 1:1 Relationship Types

Identify relations S and T for the two participating entity types. Three possible approaches:

Approach A — Foreign Key Choose one relation (preferably the one with total participation) as S, and add the primary key of T as a foreign key in S. Also add simple attributes of the relationship to S.
Approach B — Merged Relation Merge both entity types and the relationship into a single relation. Only suitable when both participations are total (same number of tuples always).
Approach C — Cross-Reference / Relationship Relation Create a third relation R (lookup table) with the primary keys of S and T as foreign keys. Primary key of R = one FK; the other FK becomes a unique key. Drawback: extra join needed.
Example (Foreign Key approach — MANAGES is 1:1):
DEPARTMENT
DnameDnumberMgr_ssnMgr_start_date
Step 4

Mapping of Binary 1:N Relationship Types

  • Identify relation S at the N-side and relation T at the 1-side.
  • Add the primary key of T as a foreign key in S. (Each N-side entity relates to at most one 1-side entity.)
  • Include any simple attributes of the relationship as attributes of S.
Example (WORKS_FOR is 1:N — Employee is N-side, Department is 1-side):
EMPLOYEE
FnameMinitLnameSsnBdateAddressSexSalarySuper_ssnDno
PROJECT
PnamePnumberPlocationDnum
Step 5

Mapping of Binary M:N Relationship Types

  • Create a new relation S to represent the M:N relationship.
  • Include the primary keys of both participating entity relations as foreign keys in S.
  • Their combination forms the primary key of S.
  • Include any simple attributes of the relationship as attributes of S.
  • Cannot use a single foreign key in one of the participating relations (unlike 1:1 or 1:N) — a separate relationship relation is required.
Example (WORKS_ON — M:N between EMPLOYEE and PROJECT):
WORKS_ON
EssnPnoHours
Step 6

Mapping of Multivalued Attributes

  • For each multivalued attribute A, create a new relation R.
  • R includes: an attribute for A + the primary key K of the entity/relationship that owns A (as a foreign key).
  • Primary key of R = combination of A and K.
  • If A is composite, include its simple components.
Example (Locations is a multivalued attribute of DEPARTMENT):
DEPT_LOCATIONS
DnumberDlocation
Step 7

Mapping of N-ary Relationship Types (n > 2)

  • For each n-ary relationship type R (where n > 2), create a new relation S.
  • Include the primary keys of all participating entity types as foreign keys in S.
  • Include any simple attributes of the relationship as attributes of S.
  • Primary key of S = combination of all foreign keys (all participating entity PKs).
  • Exception: If any participating entity type E has cardinality = 1, exclude its FK from the primary key of S.
Example (SUPPLY — ternary relationship among SUPPLIER, PROJECT, PART):
SUPPLY
SnameProj_namePart_noQuantity

Part 2 — Mapping EER Model Constructs to Relations

Step 8

Options for Mapping Specialization or Generalization

For a specialization with m subclasses {S1, S2, ..., Sm} and superclass C with attributes {k, a1, ...an} (k = primary key), choose one of four options:

Option 8A Multiple relations — superclass AND subclasses
Option 8B Multiple relations — subclass relations ONLY
Option 8C Single relation with ONE type attribute
Option 8D Single relation with MULTIPLE type attributes
Option 8A — Multiple Relations: Superclass + Subclasses
  • Create a relation L for the superclass C: Attrs(L) = {k, a1, ..., an}, PK(L) = k.
  • Create a relation Li for each subclass Si: Attrs(Li) = {k} ∪ {attributes of Si}, PK(Li) = k.
  • Each subclass relation references the superclass via the shared key.
  • Works for any specialization — total or partial, disjoint or overlapping.
Example (EMPLOYEE specialization into SECRETARY, TECHNICIAN, ENGINEER):
EMPLOYEE (superclass)
SsnFnameMinitLnameBirth_dateAddressJob_type
SECRETARY
SsnTyping_speed
TECHNICIAN
SsnTgrade
ENGINEER
SsnEng_type
Option 8B — Multiple Relations: Subclasses Only
  • Create a relation Li for each subclass Si: Attrs(Li) = {attributes of Si} ∪ {k, a1, ..., an}, PK(Li) = k.
  • No separate superclass relation — all superclass attributes are duplicated into each subclass relation.
  • Only works when specialization is total (every superclass entity must belong to at least one subclass).
  • Recommended only when the specialization is disjoint. If overlapping, the same entity would be duplicated across multiple relations.
Example (VEHICLE specialization into CAR and TRUCK — total, disjoint):
CAR
Vehicle_idLicense_plate_noPriceMax_speedNo_of_passengers
TRUCK
Vehicle_idLicense_plate_noPriceNo_of_axlesTonnage
Option 8C — Single Relation with One Type Attribute
  • Create a single relation L: Attrs(L) = {k, a1, ..., an} ∪ {all subclass attributes} ∪ {t}, PK(L) = k.
  • The attribute t is called a type (discriminating) attribute — its value indicates which subclass each tuple belongs to.
  • Only works for disjoint specializations.
  • Potential for many NULL values if subclasses have many specific attributes.
Example (EMPLOYEE with disjoint subclasses — single relation, Job_type acts as discriminator):
EMPLOYEE
SsnFnameMinitLnameBirth_dateAddressJob_typeTyping_speedTgradeEng_type
Option 8D — Single Relation with Multiple Type Attributes
  • Create a single relation L: Attrs(L) = {k, a1, ..., an} ∪ {all subclass attributes} ∪ {t1, t2, ..., tm}, PK(L) = k.
  • Each ti is a Boolean type attribute — indicates whether a tuple belongs to subclass Si (true/false).
  • Used for overlapping specializations (but also works for disjoint).
Example (PART — overlapping subclasses MANUFACTURED_PART and PURCHASED_PART):
PART
Part_noDescriptionMflagDrawing_noManufacture_dateBatch_noPflagSupplier_nameList_price

Mflag = Boolean for MANUFACTURED_PART; Pflag = Boolean for PURCHASED_PART.

🌐 Specialization/Generalization Hierarchies & Lattices

A specialization hierarchy has each subclass with only one superclass (single inheritance). A specialization lattice allows a subclass to have more than one superclass (multiple inheritance).

🔗 Mapping of Shared Subclasses (Multiple Inheritance)

  • A shared subclass is a subclass of two or more superclasses — this represents multiple inheritance.
  • All superclasses of a shared subclass must share the same key attribute. If they have different keys, the construct must be modeled as a category (union type) instead.
  • Any of the Step 8 options can be applied to a shared subclass (subject to their constraints).
University Example: STUDENT_ASSISTANT is a shared subclass of both EMPLOYEE and STUDENT.
  • In EMPLOYEE relation → Option 8C used (Employee_type discriminating attribute).
  • In STUDENT relation → Option 8D used (Student_assist_flag Boolean attribute).

Step 9 — Mapping of Categories (Union Types)

Definition

A category (union type) is a subclass of the union of two or more superclasses. Unlike shared subclasses, the superclasses of a category can have different keys because they may be of different entity types.

  • When superclasses have different keys: create a new attribute called a surrogate key as the primary key for the category relation. Add this surrogate key to each superclass relation as well (as a foreign key reference to the category).
  • When superclasses have the same key: no surrogate key is needed.
Key Concept — Surrogate Key: A surrogate key is a new, artificially created key attribute used when the category's defining superclasses have different natural keys, making it impossible to use any single existing key to identify all entities in the category.
Example — OWNER category (subset of PERSON ∪ BANK ∪ COMPANY):

PERSON, BANK, and COMPANY have different keys (Ssn, Bname, Cname), so a surrogate key Owner_id is created.
