📖 Background: Relational Model Concepts

  • A Relation is a mathematical concept based on the ideas of sets.
  • The model was first proposed by Dr. E.F. Codd of IBM Research in 1970 in the paper: "A Relational Model for Large Shared Data Banks", Communications of the ACM, June 1970.
  • This paper caused a major revolution in database management and earned Dr. Codd the ACM Turing Award.

🗄️ Data Manipulation Languages (DML)

DML deals with how to store and retrieve information from a database. Two types exist:

Procedural DML The query specifies the high-level strategy the DBMS should use to find the desired result. Relational Algebra is procedural.
Non-Procedural DML The query specifies only what data is wanted, not how to find it. Relational Calculus / SQL is non-procedural.

⚙️ Relational Algebra — Overview

  • Relational algebra is the basic set of operations for the relational model.
  • These operations enable a user to specify basic retrieval requests (queries).
  • The result of an operation is a new relation, formed from one or more input relations.
  • This property makes the algebra "closed" — all objects in relational algebra are relations.
  • A sequence of relational algebra operations forms a relational algebra expression, whose result is also a relation representing the query result.

🔍 Unary Operation: SELECT (σ)

σ — sigma
  • Used to select a subset of tuples from a relation based on a selection condition.
  • The selection condition acts as a filter.
  • Tuples satisfying the condition are kept; tuples that don't are discarded.
  • The selection condition is a Boolean expression on the attributes of relation R.

General Form

σ <selection condition>(R)

Examples

Select employees in department 4:
σ DNO = 4 (EMPLOYEE)
Select employees with salary > $30,000:
σ SALARY > 30000 (EMPLOYEE)

SELECT Properties

✦ The result has the same schema (same attributes) as R.

✦ SELECT is commutative: σc1(σc2(R)) = σc2(σc1(R))

✦ A cascade of SELECTs can be applied in any order (due to commutativity).

✦ A cascade can be replaced by a single SELECT with a conjunction (AND) of all conditions: σc1(σc2(σc3(R))) = σc1 AND c2 AND c3(R)

✦ The number of result tuples is ≤ the number of tuples in R.

📐 Unary Operation: PROJECT (π)

π — pi
  • Keeps certain columns (attributes) from a relation and discards the others.
  • PROJECT creates a vertical partitioning — specified attributes are kept, others are removed from each tuple.
  • PROJECT removes duplicate tuples from the result (since the result must be a set — mathematical sets don't allow duplicates).

General Form

π <attribute list>(R)

Example

List each employee's first name, last name, and salary:
π LNAME, FNAME, SALARY (EMPLOYEE)

PROJECT Properties

✦ Number of result tuples is ≤ tuples in R.

✦ If the attribute list includes a key of R, result tuple count equals R's tuple count.

✦ PROJECT is NOT commutative.

✦ πlist1(πlist2(R)) = πlist1(R) — as long as list2 contains all attributes in list1.

🔗 Relational Algebra Expressions

Multiple operations can be combined by nesting them into a single expression, or by using intermediate result relations with names.

Example: Retrieve first name, last name, and salary of all employees in department 5.

Method 1 — Single nested expression:

π FNAME, LNAME, SALARY (σ DNO=5 (EMPLOYEE))

Method 2 — Sequence with intermediate relations:

DEP5_EMPS ← σ DNO=5 (EMPLOYEE)
RESULT ← π FNAME, LNAME, SALARY (DEP5_EMPS)

Note: ← is the assignment operator.

✏️ Unary Operation: RENAME (ρ)

ρ — rho
  • Used to rename attributes of a relation, the relation name, or both.
  • Useful when a query requires multiple operations or when the same relation is used multiple times (e.g., in JOIN).

Three Forms of RENAME

ρS(B1,B2,...,Bn)(R) Changes both the relation name to S and all attribute names to B1, B2, ..., Bn
ρS(R) Changes the relation name only to S
ρ(B1,B2,...,Bn)(R) Changes the attribute names only to B1, B2, ..., Bn
Example — Renaming a result:
RESULT (F, M, L, S, B, A, SX, SAL, SU, DNO) ← ρ RESULT(F,M,L,S,B,A,SX,SAL,SU,DNO)(DEP5_EMPS)
The 10 attributes of DEP5_EMPS are renamed to F, M, L, S, B, A, SX, SAL, SU, DNO respectively.

🔢 Set Theory Operations

Type Compatibility is required for UNION, INTERSECTION, and SET DIFFERENCE. Two relations R1(A1,...,An) and R2(B1,...,Bn) are type compatible if they have the same number of attributes AND the domains of corresponding attributes are compatible: dom(Ai) = dom(Bi).

The result relation for R1 ∪ R2, R1 ∩ R2, or R1 – R2 always takes the attribute names of the first operand R1 (by convention).
UNION — ∪
  • Binary operation denoted by ∪.
  • R ∪ S returns a relation that includes all tuples in R, in S, or in both.
  • Duplicate tuples are eliminated.
  • R and S must be type compatible (UNION compatible).
Example: Retrieve SSNs of employees who work in department 5 OR directly supervise someone in department 5.
DEP5_EMPS ← σ DNO=5 (EMPLOYEE)
RESULT1 ← π SSN (DEP5_EMPS)
RESULT2(SSN) ← π SUPERSSN (DEP5_EMPS)
RESULT ← RESULT1 ∪ RESULT2
INTERSECTION — ∩
  • Denoted by ∩.
  • R ∩ S returns a relation that includes all tuples in both R and S.
  • R and S must be type compatible.
  • Attribute names in result = attribute names in R.
SET DIFFERENCE — –
  • Also called MINUS or EXCEPT. Denoted by –.
  • R – S returns a relation that includes all tuples in R but NOT in S.
  • R and S must be type compatible.
  • Attribute names in result = attribute names in R.

Properties of UNION, INTERSECTION, DIFFERENCE

✦ UNION and INTERSECTION are commutative: R ∪ S = S ∪ R  |  R ∩ S = S ∩ R

✦ UNION and INTERSECTION are associative: R ∪ (S ∪ T) = (R ∪ S) ∪ T  |  (R ∩ S) ∩ T = R ∩ (S ∩ T)

✦ SET DIFFERENCE is NOT commutative: R – S ≠ S – R (in general)

Example (STUDENT ∪ INSTRUCTOR, STUDENT ∩ INSTRUCTOR, STUDENT – INSTRUCTOR):
STUDENT ∪ INSTRUCTOR All unique names from both tables combined (no duplicates)
STUDENT ∩ INSTRUCTOR Only names that appear in both (e.g., Susan Yao, Ramesh Shah)
STUDENT – INSTRUCTOR Names in STUDENT but not in INSTRUCTOR (Johnny, Barbara, Amy, Jimmy, Ernest)

✖️ CARTESIAN PRODUCT (×)

CROSS PRODUCT — ×
  • Combines tuples from two relations in a combinatorial fashion.
  • Denoted by R(A1,...,An) × S(B1,...,Bm).
  • Result is a relation Q with degree n + m attributes: Q(A1,...,An, B1,...,Bm).
  • If R has nR tuples and S has nS tuples, then R × S will have nR × nS tuples.
  • The two operands do NOT need to be type compatible.
CROSS PRODUCT alone is generally NOT meaningful — it produces every possible combination of tuples regardless of whether they are actually related. It becomes meaningful when followed by a SELECT operation that filters only the relevant combinations.

Example — Making CROSS PRODUCT Meaningful

Step 1 — Not meaningful (all combinations):
FEMALE_EMPS ← σ SEX='F' (EMPLOYEE)
EMPNAMES ← π FNAME, LNAME, SSN (FEMALE_EMPS)
EMP_DEPENDENTS ← EMPNAMES × DEPENDENT
// Contains every combination of female employee + dependent — even unrelated ones
Step 2 — Meaningful (add SELECT to filter related tuples):
FEMALE_EMPS ← σ SEX='F' (EMPLOYEE)
EMPNAMES ← π FNAME, LNAME, SSN (FEMALE_EMPS)
EMP_DEPENDENTS ← EMPNAMES × DEPENDENT
ACTUAL_DEPS ← σ SSN=ESSN (EMP_DEPENDENTS)
RESULT ← π FNAME, LNAME, DEPENDENT_NAME (ACTUAL_DEPS)
RESULT now contains the names of female employees and their actual dependents.
