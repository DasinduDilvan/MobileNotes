🏗️ Goodness of Relation Schemas

Relational database design is the process of grouping attributes to form "good" relation schemas.

Logical (Conceptual) Level How users interpret the relation schemas and the meaning of their attributes.
Implementation (Physical Storage) Level How the tuples in a base relation are stored and updated.
Design is concerned mainly with base relations. There are two levels at which we can discuss the goodness of relation schemas: the logical/conceptual level and the physical storage level.

📐 Informal Design Guidelines (4 Guidelines)

Guideline 1 – Clear Semantics

Each tuple in a relation should represent one entity or relationship instance.

  • Attributes of different entities (EMPLOYEEs, DEPARTMENTs, PROJECTs) should not be mixed in the same relation.
  • Only foreign keys should be used to refer to other entities.
  • Entity and relationship attributes should be kept apart as much as possible.
  • Bottom line: Design a schema that can be explained easily relation by relation. The semantics of attributes should be easy to interpret.

Violation example: EMP_DEPT(Ename, Ssn, Bdate, Address, Dnumber, Dname, Dmgr_ssn) — employee and department attributes mixed together.

Guideline 2 – Avoid Redundancy & Update Anomalies

Mixing attributes of multiple entities causes update anomalies:

  • Insertion Anomaly: (Type 1) To insert a new employee you must supply department data. (Type 2) Cannot insert a new department without an employee tuple.
  • Deletion Anomaly: Deleting the last (or only) employee tuple for a department causes loss of the department's information.
  • Modification Anomaly: Changing an attribute of a department requires updating multiple employee tuples — inconsistency risk.

Design schemas that do not suffer from these anomalies. If anomalies are unavoidable, document them.

Guideline 3 – Minimize NULL Values

Grouping many attributes into a "fat" relation leads to many NULLs when attributes don't apply to all tuples.

A NULL value is ambiguous — it could mean:

  • The attribute does not apply to the tuple.
  • The attribute value for this tuple is unknown.
  • The value is known but absent.

Rule: Relations should have as few NULL values as possible. Attributes that are frequently NULL could be placed in separate relations (with the primary key).

Guideline 4 – Disallow Spurious Tuples (Lossless Join)

Bad decompositions of relations may result in erroneous results for certain JOIN operations, generating spurious tuples — invalid tuples not present in the original relation.

  • The "lossless join" property guarantees meaningful results for join operations.
  • Relations should be designed to satisfy the lossless join condition.
  • No spurious tuples should be generated by doing a natural-join of any relations.

Example: Decomposing EMP_PROJ into EMP_LOCS(Ename, Plocation) and EMP_PROJ1(Ssn, Pnumber, Hours, Pname, Plocation) and then joining on Plocation produces spurious rows (marked * in the result).

Summary of Problems Detected Without Formal Tools
  • Anomalies (redundant work on insert/modify; accidental data loss on delete).
  • Wasted storage and difficulty with selections, aggregations, and joins due to NULLs.
  • Generation of invalid spurious data during joins on improperly matched attributes.

🔗 Functional Dependencies (FDs)

A functional dependency is a constraint between two sets of attributes from the database.

A relational database schema with n attributes A1, A2, …, An can be thought of as a single universal relation schema R = {A1, A2, …, An} (used only for theory — not implying actual storage as a single table).

Definition — FD Notation: X → Y

A functional dependency, denoted X → Y, between two sets of attributes X and Y (subsets of R) specifies a constraint: for any two tuples t1 and t2 in a relation state r of R,

if t1[X] = t2[X], then t1[Y] = t2[Y]

This means the values of the Y component depend on (are functionally determined by) the X component. X is the left-hand side (LHS); Y is the right-hand side (RHS).

Key Points
  • X uniquely (functionally) determines Y.
  • If X is a candidate key of R, then X → R (X determines every attribute).
  • X → Y does NOT imply Y → X.
  • A FD is a property of the semantics of the attributes.
  • Relation extensions that satisfy all FD constraints are called legal relation states.
Example — EMP_PROJ(Ssn, Pnumber, Hours, Ename, Pname, Plocation)
  • Ssn → Ename  FD2
  • Pnumber → {Pname, Plocation}  FD3
  • {Ssn, Pnumber} → Hours  FD1

Armstrong's Inference Rules (Sound & Complete)

IR1 – Reflexive If Y ⊆ X, then X → Y
IR2 – Augmentation If X → Y, then XZ → YZ (XZ means X ∪ Z)
IR3 – Transitive If X → Y and Y → Z, then X → Z

IR1, IR2, IR3 form a sound and complete set — all other rules that hold can be deduced from these.

Additional Useful Inference Rules (Derived from IR1–IR3)

Decomposition If X → YZ, then X → Y and X → Z
Union If X → Y and X → Z, then X → YZ
Pseudo-Transitivity If X → Y and WY → Z, then WX → Z

🔄 Normalization of Relations

Normalization The process of decomposing unsatisfactory "bad" relations by breaking up their attributes into smaller, well-structured relations.
Normal Form A condition using keys and FDs to certify whether a relation schema meets a particular standard of design quality.
2NF, 3NF, BCNFBased on keys and FDs of a relation schema.
4NFBased on keys and multi-valued dependencies (MVDs).
5NFBased on keys and join dependencies (JDs).
Practical Use
  • Normalization ensures high-quality, anomaly-free designs.
  • In practice, designers typically normalize up to 3NF or BCNF (4NF and 5NF are rarely needed).
  • Denormalization: The reverse process — storing a join of higher normal form relations as a base relation in a lower normal form (done for performance reasons).

🔑 Key Definitions

Superkey A set of attributes S ⊆ R such that no two tuples t1 and t2 in any legal relation state r of R will have t1[S] = t2[S].
Key (Candidate Key) A superkey with the additional property that removing any attribute from K causes K to no longer be a superkey (minimal superkey).
Primary Key One candidate key arbitrarily designated as the primary key; others are called secondary keys.
Prime vs. Nonprime Attribute
  • Prime: Member of some candidate key.
  • Nonprime: Not a member of any candidate key.

📊 Normal Forms — Detailed

1NF — First Normal Form

Considered part of the basic definition of a relation. A relation must:

  • Have only atomic-valued attributes (no composite or multi-valued attributes).
  • Have values stored in an attribute that are of the same domain.
  • Have unique attribute names.
  • Not contain nested relations.
  • Order of data stored does not matter.

Example fix: DEPARTMENT(Dname, Dnumber, Dmgr_ssn, Dlocations) where Dlocations is multi-valued must be split into separate rows, one per location.

Normalizing nested relations: EMP_PROJ with a nested PROJS attribute (Pnumber, Hours) is decomposed into EMP_PROJ1(Ssn, Ename) and EMP_PROJ2(Ssn, Pnumber, Hours) by propagating the primary key.

2NF — Second Normal Form

A relation is in 2NF if:

  • It is in 1NF, AND
  • Every nonprime attribute is fully functionally dependent on the primary key (no partial dependencies).

Partial Dependency: A nonprime attribute depends on only part of a composite primary key, not the whole key.

Full FD: A FD Y → Z where removal of any attribute from Y causes the FD to no longer hold.

Example: In EMP_PROJ(Ssn, Pnumber, Hours, Ename, Pname, Plocation):

  • {Ssn, Pnumber} → Hours — Full FD ✓ (neither Ssn nor Pnumber alone determines Hours)
  • {Ssn, Pnumber} → Ename — Partial FD ✗ (because Ssn → Ename holds alone)

2NF Decomposition of EMP_PROJ: EP1(Ssn, Pnumber, Hours)  |  EP2(Ssn, Ename)  |  EP3(Pnumber, Pname, Plocation)

3NF — Third Normal Form

A relation is in 3NF if:

  • It is in 2NF, AND
  • No nonprime attribute is transitively dependent on the primary key.

Transitive FD: A FD X → Z that can be derived from X → Y and Y → Z, where Y is a nonprime attribute.

Example: In EMP_DEPT(Ename, Ssn, Bdate, Address, Dnumber, Dname, Dmgr_ssn):

  • Ssn → Dmgr_ssn is a transitive FD — because Ssn → Dnumber and Dnumber → Dmgr_ssn.
  • Ssn → Ename is non-transitive — no intermediate attribute set X exists where Ssn → X and X → Ename.

3NF Decomposition: ED1(Ename, Ssn, Bdate, Address, Dnumber)  |  ED2(Dnumber, Dname, Dmgr_ssn)

BCNF — Boyce-Codd Normal Form

BCNF is a stronger form of 3NF. A relation R is in BCNF if:

  • It is in 3NF, AND
  • For every non-trivial FD X → A in R, X must be a superkey of R.

There exist relations in 3NF that are NOT in BCNF. Hence BCNF is considered a stronger form of 3NF.

Hierarchy of Normal Forms: Every 2NF relation is in 1NF → Every 3NF relation is in 2NF → Every BCNF relation is in 3NF (but not vice versa).

Example — TEACH(Student, Course, Instructor) is in 3NF but NOT BCNF:

  • fd1: {Student, Course} → Instructor
  • fd2: Instructor → Course — Instructor is not a superkey! This violates BCNF.

Three possible BCNF decompositions of TEACH:

  • {Student, Instructor} and {Student, Course}
  • {Course, Instructor} and {Course, Student}
  • {Instructor, Course} and {Instructor, Student}
4NF — Fourth Normal Form

A relation is in 4NF if:

  • It is in BCNF, AND
  • It does not have any Multi-valued Dependency (MVD).

Multi-valued Dependency: For a dependency X →→ Y in a relation, if a single value of X is associated with multiple values of Y, then a multi-valued dependency may exist.

  • A relation must have at least 3 columns for an MVD to exist.
  • For a relation R(X, Y, Z): if there is an MVD between X and Y, then Y and Z must be independent of each other.

Example: STUDENT(reg_no, course, club) where one student has multiple courses AND multiple clubs (independent of each other) — this is an MVD. The relation is in BCNF but NOT 4NF. Decompose into STUDENT_COURSE(reg_no, course) and STUDENT_CLUB(reg_no, club).

5NF — Fifth Normal Form (Project-Join Normal Form)

A relation is in 5NF if:

  • It is in 4NF, AND
  • It does not have any Join Dependency (JD).

Join Dependency: A table is in Join Dependency if it can be recreated by joining multiple tables, each having a subset of the table's attributes. 5NF is a generalization of MVD.

Example: SUPPLY(Sname, Part_name, Proj_name) — if the data can be losslessly decomposed into R1(Sname, Part_name), R2(Sname, Proj_name), and R3(Part_name, Proj_name), then SUPPLY has a join dependency and is NOT in 5NF. Decompose into those three relations to achieve 5NF.
