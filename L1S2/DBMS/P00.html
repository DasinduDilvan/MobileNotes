Inserting Rows — 3 Patterns

-- Pattern 1: Specify all columns explicitly
INSERT INTO employee (emp_no, name, gender, dob, salary)
VALUES (1, 'Silva', 'M', '1980-03-25', 55000.00);

-- Pattern 2: No column list — value order must match table schema
INSERT INTO employee
VALUES (2, 'Zoysa', 'F', '1983-06-07', 48000.00);

-- Pattern 3: Omit AUTO_INCREMENT column — DB generates it
INSERT INTO employee (name, gender, dob, salary)
VALUES ('Fonseka', 'M', '1979-04-19', 64000.00);

-- Pattern 4: Insert multiple rows at once
INSERT INTO employee (name, gender, dob, salary)
VALUES
  ('Fernando', 'F', '1975-05-28', 50000.00),
  ('Perera',   'F', '1985-11-14', 72000.00),
  ('Liyanage', 'M', '1989-04-10', 80000.00);

Copy Data Between Tables

-- Copy rows from one table to another
INSERT INTO new_table (col1, col2)
SELECT col1, col2 FROM old_table WHERE condition;
📝 Exam Tips
  • AUTO_INCREMENT column can be omitted during INSERT — DB fills it
  • If no column list: values must match ALL columns in order
  • DROP DATABASE / DROP TABLE = permanent, no undo
  • DELETE FROM employee; removes all rows but keeps table structure; DROP TABLE removes structure too
PRACTICAL 02

INSERT & SELECT Queries

Inserting data, retrieving rows, filtering with WHERE

INSERT Variations

-- With specific column order
INSERT INTO student (student_id, mobile_no, student_name)
VALUES (3, '0773388333', 'Sunil');

-- Without student_id (AUTO_INCREMENT)
INSERT INTO student (mobile_no, student_name)
VALUES ('0773388333', 'Mala');

-- Insert multiple rows at once
INSERT INTO student (student_name)
VALUES ('Student 1'), ('Student 2'), ('Student 3');
⚠️ Common INSERT Errors
  • INSERT INTO student (mobile_no, student_name) VALUES ('Kamal', NULL); — inserts NULL for mobile_no
  • Swapped column order causes data to land in wrong columns
  • Omitting a NOT NULL column without a DEFAULT causes an error

SELECT — Retrieving Data

-- All columns
SELECT * FROM student;

-- Specific columns
SELECT student_name FROM student;
SELECT student_name, mobile_no FROM student;

-- Custom column order
SELECT student_name, mobile_no, student_id FROM student;

WHERE — Filtering Rows

-- Single condition
SELECT * FROM student_marks WHERE student_id = 2;
SELECT * FROM student_marks WHERE marks > 50;

-- AND — both conditions must be true
SELECT * FROM student_marks
WHERE subject_id = 'SUB002' AND marks < 50;

-- OR — either condition true
SELECT * FROM student_marks
WHERE (subject_id = 'SUB001' AND marks > 50)
   OR (subject_id = 'SUB002' AND marks > 50);

PRACTICAL 03

UPDATE & DELETE Queries

Modifying data, deleting rows, CASCADE, safety preview with SELECT

UPDATE Syntax & Examples

-- Single field update
UPDATE student SET mobile_no = '071234789' WHERE student_id = 3;

-- Multiple fields update
UPDATE student SET student_name = 'Amal', mobile_no = '077854632'
WHERE student_id = 5;

-- ⚠️ Mass update — affects ALL rows (no WHERE clause)
UPDATE student SET mobile_no = '911';

DELETE Syntax & Examples

-- Single row
DELETE FROM student WHERE student_id = 2;

-- Conditional delete
DELETE FROM marks WHERE score < 40;
DELETE FROM marks WHERE score BETWEEN 30 AND 35;
DELETE FROM student WHERE age IS NULL;
DELETE FROM student WHERE name = 'Nimal Perera';

-- ⚠️ Delete ALL rows (table structure remains)
DELETE FROM student;

DELETE with Multiple Conditions

-- AND on composite key
DELETE FROM student_marks
WHERE student_id = 4 AND subject_id = 'SUB005';

-- Delete by range
DELETE FROM student WHERE age BETWEEN 20 AND 22;
DELETE FROM student WHERE age > 30;

⚠️ Safety: Preview Before Delete

-- Step 1: Always preview first!
SELECT * FROM student WHERE department = 'IT';

-- Step 2: Then delete
DELETE FROM student WHERE department = 'IT';

ON DELETE CASCADE

-- Child table setup with cascade
CREATE TABLE marks (
    mark_id    INT PRIMARY KEY,
    student_id INT,
    score      INT,
    FOREIGN KEY (student_id)
        REFERENCES student(student_id)
        ON DELETE CASCADE
);

-- Deleting parent automatically removes child rows
DELETE FROM student WHERE student_id = 3;
-- → All marks rows with student_id = 3 are also deleted

PRACTICAL 04

Aliases, Arithmetic & INNER JOIN

Column/table aliases, arithmetic in SELECT, aggregate functions, INNER JOIN

Column & Table Aliases

-- Column alias (single word)
SELECT marks/2 AS halfMarks FROM subject_marks WHERE student_id = 2;

-- Column alias (multi-word — use quotes)
SELECT marks/2 AS 'half Marks' FROM subject_marks WHERE student_id = 2;

-- Table alias (shorthand reference)
SELECT a.student_id, a.marks FROM subject_marks AS a;

-- Both together
SELECT mobile_no AS mobileNumber FROM Student AS studentTable;

Arithmetic in SELECT

SELECT marks     FROM subject_marks WHERE student_id = 2;
SELECT marks/2   FROM subject_marks WHERE student_id = 2;  -- divide
SELECT marks * 2 FROM subject_marks WHERE student_id = 2;  -- multiply

Aggregate Functions

FunctionExamplePurpose
SUM()SELECT SUM(marks) FROM subject_marks WHERE student_id = 2;Total of all values
AVG()SELECT AVG(marks) FROM ict_marks;Average value
COUNT()SELECT COUNT(*) FROM student_data;Row count
MIN()SELECT MIN(marks) FROM ict_marks;Smallest value
MAX()SELECT MAX(marks) FROM ict_marks;Largest value

INNER JOIN

-- Syntax (two tables)
SELECT columnList FROM table1
INNER JOIN table2 ON table1.col = table2.col;

-- Without aliases
SELECT * FROM subject_marks
INNER JOIN student ON subject_marks.student_id = student.student_id;

-- With aliases (cleaner)
SELECT * FROM subject_marks a
INNER JOIN student b ON a.student_id = b.student_id;
How INNER JOIN Works

Returns only rows where the join condition is met in both tables. Non-matching rows from either table are excluded.

📝 Exam Tips
  • Multi-word aliases need quotes: AS 'half Marks'
  • Table alias removes need to write full table name repeatedly
  • Aggregate functions ignore NULL values (except COUNT(*))
  • INNER JOIN = only matching rows in both tables
PRACTICAL 05

All JOIN Types

INNER, LEFT, RIGHT, FULL OUTER, SELF joins — syntax and use cases

JOIN Types Comparison

JOIN TypeReturns
INNER JOINOnly rows with matching values in BOTH tables
LEFT JOINALL rows from left table + matching from right (NULL if no match)
RIGHT JOINALL rows from right table + matching from left (NULL if no match)
FULL OUTER JOINALL rows from both tables (NULL where no match) — not native in MySQL
SELF JOINTable joined to itself (use aliases)

INNER JOIN

SELECT emp.ENAME, emp.JOB, dept.DNAME, dept.LOC
FROM EMPLOYEE emp
INNER JOIN DEPARTMENT dept ON emp.DEPTNO = dept.DEPTNO;

LEFT JOIN

-- All employees + department (NULL if employee has no department)
SELECT emp.ENAME, emp.JOB, dept.DNAME, dept.LOC
FROM EMPLOYEE emp
LEFT JOIN DEPARTMENT dept ON emp.DEPTNO = dept.DEPTNO;

RIGHT JOIN

-- All departments + employees (NULL if department has no employees)
SELECT emp.ENAME, emp.JOB, dept.DNAME, dept.LOC
FROM EMPLOYEE emp
RIGHT JOIN DEPARTMENT dept ON emp.DEPTNO = dept.DEPTNO;

FULL OUTER JOIN (MySQL Workaround)

-- MySQL does not support FULL OUTER JOIN natively
-- Simulate using LEFT JOIN UNION RIGHT JOIN:
SELECT emp.ENAME, emp.JOB, dept.DNAME, dept.LOC
FROM EMPLOYEE emp LEFT JOIN DEPARTMENT dept ON emp.DEPTNO = dept.DEPTNO
UNION
SELECT emp.ENAME, emp.JOB, dept.DNAME, dept.LOC
FROM EMPLOYEE emp RIGHT JOIN DEPARTMENT dept ON emp.DEPTNO = dept.DEPTNO;

SELF JOIN

-- Employee-to-Manager relationship (same table, two aliases)
SELECT e.ENAME AS Employee, m.ENAME AS Manager
FROM EMPLOYEE e
JOIN EMPLOYEE m ON e.MGR = m.EMPNO;
Venn Diagram Mental Model

INNER JOIN = overlap only. LEFT = full left circle. RIGHT = full right circle. FULL = both circles together.

📝 Exam Tips
  • MySQL doesn't have FULL OUTER JOIN — use UNION of LEFT + RIGHT
  • SELF JOIN requires two different aliases for the same table
  • Use aliases (emp, dept, e, m) for readability in multi-table queries
  • NULL appears in result where there's no matching row

User Management & Operators

CREATE USER, GRANT, REVOKE, NULL handling, LIKE, IN, BETWEEN, ORDER BY, DISTINCT, LIMIT, Aggregates

User Management

-- Create users
CREATE USER 'admin_ict'@'localhost'  IDENTIFIED BY 'adminpassword';
CREATE USER 'lecturer'@'localhost'   IDENTIFIED BY 'lecturerpwd';
CREATE USER 'Student'@'localhost'    IDENTIFIED BY 'studentpwd';
CREATE USER 'ict_guest'@'localhost'; -- No password

-- Grant privileges
GRANT ALL PRIVILEGES ON tecict.* TO 'admin_ict'@'localhost';

-- Grant DML only
GRANT SELECT, INSERT, UPDATE, DELETE ON tecict.* TO 'lecturer'@'localhost';

-- Grant read-only on specific table
GRANT SELECT ON tecict.studentData TO 'Student'@'localhost';

-- Apply changes immediately
FLUSH PRIVILEGES;

-- Delete a user
DROP USER 'ict_guest'@'localhost';

NULL Handling

-- ❌ Wrong — never use = NULL
SELECT * FROM table WHERE column_name = NULL;

-- ✅ Correct
SELECT * FROM table WHERE column_name IS NULL;
SELECT * FROM table WHERE column_name IS NOT NULL;

AND / OR / NOT Operators

-- AND (both conditions must be true)
SELECT * FROM student_data WHERE DoB > '1997-01-01' AND gender = 'Male';

-- OR (either condition true)
SELECT * FROM student_data WHERE gender = 'Male' OR DoB > '1997-01-01';

-- NOT (negate condition)
SELECT * FROM student_data WHERE NOT gender = 'Male';

-- Combined
SELECT * FROM student_data WHERE DoB > '1997-01-01' AND NOT gender = 'Male';

LIKE — Pattern Matching

PatternMatches
'a%'Starts with "a"

'_r%'"r" in second position

SELECT * FROM student_data WHERE StuName LIKE 'K%';
SELECT * FROM student_data WHERE StuName LIKE '%a';

IN, BETWEEN, ORDER BY

-- IN — match any value in list
SELECT * FROM student WHERE department IN ('IT', 'Finance');

-- BETWEEN — inclusive range
SELECT * FROM marks WHERE score BETWEEN 50 AND 80;

-- ORDER BY
SELECT * FROM student_data ORDER BY DoB ASC;
SELECT * FROM student_data ORDER BY gender DESC;
SELECT * FROM student_data ORDER BY DoB ASC, RegNo ASC, gender DESC;

DISTINCT, LIMIT

-- Unique values only
SELECT DISTINCT gender FROM student_data;
SELECT COUNT(DISTINCT gender) FROM student_data;

-- Limit rows returned
SELECT * FROM student_data LIMIT 3;
SELECT * FROM student_data WHERE gender = 'Male' LIMIT 2;

Computed Column — Age on the Fly

SELECT
    RegNo      AS 'Registration No',
    StuName    AS 'Student Name',
    DoB        AS 'Date of Birth',
    gender     AS 'Gender',
    TIMESTAMPDIFF(YEAR, DoB, CURDATE()) AS 'Age'
FROM student_data;
📝 Exam Tips
  • GRANT syntax: GRANT privilege ON db.table TO 'user'@'host';
  • IS NULL not = NULL
  • BETWEEN is inclusive on both ends
  • ORDER BY default is ASC; add DESC to reverse
  • DISTINCT removes duplicate rows from result

Constraints & ALTER TABLE

NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY, CHECK, DEFAULT, ALTER TABLE, GROUP BY, HAVING, UNION

Constraints on CREATE TABLE

CREATE TABLE employees (
    emp_id   INT PRIMARY KEY,       -- PRIMARY KEY
    name     VARCHAR(50) NOT NULL,  -- NOT NULL
    email    VARCHAR(100) UNIQUE,   -- UNIQUE
    salary   INT,
    age      INT,
    dept_id  INT,
    CHECK (age >= 18),              -- CHECK
    FOREIGN KEY (dept_id)
        REFERENCES departments(dept_id)
        ON DELETE RESTRICT
        ON UPDATE RESTRICT
);

Constraints Summary

ConstraintPurposeNotes

UNIQUENo duplicate values in columnAllows NULL (only one NULL)
PRIMARY KEYUnique + Not Null identifierOnly one per table
FOREIGN KEYLinks to another table's PKEnforces referential integrity

ALTER TABLE — Add / Modify / Drop

-- Add a column
ALTER TABLE student_data ADD CivilStatus VARCHAR(20);

-- Modify column data type
ALTER TABLE student_data MODIFY COLUMN CivilStatus VARCHAR(30);

-- Drop a column
ALTER TABLE student_data DROP COLUMN CivilStatus;

-- Add CHECK constraint
ALTER TABLE ict_marks ADD CHECK (marks BETWEEN 0 AND 100);

-- Add FOREIGN KEY
ALTER TABLE orders ADD FOREIGN KEY (cust_id) REFERENCES customers(cust_id);

-- Add DEFAULT
ALTER TABLE student_data ALTER gender SET DEFAULT 'Male';
ALTER TABLE student_data ALTER DoB SET DEFAULT CURDATE();

-- Remove DEFAULT
ALTER TABLE student_data ALTER gender DROP DEFAULT;

GROUP BY & HAVING

-- Total salary per department
SELECT department, SUM(salary)
FROM Employees
GROUP BY department;

-- Only departments with total salary > 10000
SELECT department, SUM(salary) AS total_salary
FROM Employees
GROUP BY department
HAVING SUM(salary) > 10000;
WHERE vs HAVING

WHERE filters individual rows BEFORE grouping.
HAVING filters grouped results AFTER GROUP BY.

UNION vs UNION ALL

-- UNION — removes duplicates
SELECT student_name FROM section_A
UNION
SELECT student_name FROM section_B;

-- UNION ALL — keeps duplicates (faster)
SELECT student_name FROM section_A
UNION ALL
SELECT student_name FROM section_B;
📝 Exam Tips
  • PRIMARY KEY = NOT NULL + UNIQUE combined
  • CHECK constraint enforces value rules, e.g., marks BETWEEN 0 AND 100
  • GROUP BY column must appear in SELECT list
  • HAVING is like WHERE but applied after grouping
  • UNION removes duplicates; UNION ALL keeps them (faster)
PRACTICAL 08

Views

CREATE VIEW, DROP VIEW, OR REPLACE, DML through views, WITH CHECK OPTION

What is a View?

A view is a virtual table based on a SELECT query. It does not store data itself — it always shows up-to-date data from the underlying table(s).

Create & Drop Views

-- Create a view
CREATE VIEW high_gpa_students AS
    SELECT name, gpa
    FROM student
    WHERE gpa > 3.0;

-- Query the view like a table
SELECT * FROM high_gpa_students;

-- Delete a view
DROP VIEW high_gpa_students;

Replace / Update a View

-- Modify an existing view
CREATE OR REPLACE VIEW high_gpa_students AS
    SELECT name, gpa
    FROM student
    WHERE gpa > 3.5;  -- changed threshold

DML Through Views

-- UPDATE through view (updates base table)
UPDATE high_gpa_students SET gpa = 3.6 WHERE name = 'Amal';

-- INSERT through view (inserts into base table)
INSERT INTO high_gpa_students (name, gpa) VALUES ('Kamal', 3.6);

-- ⚠️ Insert out-of-range row (succeeds WITHOUT check option!)
INSERT INTO high_gpa_students (name, gpa) VALUES ('Ravi', 2.4);
⚠️ Non-Updatable Views

Views using aggregate functions (AVG, SUM, COUNT etc.), DISTINCT, GROUP BY, HAVING, or UNION are NOT updatable through the view.

-- Non-updatable view example
CREATE VIEW student_avg AS
    SELECT AVG(gpa) AS avg_gpa FROM student;

WITH CHECK OPTION

-- Enforces the WHERE condition on INSERT and UPDATE
CREATE VIEW high_gpa_students AS
    SELECT id, name, gpa
    FROM students
    WHERE gpa > 3.0
    WITH CHECK OPTION;

-- ✅ Valid — gpa 3.8 satisfies the view condition
INSERT INTO high_gpa_students (name, gpa) VALUES ('Nadeesha', 3.8);

-- ❌ Error — gpa 2.4 violates the view's WHERE condition
INSERT INTO high_gpa_students (name, gpa) VALUES ('Ravi', 2.4);

-- ✅ Valid update — stays above 3.0
UPDATE high_gpa_students SET gpa = 3.9 WHERE name = 'Kamal';

-- ❌ Error — update would drop below 3.0
UPDATE high_gpa_students SET gpa = 2.0 WHERE name = 'Kamal';
OperationWithout CHECK OPTIONWith CHECK OPTION
INSERT within condition✅ Succeeds✅ Succeeds
INSERT outside condition✅ Succeeds (data invisible)❌ Error thrown
UPDATE within condition✅ Succeeds✅ Succeeds
UPDATE outside condition✅ Succeeds (row hidden)❌ Error thrown
📝 Exam Tips
  • Views = virtual table = no stored data, always live from base table
  • WITH CHECK OPTION enforces the view's WHERE filter on writes
  • Aggregate-based views are NOT updatable
  • CREATE OR REPLACE VIEW is safer than DROP + CREATE
PRACTICAL 09

Physical Design — Hospital System

Ternary relationships, FK with CASCADE, multi-table JOIN verification query

Ternary Relationship Concept

A ternary relationship connects three entity types simultaneously. Represented by a relationship table with foreign keys to all three entity tables.

Hospital Management DB Schema

CREATE DATABASE hospital_management_db;
USE hospital_management_db;

-- Entity 1: PATIENTS
CREATE TABLE patients (
    patient_id INT AUTO_INCREMENT,
    name       VARCHAR(100) NOT NULL,
    age        INT NOT NULL,
    PRIMARY KEY (patient_id)
);

-- Entity 2: DOCTORS
CREATE TABLE doctors (
    doctor_id      INT AUTO_INCREMENT,
    name           VARCHAR(100) NOT NULL,
    specialization VARCHAR(100) NOT NULL,
    PRIMARY KEY (doctor_id)
);

-- Entity 3: MEDICATIONS
CREATE TABLE medications (
    medication_id   INT AUTO_INCREMENT,
    medication_name VARCHAR(100) NOT NULL,
    type            VARCHAR(50) NOT NULL,
    PRIMARY KEY (medication_id)
);

-- Ternary Relationship: PRESCRIPTIONS
CREATE TABLE prescriptions (
    prescription_id INT AUTO_INCREMENT,
    patient_id      INT NOT NULL,
    doctor_id       INT NOT NULL,
    medication_id   INT NOT NULL,
    dosage          VARCHAR(50) NOT NULL,
    date_prescribed DATE NOT NULL,
    PRIMARY KEY (prescription_id),
    FOREIGN KEY (patient_id)    REFERENCES patients(patient_id)    ON DELETE CASCADE,
    FOREIGN KEY (doctor_id)     REFERENCES doctors(doctor_id)      ON DELETE CASCADE,
    FOREIGN KEY (medication_id) REFERENCES medications(medication_id) ON DELETE CASCADE
);

Verification Query — Multi-Table JOIN

SELECT
    pr.prescription_id AS 'Prescription ID',
    p.name             AS 'Patient Name',
    p.age              AS 'Patient Age',
    d.name             AS 'Prescribing Doctor',
    d.specialization   AS 'Specialization',
    m.medication_name  AS 'Medication Name',
    pr.dosage          AS 'Dosage',
    pr.date_prescribed AS 'Date Prescribed'
FROM prescriptions pr
INNER JOIN patients    p  ON pr.patient_id    = p.patient_id
INNER JOIN doctors     d  ON pr.doctor_id     = d.doctor_id
INNER JOIN medications m  ON pr.medication_id = m.medication_id;
Key Pattern

Ternary: One relationship table holds 3 FK columns. When joining back, chain 3 INNER JOINs to the relationship table. ON DELETE CASCADE ensures removing a doctor/patient/medication also removes their prescriptions.

📝 Exam Tips
  • Ternary relationship = 3 entity tables + 1 relationship table with 3 FKs
  • Use ON DELETE CASCADE so orphan records are cleaned automatically
  • Multi-table JOIN: each INNER JOIN adds one more table with its condition
PRACTICAL 10

Stored Procedures

DELIMITER, CREATE PROCEDURE, no-parameter and IN-parameter procedures, CALL

Why Stored Procedures?

A stored procedure is a saved SQL block that can be called repeatedly. Reduces repetitive code, improves performance, and centralizes logic.

DELIMITER — Why It's Needed

Concept

SQL uses ; as statement terminator. Inside a procedure body, ; would confuse the parser. We temporarily change the delimiter to $$ so MySQL treats the entire procedure as one unit, then restore it.

No-Parameter Procedure

-- Create
DELIMITER $$
CREATE PROCEDURE GetAllStudents()
BEGIN
    SELECT * FROM students;
END $$
DELIMITER ;

-- Call
CALL GetAllStudents();
-- Another example
DELIMITER $$
CREATE PROCEDURE GetAllCourses()
BEGIN
    SELECT * FROM courses;
END $$
DELIMITER ;

CALL GetAllCourses();

IN Parameter Procedure

-- One IN parameter
DELIMITER $$
CREATE PROCEDURE GetStudentsByCity (IN city_name VARCHAR(50))
BEGIN
    SELECT * FROM students WHERE city = city_name;
END $$
DELIMITER ;

-- Call with different values
CALL GetStudentsByCity('Kandy');
CALL GetStudentsByCity('Galle');
-- Multiple IN parameters
DELIMITER $$
CREATE PROCEDURE GetStudentsByCityAndAge (
    IN city_name   VARCHAR(50),
    IN student_age INT
)
BEGIN
    SELECT * FROM students
    WHERE city = city_name AND age = student_age;
END $$
DELIMITER ;

-- Call
CALL GetStudentsByCityAndAge('Colombo', 23);
CALL GetStudentsByCityAndAge('Galle', 21);

Procedure Structure Template

PartCodePurpose
Change delimiterDELIMITER $$Prevents ; confusion inside body
CreateCREATE PROCEDURE name(params)Define procedure
Body startBEGINStart of SQL block
SQL logicSELECT / INSERT / ...Actual work
Body endEND $$End using custom delimiter

📝 Exam Tips
  • Always wrap procedure creation with DELIMITER $$ ... DELIMITER ;
  • No-param: GetAll() — empty parentheses
  • IN param: passes a value into the procedure; cannot return it back
  • CALL to execute; pass values matching the parameter types and order
PRACTICAL 11

Normalization in Practice

3NF decomposition — university results database implementation

Why Normalize?

❌ Insertion Anomaly

Can't add a new course unless a student is enrolled in it.

❌ Deletion Anomaly

Deleting a student removes the only record of a course.

❌ Update Anomaly

Changing coordinator address needs updating many rows → risk of inconsistency.

Problem: Unnormalized Table

-- Original flat table (violates 2NF and 3NF)
-- RESULTS (student_id, student_name, address, degree,
--          course_id, course_name, coordinator_name, coordinator_address,
--          grade)
--
-- Partial dependencies  → student_name, address, degree depend only on student_id
--                        → course_name, coordinator_name depend only on course_id
-- Transitive dependency → coordinator_address depends on coordinator_name (not course_id)

3NF Decomposition Solution

CREATE DATABASE IF NOT EXISTS university_normalization_db;
USE university_normalization_db;

-- Table A: COORDINATOR (removes transitive dependency)
-- coordinator_address depends on coordinator_name, not course_id
CREATE TABLE coordinator (
    coordinator_name    VARCHAR(100) NOT NULL,
    coordinator_address VARCHAR(100) NOT NULL,
    PRIMARY KEY (coordinator_name)
);

-- Table B: STUDENT (removes partial dependency from student_id)
CREATE TABLE student (
    student_id   VARCHAR(20)  NOT NULL,
    student_name VARCHAR(100) NOT NULL,
    address      VARCHAR(100) NOT NULL,
    degree       VARCHAR(50)  NOT NULL,
    PRIMARY KEY (student_id)
);

-- Table C: COURSE (removes partial dependency from course_id)
CREATE TABLE course (
    course_id        VARCHAR(20)  NOT NULL,
    course_name      VARCHAR(100) NOT NULL,
    coordinator_name VARCHAR(100) NOT NULL,
    PRIMARY KEY (course_id),
    FOREIGN KEY (coordinator_name)
        REFERENCES coordinator(coordinator_name) ON UPDATE CASCADE
);

-- Table D: RESULTS (composite PK = many-to-many: student ↔ course)
CREATE TABLE results (
    student_id VARCHAR(20) NOT NULL,
    course_id  VARCHAR(20) NOT NULL,
    grade      CHAR(2)     NOT NULL,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES student(student_id) ON DELETE CASCADE,
    FOREIGN KEY (course_id)  REFERENCES course(course_id)   ON DELETE CASCADE
);

Normalization Steps Summary

1NF
Atomicity: All values atomic. No repeating groups. Identify a primary key.
2NF
Remove partial dependencies: Every non-key attribute must depend on the WHOLE primary key (not just part of it). Only applies when PK is composite.
3NF
Remove transitive dependencies: Non-key attribute must NOT depend on another non-key attribute. Move it to its own table.
Dependency TypeViolationFix
Partial (2NF)Attribute depends on PART of composite PKExtract to new table with that PK part
Transitive (3NF)Non-key → Non-key → PK chainExtract the transitively dependent group to new table
📝 Exam Tips
  • 2NF: Only relevant when primary key is composite
  • 3NF: Look for A → B → C where B is not a key
  • Many-to-many = bridge/results table with composite PK
  • Use ON UPDATE CASCADE on coordinator FK so name changes propagate

Backup & Restore

mysqldump for backups, mysql for restoration, single table or full database

Setup: Sample School Database

CREATE DATABASE school;

    name   VARCHAR(50),

    (2, 'Nimal', 'Networking');

Backup Commands (Command Prompt / Terminal)

⚠️ Important

These commands are run in the OS terminal / command prompt — NOT inside the MySQL client.

-- Full database backup
mysqldump --set-gtid-purged=OFF -u root -p --databases school > school.sql

-- Single table backup only
mysqldump --set-gtid-purged=OFF -u root -p school student > student.sql

Restore Commands (Command Prompt / Terminal)

-- Full database restore
mysql -u root -p < school.sql

-- Single table restore into existing database
mysql -u root -p school < student.sql

What's Inside a .sql Backup File?

-- A backup file contains DDL + DML:
CREATE TABLE student (
    id   int,
    name varchar(50)
);

INSERT INTO student VALUES (1, 'Kamal', 'Computing');

Quick Reference

📦 Backup

mysqldump ... > output.sql

  • Full DB: add --databases dbname
  • Single table: dbname tablename
  • Always run in OS terminal
♻️ Restore

mysql ... < input.sql

  • Full DB: no need to specify DB name
  • Single table: must specify target DB
  • DB must exist first for table restore
📝 Exam Tips
  • Backup uses mysqldump; restore uses mysql
  • > = write to file (backup); < = read from file (restore)
  • Backup commands run in OS terminal, NOT MySQL shell
  • --set-gtid-purged=OFF is needed to avoid errors on most setups
  • A backup file is plain SQL — CREATE TABLE + INSERT statements
