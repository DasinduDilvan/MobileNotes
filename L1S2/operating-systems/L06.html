⚙️ Process Synchronization

Lecture 06 — Operating Systems | Concurrency, Critical Sections & Classical Problems

1. Background

  • Processes can execute concurrently and may be interrupted at any time by context switching.
  • This leads to partially completed executions.
  • Concurrent access to shared data may result in data inconsistency.
  • Maintaining data consistency requires mechanisms to ensure the orderly execution of cooperating processes.
Producer-Consumer Problem is a classic paradigm developed to illustrate a system with cooperating processes running concurrently while possibly sharing data (e.g., shared variable A=10, where P1 adds 5 and P2 subtracts 5 simultaneously).

2. Producer-Consumer Problem

A producer process produces information that is consumed by a consumer process. They communicate via a shared circular buffer.

Buffer Full Condition (in + 1) % BUFFER_SIZE == out
Buffer Empty Condition in == out
The original model allowed at most BUFFER_SIZE – 1 items in the buffer at a given time. To fill all buffers, an integer counter variable is used — incremented by producer, decremented by consumer.

Producer Code

while (true) {
  // produce item in next_produced
  while (counter == BUFFER_SIZE)
    ; // do nothing (wait)
  buffer[in] = next_produced;
  in = (in + 1) % BUFFER_SIZE;
  counter++;
}

Consumer Code

while (true) {
  while (counter == 0)
    ; // do nothing (wait)
  next_consumed = buffer[out];
  out = (out + 1) % BUFFER_SIZE;
  counter--;
  // consume next_consumed
}
Although each routine is correct separately, they may not function correctly when executed concurrently — this leads to a Race Condition.

3. Race Condition

Definition: Several processes access and manipulate the same data concurrently, and the outcome depends on the particular order in which the shared resource is accessed.

Low-level implementation of counter++ and counter-- 
// counter++ (Producer)
register1 = counter
register1 = register1 + 1
counter = register1

// counter-- (Consumer)
register2 = counter
register2 = register2 - 1
counter = register2

Interleaved execution example (initial counter = 5):

StepProcessOperationResult
T0Producerregister1 = counterregister1 = 5
T1Producerregister1 = register1 + 1register1 = 6
T2Consumerregister2 = counterregister2 = 5
T3Consumerregister2 = register2 – 1register2 = 4
T4Producercounter = register1counter = 6
T5Consumercounter = register2counter = 4 ⚠️
Final value of counter is 4, but the correct value should be 5. To prevent this, only one process at a time must be allowed to manipulate the shared variable.

4. Critical Section Problem

Consider a system of n processes {p₀, p₁, … pₙ₋₁}. Each process has a critical section — a segment of code where shared variables, tables, or files may be modified.

When one process executes in its critical section, no other process is allowed to execute in its critical section. The goal is to design a protocol for cooperating processes.

Entry Section Code requesting permission to enter the critical section
Critical Section Code in which only one process can execute at any one time
Exit Section The end of the critical section — releasing or allowing others in
Remainder Section Rest of the code after the critical section

General structure of process Pᵢ:

do {
    entry section Entry

    critical section Critical

    exit section Exit

    remainder section Remainder
} while (true);

5. Requirements for a Critical Section Solution

1. Mutual Exclusion If process Pᵢ is executing in its critical section, then no other processes can be executing in their critical sections.
2. Progress If no process is executing in its CS and some wish to enter, the selection of who enters next must be decided in a finite time.
3. Bounded Waiting A bound must exist on the number of times other processes can enter their CS after a request has been made and before it is granted.
Assume that each process executes at a nonzero speed, and no assumption is made about the relative speed of the n processes.

6. Peterson's Solution

A software-based solution to the critical section problem, restricted to two processes Pᵢ and Pⱼ that alternate execution between their Critical Section (CS) and Remainder Section (RS).

Assumption: The load and store machine-language instructions are atomic (cannot be interrupted).

Shared Variables:

int turn Indicates whose turn it is to enter the CS. When turn == i, process Pᵢ is allowed to execute in its CS.
boolean flag[2] Indicates if a process is ready to enter the CS. flag[i] = true means Pᵢ is ready.

Algorithm for Process Pᵢ:

do {
    flag[i] = true;   // Entry: signal intent
    turn = j;            // Entry: yield turn to other process
    while (flag[j] && turn == j); // Busy-wait if other wants in

    /* --- CRITICAL SECTION --- */

    flag[i] = false;  // Exit: signal done

    /* --- REMAINDER SECTION --- */
} while (true);

Proof that all 3 CS requirements are met:

  • Mutual Exclusion: Pᵢ enters CS only if flag[j] = false OR turn = i. Both cannot be in CS simultaneously.
  • Progress: The while loop condition will eventually become false, ensuring progress.
  • Bounded Waiting: After Pⱼ exits its CS, it sets flag[j] = false, allowing Pᵢ to enter — so Pᵢ waits at most one entry by Pⱼ.
Limitation: Peterson's Solution is not guaranteed to work on critical section problems in modern computer architectures (due to instruction reordering by CPUs).

7. Synchronization Hardware & Locks

Many systems provide hardware support for implementing the critical section code. All solutions are based on the idea of locking — protecting critical regions via locks.

// General lock-based structure
do {
    acquire lock
    /* critical section */
    release lock
    /* remainder section */
} while (TRUE);

8. Mutex Locks

The simplest software tool built by OS designers to solve the critical section problem.

acquire() Called before entering the critical section. Busy-waits until the lock is available, then sets it unavailable.
release() Called after exiting the critical section. Sets the lock back to available.

Implementation:

acquire() {
    while (!available)
        ; /* busy wait */
    available = false;
}

release() {
    available = true;
}
Key Properties:
  • Implemented using a boolean variable indicating if the lock is available.
  • Calls to acquire() and release() must be atomic — usually implemented via hardware atomic instructions.
  • This solution requires busy waiting (the process continuously calls acquire() in a loop).
  • A lock that uses busy waiting is called a spinlock.

9. Semaphores

A more sophisticated synchronization tool than mutex locks. A Semaphore S is an integer variable that can only be accessed via two atomic (indivisible) operations: wait() and signal().

wait(S) — also called P()

wait(S) {
    while (S <= 0)
        ; // busy wait
    S--;
}

signal(S) — also called V()

signal(S) {
    S++;
}

Types of Semaphores:

Binary Semaphore Integer value ranges only between 0 and 1. Equivalent to a mutex lock. Can be used for critical section problems.
Counting Semaphore Integer value ranges over an unrestricted domain. Used for resource allocation and other synchronization problems.
Counting Semaphore Usage for Resource Allocation:
  • Semaphore is initialized to the number of resources available.
  • Each process wishing to use a resource performs wait() — decreasing the count.
  • When a process releases a resource, it performs signal() — increasing the count.
  • When count goes to 0, all resources are being used — further processes block until count > 0.

10. Classical Problems of Synchronization

🔲 Bounded-Buffer Problem Producer-consumer with a fixed-size buffer, managed using semaphores.
📖 Readers-Writers Problem Multiple readers can read simultaneously; only one writer can write at a time.
🍜 Dining-Philosophers Problem 5 philosophers sharing 5 chopsticks — models resource allocation and deadlock.

11. Bounded-Buffer Problem (with Semaphores)

The Producer-Consumer problem solved using wait() and signal(). There are n buffers, each holding one item.

Shared Variables:

Semaphore mutex = 1 (Binary) Provides mutual exclusion to access the buffer.
Semaphore full = 0 (Counting) Counts the number of full buffers.
Semaphore empty = n (Counting) Counts the number of empty buffers.

Producer Process

while (true) {
  // produce item in next_produced

  wait(empty);  // wait for space
  wait(mutex);  // lock buffer

  // add next_produced to buffer

  signal(mutex); // unlock buffer
  signal(full);  // signal item added
}

Consumer Process

while (true) {
  wait(full);   // wait for item
  wait(mutex);  // lock buffer

  // remove item → next_consumed

  signal(mutex);  // unlock buffer
  signal(empty);  // signal space freed

  // consume next_consumed
}

12. Readers-Writers Problem

A data set is shared among concurrent processes:

  • Readers — only read the data set; they do not perform any updates.
  • Writers — can both read and write.
Rule: Multiple readers may read simultaneously. Only one single writer can access the shared data at a time. When a writer is active, no other reader or writer is allowed.

Shared Variables:

Semaphore rw_mutex = 1 Controls writer access; protects the data set.
Semaphore mutex = 1 Protects the read_count variable.
int read_count = 0 Tracks the number of active readers.

Writer Process

do {
  wait(rw_mutex);

  /* writing is performed */

  signal(rw_mutex);
} while (true);

Reader Process

while (true) {
  wait(mutex);
  read_count++;
  if (read_count == 1)
    wait(rw_mutex); // first reader locks
  signal(mutex);

  /* reading is performed */

  wait(mutex);
  read_count--;
  if (read_count == 0)
    signal(rw_mutex); // last reader unlocks
  signal(mutex);
}
The first reader locks rw_mutex to block any writers. The last reader releases rw_mutex, allowing writers to proceed.

13. Dining-Philosophers Problem

5 philosophers sit at a circular table with a bowl of rice in the center and 5 chopsticks, one between each pair of adjacent philosophers. They alternate between thinking and eating.

To eat: A philosopher must acquire both the left and right chopstick (one at a time). After eating, chopsticks are placed back down and the philosopher resumes thinking.

Shared Data:

  • Bowl of rice (the shared data set)
  • Semaphore chopstick[5] — all initialized to 1

Philosopher i Algorithm:

do {
    wait(chopstick[i]);           // pick up LEFT chopstick
    wait(chopstick[(i + 1) % 5]); // pick up RIGHT chopstick

    /* eat */

    signal(chopstick[i]);           // put down LEFT chopstick
    signal(chopstick[(i + 1) % 5]); // put down RIGHT chopstick

    /* think */
} while (TRUE);
Problem with this algorithm: If all 5 philosophers pick up their left chopstick simultaneously, none can pick up their right chopstick — resulting in Deadlock.

14. Problems with Semaphores

Semaphores, while powerful, are prone to incorrect usage. Common mistakes include:

❌ signal before wait signal(mutex) … CS … wait(mutex)

Several processes may execute in the CS simultaneously — violates Mutual Exclusion.

❌ Double wait wait(mutex) … CS … wait(mutex)

Deadlock will occur — no one can release the lock.

❌ Omitting wait or signal Forgetting either wait(mutex) or signal(mutex) (or both).

Either mutual exclusion is violated or deadlock will occur.

Key risks with incorrect semaphore usage:
  • Deadlock — two or more processes are waiting indefinitely for an event that can only be caused by one of the waiting processes.
  • Starvation — a process waits indefinitely; it may never be removed from the semaphore queue.
