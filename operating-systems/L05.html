💻 CPU Scheduling

Lecture 05 — Operating Systems

Overview

  • In a single processor system, only one process runs at a time.
  • When that process waits (e.g., for I/O), the CPU sits idle — this is wasteful.
  • In a multiprogramming system, several processes are kept in memory at once.
  • When one process waits, the OS gives the CPU to another — CPU never stays idle.
  • To achieve this, the CPU must be scheduled among processes.
Key Concept — CPU–I/O Burst Cycle:
Process execution consists of a cycle of CPU execution and I/O wait. A CPU burst is followed by an I/O burst. There are large numbers of short CPU bursts and small numbers of long CPU bursts.

CPU Scheduler

When the CPU becomes idle, the OS selects one process from the ready queue (in memory) to be executed. This selection is done by the Short-term scheduler (CPU scheduler). The records in the queue are generally PCBs of the processes.

Scheduling Decision Points

CPU scheduling decisions occur when a process:

Nonpreemptive Situations
  • 1. Switches from running → waiting (I/O request or wait() call)
  • 4. Terminates

No choice — a new process must be selected.

Preemptive Situations
  • 2. Switches from running → ready (interrupt)
  • 3. Switches from waiting → ready (I/O completion)

There is a choice of which process to run next.

Nonpreemptive Scheduling: A running task executes until completion — it cannot be interrupted.

Preemptive Scheduling: A running task can be interrupted and resumed later. Requires consideration of: shared data access, preemption while in kernel mode, and interrupts during crucial OS activities.

Dispatcher

The Dispatcher module gives control of the CPU to the process selected by the short-term scheduler. It involves:

  • Switching context
  • Switching to user mode
  • Jumping to the proper location in the user program to restart it
Dispatch Latency: The time it takes for the dispatcher to stop one process and start another running. It should be as fast as possible since it is invoked during every process switch.

Scheduling Criteria

CPU Utilization Keep the CPU as busy as possible (0%–100%).
Throughput Number of processes completing execution per time unit.
Turnaround Time Time from submission to completion. Includes: waiting to get into memory + ready queue wait + CPU execution + I/O time.
Waiting Time Total time a process has been waiting in the ready queue.
Response Time Time from when a request was submitted until the first response is produced (not output).
Optimization Goals: Maximize CPU utilization & throughput. Minimize turnaround time, waiting time, and response time.

First-Come, First-Served (FCFS)

  • Simplest CPU scheduling algorithm.
  • Process requesting the CPU first is allocated the CPU first.
  • Nonpreemptive — implemented via a FIFO queue.
  • New PCB is attached to the end of the queue; when CPU is free, head of queue gets the CPU.
  • Simple to write and understand, but average waiting time is often quite long.

Shortest-Job-First (SJF) Scheduling

  • Associates each process with the length of its next CPU burst.
  • Assigns CPU to the process with the smallest next CPU burst.
  • Optimal — gives minimum average waiting time for a given set of processes.
  • Difficulty: knowing the length of the next CPU request (could ask the user or estimate).
  • Can be either Preemptive or Nonpreemptive.
  • Preemptive version is called Shortest-Remaining-Time-First (SRTF).

Round Robin (RR) Scheduling

  • Designed specially for time-sharing systems.
  • Similar to FCFS but with preemption to switch between processes.
  • A small unit of time called a time quantum (q) is defined — usually 10–100 ms.
  • Each process gets at most q time units of CPU. After that, it is preempted and added to the end of the ready queue.
  • The ready queue is a FIFO circular queue.
  • Timer interrupts every quantum to schedule the next process.

Execution Logic

If burst time < time quantum Process releases CPU voluntarily. CPU takes the next process in queue.
If burst time > time quantum Timer fires → OS interrupt → context switch → process moved to end of ready queue.

Performance Characteristics

  • With n processes and time quantum q, each process gets 1/n of CPU time in chunks of at most q.
  • No process waits more than (n−1)×q time units until its next quantum.
  • Typically higher average turnaround than SJF, but better response time.
Effect of Time Quantum Size:
• q very large → behaves like FIFO/FCFS
• q very small → too many context switches → overhead too high
• Rule of thumb: 80% of CPU bursts should be shorter than q.
• Modern systems: q = 10–100 ms; context switch < 10 µs.

Priority Scheduling

  • A priority number (integer) is associated with each process.
  • CPU is allocated to the process with the highest priority (smallest integer = highest priority).
  • Can be Preemptive or Nonpreemptive.
  • Equal-priority processes scheduled in FCFS or Round Robin order.
  • SJF is a special case of priority scheduling — priority = inverse of predicted next CPU burst time.
Problem — Starvation:
Low-priority processes may never execute if high-priority processes keep arriving.
Solution — Aging:
As time progresses, increase the priority of waiting processes so they eventually get the CPU.

Multilevel Queue Scheduling

Processes are classified into groups with different response time requirements. The ready queue is partitioned into separate queues.

Foreground Queue (Interactive) Uses Round Robin scheduling. Gets 80% of CPU time.
Background Queue (Batch) Uses FCFS scheduling. Gets 20% of CPU time.
  • Processes are permanently assigned to a given queue.
  • Scheduling between queues uses fixed priority preemptive scheduling (foreground before background — risk of starvation) or time slicing.
Limitation: Low scheduling overhead but inflexible — processes cannot move between queues.

Multilevel Feedback Queue Scheduling

Solves the inflexibility of multilevel queue — processes can move between queues. Aging can be implemented this way to prevent starvation.

Defined by Five Parameters:

  • Number of queues
  • Scheduling algorithm for each queue
  • Method to determine when to upgrade a process (to higher priority queue)
  • Method to determine when to demote a process (to lower priority queue)
  • Method to determine which queue a process enters when it first needs service

Classic Example — Three Queues

Q0 RR with time quantum = 8 ms (highest priority)
Q1 RR with time quantum = 16 ms
Q2 FCFS (lowest priority)
How it works:
1. New job enters Q0 (RR, q=8). If it doesn't finish in 8 ms → moved to Q1.
2. In Q1 (RR, q=16). If it doesn't finish in 16 ms → moved to Q2.
3. In Q2 (FCFS) — runs only when Q0 and Q1 are empty.
Preemption Rule: A process in a lower priority queue will be preempted (only after its given time quantum) by a process arriving in a higher priority queue.
Short Processes: Complete quickly without migrating far down the hierarchy.
Long CPU-bound Processes: Danger of starvation on a system with many interactive processes.

Thread Scheduling

  • User-level threads are managed by thread APIs; kernel-level threads by the OS.
  • When threads are supported, threads are scheduled (not processes).
  • In many-to-one and many-to-many models, the thread library schedules user-level threads to run on LWPs (Lightweight Processes).
Process-Contention Scope (PCS) Scheduling competition is within the process. Thread library schedules user-level threads. Typically done via priority set by programmer.
System-Contention Scope (SCS) Kernel decides which kernel-level thread to schedule onto a CPU. One-to-one model uses only SCS. Linux and Mac OS X allow only PTHREAD_SCOPE_SYSTEM.

Pthread Scheduling API

  • PTHREAD_SCOPE_PROCESS — schedules threads using PCS scheduling.
  • PTHREAD_SCOPE_SYSTEM — schedules threads using SCS scheduling.
#include <pthread.h>
#include <stdio.h>
#define NUM_THREADS 5

int main(int argc, char *argv[]) {
    int i, scope;
    pthread_t tid[NUM_THREADS];
    pthread_attr_t attr;

    /* get default attributes */
    pthread_attr_init(&attr);

    /* inquire on current scope */
    if (pthread_attr_getscope(&attr, &scope) != 0)
        fprintf(stderr, "Unable to get scheduling scope\n");
    else {
        if (scope == PTHREAD_SCOPE_PROCESS)
            printf("PTHREAD_SCOPE_PROCESS");
        else if (scope == PTHREAD_SCOPE_SYSTEM)
            printf("PTHREAD_SCOPE_SYSTEM");
    }

    /* set the scheduling algorithm to SCS */
    pthread_attr_setscope(&attr, PTHREAD_SCOPE_SYSTEM);

    /* create the threads */
    for (i = 0; i < NUM_THREADS; i++)
        pthread_create(&tid[i], &attr, runner, NULL);

    /* join on each thread */
    for (i = 0; i < NUM_THREADS; i++)
        pthread_join(tid[i], NULL);
}

/* Each thread begins here */
void *runner(void *param) {
    /* do some work ... */
    pthread_exit(0);
}

Multiple-Processor Scheduling

CPU scheduling is more complex when multiple CPUs are available.

Homogeneous Processors All processors are identical within the multiprocessor.
Asymmetric Multiprocessing Only one processor accesses system data structures and handles scheduling. Reduces data sharing needs.
Symmetric Multiprocessing (SMP) Each processor is self-scheduling. Can use a common ready queue or each processor has its own private queue. Currently most common.
In SMP, all threads may be in a common ready queue (a), or each processor may have its own private queue of threads (b).

Algorithm Evaluation

To select a CPU scheduling algorithm for an OS, first determine criteria then evaluate algorithms.

Deterministic Modeling

  • A type of analytic evaluation.
  • Takes a particular predetermined workload and defines the performance of each algorithm for that workload.
  • Simple and fast, but requires exact numbers for input — applies only to those specific inputs.
Example comparison for 5 processes arriving at time 0:
• FCFS → 28 ms average waiting time
• Non-preemptive SJF → 13 ms average waiting time
• Round Robin → 23 ms average waiting time

Rules:

  • New processes enter Queue 0.
  • If not finished in first Q0 iteration → moved to end of Queue 1.

  • If not finished in first Q1 iteration → moved to end of Queue 2.
