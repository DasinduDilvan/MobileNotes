💾 Virtual Memory

Lecture 09  ·  Operating Systems

1. Background

Definition: Virtual memory is the separation of a user's logical memory from physical memory, allowing a process to have an address space larger than the available physical RAM.

Key ideas:

  • Only part of the program needs to be in memory for execution.
  • Logical address space can be much larger than physical address space.
  • Allows address spaces to be shared by several processes.
  • Allows for more efficient process creation.
  • More programs can run concurrently.
  • Increased CPU utilization and throughput with no increase in response time or turnaround time.
  • Less I/O needed to load or swap processes.
Virtual Address Space: The logical view of how a process is stored in memory. Addresses typically start at 0 and are contiguous. Physical memory is organized in page frames. The MMU (Memory Management Unit) maps logical addresses to physical addresses.

Virtual memory can be implemented via:

Demand Paging Pages are loaded into memory only when they are needed.
Demand Segmentation Segments are loaded only when required by the process.

3. Demand Paging

Two approaches at load time:

  • Swapping: Bring the entire process into memory at load time.
  • Demand Paging: Bring a page into memory only when it is needed.

Advantages of Demand Paging:

  • Less I/O needed; no unnecessary I/O
  • Less memory needed
  • Faster response
  • Supports more concurrent users
When is a page "needed"? When there is a reference to it.
  • Invalid reference → abort the process
  • Not in memory → bring the page into memory
A pager is the swapper that deals with individual pages (rather than whole processes).

4. Valid–Invalid Bit

Each entry in the page table has a valid–invalid bit to track whether a page is currently in physical memory.

v — Valid (in-memory) The page is currently memory-resident. The MMU can translate the address normally.
i — Invalid (not in memory) The page is either not yet loaded or does not belong to the process's address space.
Initial State: All entries start with the valid–invalid bit set to i.
Page Fault Trigger: During MMU address translation, if the valid–invalid bit is i, a page fault is generated (a trap/interrupt to the OS).

5. Page Fault

Definition: A page fault occurs whenever a process accesses a logical memory page that does not currently reside in physical memory. The MMU causes a trap (interrupt), resulting in a context switch to the OS paging mechanism.

Steps in Handling a Page Fault:

1
Process references a page → MMU checks the page table entry and finds valid–invalid bit = i.
2
MMU generates a trap (interrupt) to the operating system.
3
OS checks another internal table:
  • If invalid reference → abort the process.
  • If just not in memory → proceed to load it.
4
OS finds (or creates) a free frame in physical memory.
5
Swap the missing page in from the backing store (disk) into the free frame.
6
Reset page table: Update the frame number; set valid–invalid bit = v.
7
Restart the instruction that caused the page fault.

6. Page Replacement

When There is No Free Frame: The OS must perform page replacement — find a page currently in memory that is not actively needed and evict it (write to disk if modified), freeing up the frame for the requested page.

Key concepts:

  • Victim frame: The page chosen to be replaced/evicted.
  • Modify (dirty) bit: Tracks whether a page has been modified. If not dirty, the page can be discarded without writing to disk — reduces overhead.
  • Page replacement completes the separation between logical and physical memory: large virtual memory can be provided on smaller physical memory.
Goal: Minimize the number of page faults. The same page may be brought into memory several times during a process's lifetime.

Basic Page Replacement Steps:

1
Find the location of the desired page on secondary storage.
2
Find a free frame:
  • If a free frame exists → use it.
  • If no free frame → use a page-replacement algorithm to select a victim frame.
  • Write the victim frame to secondary storage if necessary; update page and frame tables.
3
Read the desired page into the newly freed frame; update page and frame tables.
4
Continue the process from where the page fault occurred.

7. Page Replacement Algorithms

Evaluation Method: Algorithms are evaluated by running them on a reference string (sequence of page numbers accessed) and counting the number of page faults.

Reference string used in examples: 7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
Repeated access to the same page does NOT cause a page fault. Results depend on the number of frames available.
General Principle: As the number of available frames increases, the number of page faults drops toward a minimal level.

7.1 First-In-First-Out (FIFO)

Strategy: Replace the page that has been in memory the longest (the oldest page).
  • Simple to implement — maintain a queue of pages in memory.
  • Example with 3 frames on reference string above: 15 page faults.
  • Suffers from Belady's Anomaly.
Belady's Anomaly: For some page-replacement algorithms (including FIFO), the page-fault rate may increase as the number of allocated frames increases. This is counter-intuitive — more frames should mean fewer faults, but FIFO doesn't guarantee this. Example: with string 1,2,3,4,1,2,5,1,2,3,4,5, going from 3 → 4 frames increases faults from 9 to 10.

7.2 Optimal Algorithm (OPT / MIN)

Strategy: Replace the page that will not be used for the longest period of time in the future.
  • Example with 3 frames: 9 page faults — the theoretical minimum.
  • No replacement algorithm can do better (fewest possible page faults).
  • Cannot be implemented in practice — requires future knowledge of the reference string.
  • Used mainly as a benchmark for comparison studies to evaluate how close other algorithms come to optimal.
OPT does not suffer from Belady's Anomaly.

7.3 Least Recently Used (LRU)

Strategy: Replace the page that has not been used for the longest time in the past. Uses past knowledge as an approximation of future use.
  • Associates a timestamp of last use with each page.
  • Example with 3 frames: 12 page faults.
  • Better than FIFO, worse than OPT.
  • Generally a good algorithm and frequently used in practice.
  • Does not suffer from Belady's Anomaly.

9. Thrashing

Definition: A process is thrashing if it is spending more time paging than executing. This causes severe performance degradation.

How thrashing occurs:

  • A process does not have enough frames to support the pages it actively uses.
  • The process quickly triggers a page fault.
  • It must replace some page — but since all its pages are in active use, it must replace one that will be needed again right away.
  • This leads to a cycle of repeated page replacements.
Result: The system is almost entirely occupied with paging operations rather than executing useful work. CPU utilization drops dramatically, and the system appears to hang or run extremely slowly.
Root Cause: Too many processes in memory competing for too few frames. Each process needs more frames than it has been allocated for its working set (the set of pages it actively references).

📖 Key Terms Quick Reference

Virtual Memory Separation of logical memory from physical memory; allows address space larger than RAM.
MMU Memory Management Unit — hardware that maps logical addresses to physical addresses.
Demand Paging Load pages into memory only when they are referenced, not at load time.
Pager A swapper that deals specifically with pages (not whole processes).
Valid–Invalid Bit A per-page flag: v = page in memory; i = page not in memory.
Page Fault Occurs when a referenced page is not in physical memory; triggers OS to load it.
Page Replacement When no free frame exists, an existing page (victim) is evicted to make room.
Dirty / Modify Bit Tracks whether a page was modified; dirty pages must be written to disk before eviction.
Belady's Anomaly Phenomenon (in FIFO) where more frames can lead to more page faults.
Reference String Sequence of page numbers used to evaluate page replacement algorithms.
Backing Store Disk/secondary storage where pages are stored when not in physical memory.
Thrashing A state where a process spends more time paging than executing — severe performance loss.
