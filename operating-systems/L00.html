Lecture 01 — Introduction to Operating Systems

What an OS is, its components, startup, interrupts, storage, and architecture fundamentals.

What is an Operating System?

Definition: An OS is an intermediary between the user and hardware that executes user programs, makes the computer convenient to use, and uses hardware efficiently.
Resource AllocatorManages all resources: CPU time, memory space, file storage, I/O devices.
Control ProgramControls execution of programs to prevent errors and improper use.
KernelThe one program running at all times on the computer. System programs ship with the OS but are not part of the kernel.
Without an OS, a computer is just useless hardware.

Computer System Components

HardwareCPU, Memory, I/O devices — provides basic computing resources.
Operating SystemControls and coordinates hardware use among applications.
Application ProgramsWord processors, compilers, web browsers, games.
UsersPeople, machines, other computers.

Computer Startup Process

1
Bootstrap Program loaded at power-up or reboot — stored in ROM/EPROM (firmware).
2
Initializes all aspects of the system.
3
Loads the OS kernel into memory.
4
Kernel starts the initial process; system is now running and ready.

Interrupts

Hardware InterruptTriggered by hardware devices via the system bus.
Software Interrupt (Trap)Triggered by software via a system call.
  • CPU stops current execution and transfers to the interrupt service routine.
  • CPU saves its current state before handling the interrupt.
  • Interrupt vector: contains addresses of interrupt service routines.
  • The OS is interrupt-driven — it waits for interrupts to occur.
DMA (Direct Memory Access): Used for high-speed I/O — device controller transfers data directly to/from memory without CPU intervention. Only one interrupt per block of data; CPU is free for other tasks.

Storage Hierarchy & Concepts

Volatile MemoryLoses contents when power is removed (RAM).
Non-volatile MemoryRetains contents without power (ROM, hard disk, SSD).
Caching: Copies information into faster storage. CPU checks cache before main memory. Based on the principle of locality of reference.

Computer System Architecture

Single ProcessorOne main CPU executing instructions.
SMP (Symmetric Multiprocessing)All processors are peers; no master-slave relationship.
AMP (Asymmetric)Master-slave relationship among processors.

Advantages of multiprocessing: increased throughput, economy of scale, increased reliability and fault tolerance.

Multiprogramming & Timesharing

MultiprogrammingMultiple jobs in memory; CPU switches between jobs to increase utilization. Job scheduling selects which job to run next.
Timesharing (Multitasking)CPU switches rapidly; response time < 1 second; enables interactive computing for multiple users simultaneously.
Virtual memory allows execution of processes not completely in memory.

Dual Mode Operation

User Mode (bit = 1)User applications execute here. Cannot run privileged instructions.
Kernel Mode (bit = 0)OS kernel executes here. Can run all privileged instructions.
  • System calls cause a mode switch from user → kernel mode.
  • After system call completion, returns to user mode.
  • Protects OS and critical resources from user programs.

OS Management Responsibilities

Process ManagementCreate/delete, suspend/resume, synchronization, communication, deadlock handling, CPU scheduling.
Memory ManagementTrack usage, allocate/deallocate, move data between memory and storage, manage virtual memory.
File ManagementCreate/delete files and directories, map files to storage, backup, access control.
Mass StorageDisk scheduling, partitioning, protection, free-space management.
I/O SubsystemDevice drivers, buffering, caching, spooling, interrupt handling.
Protection & SecurityUser IDs, group IDs, access control, authentication, encryption.

Lecture 02 — Operating-System Structures

OS services, CLI, GUI, system calls, APIs, and OS structural approaches.

Two Major Tasks of an OS

User-Centric ServicesProvide convenient environment, execute programs, I/O operations, file-system manipulation, communication, error detection. Visible to users.
System ManagementResource allocation, logging, protection & security. Not much visible to users — behind-the-scenes operations.

User Interface Types

CLI (Command-Line)Direct command entry. Implemented in kernel or as shell programs. UNIX shells: Bourne, C shell, BASH, Korn.
GUI (Graphical)Desktop with mouse-based window-and-menu system. Windows, Mac Aqua, GNOME/KDE on Linux.
TouchscreenGesture-based actions, virtual keyboard, voice commands. Required for mobile devices.
CLI commands: either built-in (inside the interpreter) or program files (named system programs, e.g., rm filename.txt).

System Calls

Definition: Programming interface to the services provided by the OS. Typically written in C/C++. Mostly accessed via an API rather than direct use.
Win32 APIFor Windows systems.
POSIX APIFor UNIX, Linux, Mac OS X.
Java APIFor the JVM.
Why use APIs over direct system calls? Portability across systems, easier to use, OS details hidden, better abstraction.

Example call chain: Program calls printf() → C library → write() system call → OS kernel executes operation.

System-call interface: maintains a table indexed by system call numbers; each entry = (number, address of system call function).

Types of System Calls

Process Controlcreate/terminate, load/execute, get/set attributes, wait, allocate/free memory.
File Managementcreate/delete, open/close, read/write/reposition, get/set attributes.
Device Managementrequest/release device, read/write/reposition, get/set attributes.
Information Maintenanceget/set time & date, system data, process/file/device attributes.
Communicationscreate/delete connection, send/receive messages, shared memory, transfer status.
Protectioncontrol access, get/set permissions, allow/deny user access.

OS Structure Approaches

Simple / Monolithic (MS-DOS)Not divided into modules. Interfaces and levels not well separated. Application programs can access I/O directly → system-wide crashes on user failure. Lacks modularity and protection.
Layered ApproachOS divided into layers, each built on top of lower layers. Bottom = hardware; top = user interface. Simpler debugging and verification, but less efficient and harder to define layer boundaries.
Traditional UNIXBeyond simple but not fully layered. Kernel provides everything below the system-call interface: file system, CPU scheduling, memory management.
Microkernel (Mach)Removes nonessential components from the kernel; implements them as user-level programs. Communication via message passing. Easier to extend and port; more reliable and secure. Performance overhead from user↔kernel communication.

Lecture 03 — Processes

Process concept, memory layout, states, PCB, scheduling, creation, termination, and IPC.

Process Concept

Program (Passive)Entity stored on disk; executable file; static code.
Process (Active)Program loaded into memory and executing. Has a program counter and resources. Must execute sequentially.
Memory layout of a process (high → low): Stack → ↕ Heap → Data → Text (code).
Stack = function params, return addresses, local variables. Heap = dynamic allocation. Data = global variables. Text = program code.

Multiple processes can be associated with the same program — the text sections are the same, but data/heap/stack sections vary.

Process States

newProcess is being created.
readyWaiting to be assigned to a processor.
runningInstructions are being executed.
waitingWaiting for some event to occur (e.g., I/O).
terminatedProcess has finished execution.
Key rule: Only ONE process may be in the running state on a single-CPU system. Many can be in ready and waiting states.

State transitions: new→ready (admitted) · ready→running (scheduler dispatch) · running→ready (interrupt) · running→waiting (I/O or event wait) · waiting→ready (I/O completion) · running→terminated (exit)

Process Control Block (PCB)

Also called task control block. Stores all information associated with a process.

Process StateRunning, waiting, etc.
Program CounterLocation of next instruction.
CPU RegistersContents of all process-centric registers.
CPU Scheduling InfoPriorities, scheduling queue pointers.
Memory-Management InfoMemory allocated to the process.
Accounting InfoCPU used, clock time, process numbers.
I/O Status InfoI/O devices allocated, list of open files.

Schedulers

Short-term (CPU Scheduler)Selects which process to execute next and allocates CPU. Invoked frequently (ms). Must be fast.
Long-term (Job Scheduler)Selects which processes enter the ready queue. Invoked infrequently (seconds/minutes). Controls degree of multiprogramming.
Intermediate (Swapping)Reduces degree of multiprogramming. Swaps processes out to disk and back in. Saves and restores process status.
I/O-bound ProcessSpends more time doing I/O; many short CPU bursts.
CPU-bound ProcessSpends more time computing; generates I/O infrequently.

Context Switch

Definition: Saving the state of the old process (into its PCB) and loading the saved state of the new process. Context-switch time is pure overhead — no useful work is done during switching.

The more complex the OS and PCB → the longer the context switch. Time is hardware-dependent.

Process Creation & Termination

Creation: Parent creates children → tree of processes. Each identified by a pid.

  • fork() — creates new process (duplicate of parent).
  • exec() — replaces process memory with a new program.
  • wait() — parent waits for child to terminate.

Termination:

  • Normal: exit() — returns status to parent; OS deallocates resources.
  • Parent-initiated: abort() — when child exceeds resources or task is no longer needed.
  • Cascading termination: if parent exits, all children are also terminated.
  • Zombie: process finished but parent hasn't called wait().
  • Orphan: parent terminated without calling wait().

Interprocess Communication (IPC)

Shared MemoryAn area of memory shared among processes. Communication under control of user processes (not OS). Issue: synchronization when accessing shared memory.
Message PassingProcesses communicate without shared variables via send(message) and receive(message). Message size: fixed or variable.
Producer-Consumer Problem: Producer produces data; consumer consumes it via a shared buffer. Bounded buffer (fixed size — producer waits if full) vs. unbounded buffer (producer never waits).
Direct CommunicationProcesses name each other explicitly: send(P, msg), receive(Q, msg). One link per pair.
Indirect CommunicationMessages sent to/from mailboxes (ports) with unique IDs. Multiple processes can share a mailbox.

Lecture 04 — Threads

Thread concepts, multithreading models, thread libraries, mutexes, and threading issues.

What is a Thread?

Definition: A thread is a basic unit of CPU utilization. A multithreaded process can perform more than one task concurrently.
Each Thread Has (Private)Thread ID · Program Counter · Register Set · Stack
Threads in Same Process ShareCode section · Data section · OS resources (open files, signals)
Key insight: Process creation is heavyweight (time-consuming and resource-intensive); thread creation is lightweight.

Benefits of Multithreading

ResponsivenessContinued execution if part of process is blocked. Crucial for UIs.
Resource SharingThreads share process resources — easier than shared memory or message passing.
EconomyThread creation is cheaper than process creation. Lower context-switch overhead.
ScalabilityThreads run in parallel on multiple cores in multiprocessor architectures.

Parallelism vs. Concurrency

ParallelismMore than one task executes simultaneously. Requires multi-core hardware.
ConcurrencySupports more than one task by allowing all tasks to make progress. Can occur on a single-core (via rapid switching).
Data ParallelismDistributes subsets of the same data across cores, performing the same operation on each.
Task ParallelismDistributes threads across cores; each thread performs a unique operation.

Multithreading Models

ModelDescriptionExamplesDrawback
Many-to-OneMany user threads → one kernel threadSolaris Green Threads, GNU Portable ThreadsOne block = entire process blocks; no true parallelism
One-to-OneEach user thread → one kernel threadWindows, Linux, Solaris 9+Each user thread creates a kernel thread (overhead)
Many-to-ManyMany user threads → many kernel threadsSolaris <9, Windows+ThreadFiberDifficult to implement
Two-LevelLike M:M but allows binding user↔kernel threadIRIX, HP-UX, Solaris 8Complex
In practice today: Most modern OSes use the One-to-One model (Windows, Linux) because increasing core counts have reduced the need to limit kernel threads.

Mutexes & Race Conditions

Race Condition: Two or more threads access and modify the same shared resource simultaneously, and the result depends on execution order.
Mutex (Mutual Exclusion Lock): Blocks access to variables by other threads — enforces exclusive access. Protects the critical region. Only applicable to threads in a single process (unlike semaphores).
Without Mutex (Unsafe)int counter = 0;
Both threads read same value → write 1 → final value: 1 instead of 2.
With Mutex (Safe)pthread_mutex_lock(&mutex1);
counter++;
pthread_mutex_unlock(&mutex1);
Exclusive access → final value: 2 ✓

Threading Issues

fork() & exec()Does new process duplicate all threads or only the calling thread? exec() normally replaces all threads.
Signal HandlingSignals notify a process of an event. In multithreaded processes: deliver to thread it applies to, every thread, certain threads, or one designated thread.
Thread CancellationAsynchronous: immediately terminate target thread. Deferred: thread checks periodically at cancellation points.
Thread-Local Storage (TLS)Each thread has its own copy of data. Visible across function invocations (unlike local variables). Unique per thread (unlike static data).
Thread SchedulingPreemptive: higher-priority thread preempts current. Starvation risk for low-priority threads. Aging: increase priority of waiting threads over time.
Thread PoolsPre-create threads waiting for work. Faster than creating new thread per request. Bounds maximum thread count (prevents resource exhaustion).
Fork-Join Parallelism: Main thread forks tasks → tasks execute in parallel → results joined back. Large problems recursively broken into subtasks. General pattern: if small → solve directly; else fork subtasks, join results.
Linux Threads: Linux refers to threads as tasks. Created via clone() system call, which allows a child task to share the address space of the parent.

Lecture 05 — CPU Scheduling

CPU burst cycle, scheduling algorithms, criteria, dispatcher, and multi-processor scheduling.

Background

CPU–I/O Burst Cycle: Process execution alternates between CPU execution (burst) and I/O wait (burst). Many short CPU bursts + few long CPU bursts. Goal: maximize CPU use by switching processes during I/O waits.
Dispatcher: Gives control of the CPU to the selected process. Involves: switching context, switching to user mode, jumping to the proper location. Dispatch latency = time to stop one process and start another — should be minimal.

Scheduling Decision Points

Nonpreemptive (no choice)1. Running → Waiting (I/O request)
4. Terminates
A new process must be selected.
Preemptive (choice exists)2. Running → Ready (interrupt)
3. Waiting → Ready (I/O completion)
Choice of which process runs next.

Scheduling Criteria

CriterionDescriptionGoal
CPU UtilizationKeep CPU as busy as possible (0–100%)Maximize
ThroughputProcesses completing per time unitMaximize
Turnaround TimeSubmission to completion (includes all wait + exec + I/O)Minimize
Waiting TimeTotal time in the ready queueMinimize
Response TimeTime from request to first response producedMinimize

Scheduling Algorithms

FCFS — First-Come, First-Served

  • Nonpreemptive; implemented via FIFO queue.
  • Simple but average waiting time can be long.
  • Convoy Effect: Short processes wait for one long process (I/O-bound processes stuck behind CPU-bound).
Example: P1(24ms), P2(3ms), P3(3ms) arriving in order → Avg wait = 17ms. If reversed → Avg wait = 3ms.

SJF — Shortest Job First

  • Assigns CPU to process with the smallest next CPU burst.
  • Optimal — gives minimum average waiting time.
  • Difficulty: knowing the length of the next CPU burst (must estimate).
  • Can be preemptive or nonpreemptive.
  • Preemptive version = SRTF (Shortest-Remaining-Time-First).

Round Robin (RR)

  • Each process gets at most q time units (time quantum, usually 10–100 ms).
  • After q ms, preempted and added to end of the ready queue (circular FIFO).
  • No process waits more than (n−1) × q time units.
  • q very large → behaves like FCFS. q very small → too many context switches (overhead).
  • Rule of thumb: 80% of CPU bursts should be shorter than q.

Priority Scheduling

  • CPU allocated to process with highest priority (smallest integer = highest priority).
  • Preemptive or nonpreemptive. SJF is a special case (priority = inverse of burst time).
  • Starvation: low-priority processes may never execute.
  • Aging: gradually increase the priority of waiting processes.

Multilevel Queue Scheduling

  • Ready queue partitioned into separate queues (e.g., foreground/background).
  • Processes permanently assigned to a queue.
  • Foreground (interactive): RR, 80% CPU. Background (batch): FCFS, 20% CPU.
  • Limitation: Inflexible — processes cannot move between queues.

Multilevel Feedback Queue

  • Processes can move between queues. Implements aging. Most flexible.
  • Defined by: number of queues, algorithm per queue, upgrade/demote rules, entry method.
Classic 3-queue example: Q0 (RR, q=8ms) → Q1 (RR, q=16ms) → Q2 (FCFS). New jobs enter Q0. If not finished → move to Q1. If not finished → move to Q2. Lower queue runs only when higher queues are empty.

Algorithm Comparison Summary

AlgorithmPreemptive?Starvation?Key Property
FCFSNoNoConvoy effect; simple
SJF / SRTFOptionalYesOptimal avg wait; needs future knowledge
Round RobinYesNoFair; response time depends on q
PriorityOptionalYes (fix with aging)Priority-based; SJF is a special case

Lecture 06 — Process Synchronization

Race conditions, critical section, Peterson's solution, semaphores, and classical synchronization problems.

Background

Problem: Concurrent access to shared data may result in data inconsistency. Processes can be interrupted at any time, leading to partially completed executions on shared data.
Race Condition: Several processes access and manipulate the same data concurrently, and the outcome depends on the particular order in which the resource is accessed.

Critical Section Problem

Critical Section: A segment of code where shared variables, tables, or files may be modified. When one process is in its CS, no other process may enter its CS.

Structure of process Pᵢ:

Entry Section
→
Critical Section
→
Exit Section
→
Remainder Section

3 Requirements for a Valid CS Solution

1. Mutual ExclusionIf Pᵢ is in CS, no other process can be in CS simultaneously.
2. ProgressIf no process is in CS and some wish to enter, the selection must be decided in finite time.
3. Bounded WaitingA bound exists on how many times others can enter CS after a request is made but before it is granted.

Peterson's Solution

Software solution for two processes Pᵢ and Pⱼ. Assumes load/store instructions are atomic. Not guaranteed on modern hardware due to CPU instruction reordering.

Shared Variables:

  • int turn — indicates whose turn it is (turn == i → Pᵢ may enter CS).
  • boolean flag[2] — flag[i] = true means Pᵢ is ready to enter CS.

Algorithm for Pᵢ: Set flag[i]=true; set turn=j; busy-wait while (flag[j] && turn==j); enter CS; set flag[i]=false.

Proves all 3 requirements: Mutual exclusion (both cannot be in CS simultaneously because of turn), Progress (while loop eventually becomes false), Bounded waiting (Pᵢ waits at most one entry by Pⱼ).

Mutex Locks & Semaphores

Mutex Lock:

  • acquire() — busy-waits until lock available, then sets unavailable.
  • release() — sets lock back to available.
  • Uses busy waiting → called a spinlock.
  • Calls must be atomic (via hardware instructions).

Semaphore (S = integer):

  • wait(S) — also P(): while (S≤0) busy-wait; S--
  • signal(S) — also V(): S++
  • Binary semaphore: 0 or 1 (like mutex).
  • Counting semaphore: unrestricted; used for resource allocation (initialized to # of resources).

Classical Synchronization Problems

🔲 Bounded-Buffer Problem

Semaphores: mutex=1 (mutual exclusion), full=0 (count of full buffers), empty=n (count of empty buffers).
Producer: wait(empty) → wait(mutex) → add item → signal(mutex) → signal(full).
Consumer: wait(full) → wait(mutex) → remove item → signal(mutex) → signal(empty).

📖 Readers-Writers Problem

Multiple readers may read simultaneously. Only one writer at a time; when writing, no readers or other writers allowed.
Variables: rw_mutex=1 (writer access), mutex=1 (protects read_count), read_count=0.
First reader: wait(rw_mutex). Last reader: signal(rw_mutex). Writer: wait(rw_mutex) … signal(rw_mutex).

🍜 Dining-Philosophers Problem

5 philosophers, 5 chopsticks. Each must acquire both left and right chopstick to eat.
Semaphore: chopstick[5] initialized to 1.
Philosopher i: wait(chopstick[i]) → wait(chopstick[(i+1)%5]) → eat → signal both.
Problem: If all 5 pick up left chopstick simultaneously → Deadlock.

Problems with Semaphores

signal before waitsignal(mutex)…wait(mutex)Multiple processes in CS simultaneously → violates mutual exclusion.
Double waitwait(mutex)…wait(mutex)Deadlock — no one can release the lock.
Omitting wait/signalForgetting either → mutual exclusion violated or deadlock.
DeadlockTwo or more processes waiting indefinitely for events that can only be caused by waiting processes.
StarvationProcess waits indefinitely; may never be removed from the semaphore queue.

Lecture 07 — Deadlocks

Deadlock characterization, resource-allocation graphs, prevention, avoidance (Banker's), detection, and recovery.

System Model & Definition

Deadlock: Every process in a set is waiting for an event that only another process in the set can cause. System consists of finite resources (CPU cycles, memory, I/O devices), each with a number of instances.
Request
→
Use
→
Release
Real-world analogy: Bridge crossing — each section is a resource. Deadlock resolved by backing up (preemption). Starvation possible. Most OSes (UNIX/Linux) ignore deadlocks, assuming they are rare.

4 Necessary Conditions for Deadlock

1. Mutual ExclusionOnly one process can use a resource at a time (non-sharable resource).
2. Hold and WaitProcess holds at least one resource while waiting for additional resources held by others.
3. No PreemptionResources released only voluntarily by the process holding them.
4. Circular WaitSet {P₀,P₁,…,Pₙ} where each Pᵢ waits for a resource held by P(i+1 mod n).
All four conditions must hold simultaneously for deadlock to exist. Prevention = negate at least one condition.

Resource-Allocation Graph (RAG)

Request EdgePᵢ → Rⱼ : Process Pᵢ requests an instance of resource Rⱼ.
Assignment EdgeRⱼ → Pᵢ : An instance of Rⱼ is assigned to Pᵢ.
No cycleNo deadlock.
Cycle + single instance per resourceDeadlock.
Cycle + multiple instancesPossible deadlock (not certain).
Wait-for Graph: Derived from RAG by removing resource nodes. Edge Pᵢ → Pⱼ means Pᵢ is waiting for Pⱼ. Cycle in wait-for graph → deadlock (for single-instance resources). Detection requires O(n²) cycle check.

Deadlock Prevention

Condition to BreakMethodIssues
Mutual ExclusionNot required for sharable resources (e.g., read-only files). Must hold for non-sharable.Cannot always avoid
Hold & WaitRequest all resources before execution OR release all held resources before requesting more.Low utilization; starvation possible
No PreemptionIf request cannot be granted, release all held resources. Added to waiting list.Rollback overhead
Circular WaitImpose total ordering on resource types. Processes must request in increasing order.May limit flexibility

Deadlock Avoidance — Safe State & Banker's Algorithm

Safe State: A sequence <P₁, P₂, …, Pₙ> exists such that each process's remaining needs can be satisfied by currently available + resources held by earlier processes. Safe state → no deadlock. Avoidance ensures system never enters unsafe state.

Banker's Algorithm (Multiple Instances)

AvailableVector length m — resources currently available.
Maxn×m matrix — max demand of each process.
Allocationn×m matrix — resources currently allocated.
Needn×m matrix — Need = Max − Allocation.
Safety Algorithm: Find a process with Need ≤ Available; simulate its completion (add allocation to Available); repeat. If all Finish[i]=true → safe state.
Resource-Request Algorithm: If Request ≤ Need AND Request ≤ Available → pretend allocation → run safety algorithm. If safe → grant; else → restore state and wait.

Deadlock Detection & Recovery

Detection Algorithm (Multiple Instances):

  1. Work = Available; Finish[i] = false if Allocation[i] ≠ 0.
  2. Find i: Finish[i]==false AND Requestᵢ ≤ Work.
  3. Work += Allocationᵢ; Finish[i] = true. Go to step 2.
  4. If Finish[i]==false for any i → those processes are deadlocked.

Complexity: O(m × n²).

Recovery Options:

Process TerminationAbort all deadlocked processes, OR abort one at a time until cycle is eliminated. Selection: consider priority, computation time, resources used.
Resource PreemptionSelect victim (minimize cost). Rollback to safe state. Starvation risk → include rollback count in cost factor.

Lecture 08 — Main Memory Management

Address binding, MMU, dynamic loading, contiguous allocation, fragmentation, and paging.

Background & Memory Protection

Requirement: Programs must be in main memory (at least partially) to execute. Each process must have a separate memory space to protect processes from each other.
Base RegisterHolds the smallest legal physical memory address for a process.
Limit RegisterSpecifies the size of the valid address range.
CPU checks every user-mode memory access: if address ≥ base + limit → trap to OS (fatal error). Prevents user programs from accessing OS or other processes' memory.

Address Binding

Compile TimeAbsolute code generated. If memory location changes, must recompile.
Load TimeRelocatable code. Binding occurs at load time by the loader.
Execution TimeBinding delayed until run-time. Requires MMU hardware. Used by most modern OSes.
Logical Address (Virtual): Generated by CPU; relative address. Physical Address: Seen by memory unit; absolute address. The MMU maps logical → physical at run-time. User programs deal only with logical addresses.
Dynamic relocation (relocation register): physical address = logical address + relocation register value.

Dynamic Loading

A routine is loaded into memory only when it is called. Main program loaded first; when a routine calls another, the calling routine checks if target is loaded. If not → relocatable linking loader loads it and updates address tables.
  • Advantage: unused routines are never loaded → better memory utilization.
  • No special OS support required; users can implement with library routines.
  • Different from paging — dynamic loading is user-managed; paging is OS-managed.

Contiguous Memory Allocation

Fixed-Size PartitionsMemory divided into static partitions. Simple but suffers from internal fragmentation.
Variable Partitions (Dynamic)Allocate exactly needed size. Hole table tracks free blocks. External fragmentation occurs over time.

Hole-Selection Strategies

First-FitAllocate the first hole big enough. Fast. Generally good performance.
Best-FitAllocate the smallest hole that fits. Minimizes leftover. Must search entire list. Generally good performance.
Worst-FitAllocate the largest hole. Leaves largest leftover. Worst performance overall.

Fragmentation

External FragmentationTotal free memory exists but is not contiguous; scattered holes. Cannot satisfy a request even though total free space is enough.
Internal FragmentationAllocated memory slightly larger than requested; unused space inside the allocated region (e.g., fixed partition larger than process needs).
Compaction: Shuffle memory contents to combine free blocks into one large hole. Reduces external fragmentation. Only possible with dynamic (execution-time) relocation. Requires updating base registers.
Alternative to compaction: Allow non-contiguous logical address space → Paging and Segmentation.

Paging

Basic Idea: Break physical memory into fixed-size blocks called frames; break logical memory into same-size blocks called pages. Frame size = power of 2 (512B – 16MB).
  • To run a program of N pages → find N free frames and load pages into them.
  • Page table (per process): maps logical page number → physical frame number. Set up by OS, translated by hardware (MMU).
  • No external fragmentation. Internal fragmentation possible on the last page.
  • Longer memory access time due to page table lookups.
Address Translation: Logical address = [page number (p) | offset (d)]. MMU uses p to index page table → gets frame number f. Physical address = (f × frame size) + d.
Free-frame list: OS maintains a list of all unused frames. When process requests memory → allocate from free list; update page table. When process terminates → frames returned to free list.
Paging ✓No external fragmentation; supports virtual memory; efficient memory use.
Paging ✗Internal fragmentation (last page); page table overhead; longer access time.

Lecture 09 — Virtual Memory

Demand paging, valid-invalid bit, page faults, page replacement algorithms, and thrashing.

1. Background

Definition: Virtual memory is the separation of a user's logical memory from physical memory, allowing a process to have an address space larger than the available physical RAM.
Only part of the program needs to be in memory for execution.
Logical address space can be much larger than physical address space.
Allows address spaces to be shared by several processes.
More programs running concurrently.
Increased CPU utilization and throughput with no increase in response/turnaround time.
Less I/O needed to load or swap processes.
Demand PagingPages loaded into memory only when they are referenced.
Demand SegmentationSegments loaded only when required by the process.

3. Demand Paging & Valid–Invalid Bit

Pager: A swapper that deals specifically with individual pages (not whole processes). A page is needed when there is a reference to it. Invalid reference → abort. Not in memory → bring to memory.
v — Valid (in-memory)Page is memory-resident. MMU translates address normally.
i — Invalid (not in memory)Page not yet loaded or does not belong to process's address space.
Initial State: All entries start with bit = i. During MMU translation, if bit = i → page fault (trap to OS).

4. Page Fault Handling Steps

1

2
MMU generates a trap (interrupt) to the OS.
3
OS checks its table: invalid reference → abort; just not in memory → proceed.
4
OS finds a free frame in physical memory.
5
Swap the missing page in from backing store (disk) into the free frame.
6
Reset page table: update frame number; set valid–invalid bit = v.
7
Restart the instruction that caused the page fault.

5. Page Replacement

When no free frame exists: OS must perform page replacement — evict a victim page (write to disk if modified), freeing a frame for the requested page.
Victim FrameThe page chosen to be replaced/evicted from physical memory.
Modify (Dirty) BitIf page was not modified → discard without writing to disk. Reduces I/O overhead.

Basic Page Replacement Steps

1
Find location of desired page on secondary storage.
2
Find free frame (or select victim and write it to disk if necessary). Update tables.
3
Read desired page into freed frame; update page and frame tables.
4
Continue process from where page fault occurred.

6. Page Replacement Algorithms

Evaluation: Run algorithm on a reference string and count page faults. Reference string used: 7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1. Repeated access to same page = no fault. More frames → fewer faults (generally).

FIFO — First-In-First-Out

  • Replace the oldest page in memory. Maintain a queue.
  • 3 frames on reference string → 15 page faults.
Belady's Anomaly: For FIFO, adding more frames can increase page faults. Example: 3→4 frames on string 1,2,3,4,1,2,5,1,2,3,4,5 increases faults from 9 to 10.

OPT — Optimal Algorithm

  • Replace the page that will not be used for the longest period in the future.
  • 3 frames → 9 page faults (theoretical minimum — cannot do better).
  • Cannot be implemented in practice — requires future knowledge.
  • Used as a benchmark for comparison studies.
  • Does NOT suffer from Belady's Anomaly.

LRU — Least Recently Used

  • Replace the page not used for the longest time in the past. Associates a timestamp with each page.
  • 3 frames → 12 page faults. Better than FIFO, worse than OPT.
  • Generally a good, frequently used algorithm in practice.
  • Does NOT suffer from Belady's Anomaly.

7. Thrashing

Definition: A process is thrashing if it is spending more time paging than executing. Results in severe performance degradation — CPU utilization drops dramatically.

How it occurs:

  • Process doesn't have enough frames for the pages it actively uses.
  • Process quickly triggers a page fault.
  • Must replace a page — but all pages are active → replaced page needed again immediately.
  • Cycle of repeated page replacements → system almost entirely occupied with paging.
Root Cause: Too many processes in memory competing for too few frames. Each process needs more frames than allocated for its working set (set of pages it actively references).
