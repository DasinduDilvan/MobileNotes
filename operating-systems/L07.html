📘 Deadlocks in Operating Systems

Theory & Core Concepts: Prevention, Avoidance, Detection, Recovery
🚦 System Model & Resource Essentials

A deadlock occurs when every process in a set is waiting for an event that only another process in the set can cause. System consists of finite resources (CPU cycles, memory, I/O devices) with instances. Each resource type Ri has Wi instances.

📌 Bridge crossing analogy: Traffic one direction; each bridge section = resource. Deadlock resolved by backing up (preemption). Starvation possible. Most OSs ignore deadlocks (assume rare).

Resource utilization cycle: request → use → release. Example: 3 CD-RW drives, each process holds one and requests another → circular wait → deadlock.

⚙️ Deadlock Characterization (4 Necessary Conditions)

All four conditions must hold simultaneously for deadlock to exist:

Mutual exclusion
Only one process can use a resource at a time (non-sharable).
Hold and wait
Process holds at least one resource while waiting for additional resources held by others.
No preemption
Resources released only voluntarily by the process holding them.
Circular wait
Set {P₀, P₁,…, Pₙ} where each Pᵢ waits for resource held by Pᵢ₊₁ (mod n).
📊 Resource-Allocation Graph (RAG)

Vertices: Processes P = {P₁..Pₙ} and Resource types R = {R₁..Rₘ}. Edges: Request edge Pᵢ → Rⱼ (process requests instance). Assignment edge Rⱼ → Pᵢ (resource assigned).

🔵 Graph facts
• No cycle → no deadlock.
• Cycle + single instance per resource → deadlock.
• Cycle + multiple instances → possible deadlock.
🔄 Wait-for Graph
Derived from RAG by collapsing resources. Vertices = processes; edge Pᵢ → Pⱼ means Pᵢ waiting for Pⱼ. Cycle → deadlock (single-instance resources).
📌 Example with no deadlock despite cycle: If P₄ holds R₂ and may release it, then P₃ can acquire → cycle but no deadlock (multiple instances).
🛠️ Methods for Handling Deadlocks
  • Deadlock prevention: Negate at least one necessary condition.
  • Deadlock avoidance: Use additional a priori info (Banker's algorithm / RAG with claim edges).
  • Deadlock detection & recovery: Allow deadlock, then detect and recover.
  • Ignore deadlocks: Used by UNIX, most OSs; assume rare, cost of prevention high.
🛡️ Deadlock Prevention
💎 Mutual Exclusion
Not required for sharable resources (read-only files). Must hold for non-sharable.
🔄 Hold and Wait
Request all resources before execution OR request only when holding none. Low utilization, starvation possible.
⚡ No Preemption
If a request cannot be granted, release all held resources. Preempted resources added to waiting list.
🔁 Circular Wait
Impose total ordering of resource types. Process requests resources in increasing order.
Circular wait example ordering: Resources ordered (R1, R2, R3, R4). Process can only request R1 then R2 then R3 … sequence prevents cycles.
🧠 Deadlock Avoidance & Safe State

Requires a priori knowledge: each process declares maximum needs. System dynamically examines resource-allocation state (available, allocated, max demands) to avoid circular-wait.

✅ Safe state: A sequence <P₁, P₂, …, Pₙ> exists such that each process’s remaining needs can be satisfied by currently available + resources held by earlier processes.
Safe state → no deadlock. Unsafe state → may lead to deadlock. Avoidance ensures system never enters unsafe state.
Single instance per resource: Use Resource-Allocation Graph with claim edges (dashed line = future request). Grant only if conversion to assignment edge does not create cycle.
Multiple instances: Use Banker's Algorithm (Dijkstra).
🏦 Banker's Algorithm – Avoidance for Multiple Instances

Data structures: Available (vector length m), Max (n×m), Allocation (n×m), Need = Max – Allocation.
Safety algorithm: find process with Need ≤ Available, then simulate release (Available += Allocation), repeat. If all Finish[i]=true → safe state.

Resource-request algorithm (Requesti):
1. If Request ≤ Needi → proceed else error.
2. If Request ≤ Available → step 3, else wait.
3. Pretend allocation: Available -= Request; Allocation += Request; Need -= Request.
4. Run safety algorithm. If safe → allocate, else restore state and process waits.
📌 Classic example (10 A, 5 B, 7 C): Safe sequence <P₁, P₃, P₄, P₂, P₀> satisfies criteria. (Max – Allocation matrix yields feasible order). 
System snapshot T₀:
        Allocation   Max      Available
P0:     0 1 0       7 5 3    3 3 2
P1:     2 0 0       3 2 2
P2:     3 0 2       9 0 2
P3:     2 1 1       2 2 2
P4:     0 0 2       4 3 3
→ Need = Max-Alloc. Sequence verified.
🔍 Deadlock Detection

Single instance per resource: Maintain wait-for graph; cycle detection O(n²). Periodically search for cycle.

Multiple instances: detection algorithm similar to safety but uses Request matrix (current pending requests).

Detection algorithm steps:
• Work = Available; Finish[i] = false if Allocation[i] ≠ 0, else true.
• Find i: Finish[i]==false and Requestᵢ ≤ Work. If found: Work = Work + Allocationᵢ, Finish[i]=true.
• If Finish[i]==false for any i → deadlock exists; Pᵢ deadlocked.
Complexity: O(m × n²).
⚠️ Example: At time T₀, detection may find no deadlock (sequence P₀, P₂, P₃, P₁, P₄). Later, P₂ requests extra C → deadlock among P₁, P₂, P₃, P₄.
🔄 Recovery from Deadlock
Process Termination
• Abort all deadlocked processes.
• Abort one at a time until cycle eliminated.
Selection criteria: priority, computation time, resources used, interactive/batch.
Resource Preemption
• Select victim (minimize cost).
• Rollback to safe state / restart process.
• Starvation: include number of rollbacks in cost factor.

📚 Core Summary & Facts
  • Deadlock characterization: Mutual exclusion, hold and wait, no preemption, circular wait — all needed.
  • Resource-Allocation Graph: Cycle detection distinguishes single vs multiple instances.
  • Prevention: Break any of the 4 conditions (resource ordering, preemption, etc.).
  • Avoidance (Banker's): Safe sequence ensures no deadlock; pretends allocation and checks safety.
  • Detection + Recovery: Algorithms O(m×n²), then victim selection / rollback.
  • Most modern OS (Unix/Linux) ignore deadlocks for simplicity, assuming rare occurrence.
🌟 Exam ready: Safe state definition — there exists a process execution order where each process can obtain max resources without deadlock. Avoidance keeps system always in safe states.
🧩 Resource-Allocation Graph Scheme (Avoidance)

Introduce claim edge Pᵢ → Rⱼ (dashed line): future potential request. Convert to request edge when process actually requests, then assignment edge. When resource released, assignment edge reconverts to claim edge.
Cycle detection in graph with claim edges: Grant request only if converting to assignment edge does NOT create a cycle. Works for single-instance resources.

📘 Example: P1 and P2 claim R2. If P2 requests R2 but cycle appears after grant → unsafe → deny.
🚗 Bridge Crossing & Real-World View

Bridge sections as resources: deadlock resolved by one car backing up (preemption). Starvation may occur. OS perspective: prevention/avoidance costly → most systems (incl. UNIX) assume deadlocks are extremely rare.

Yet for critical systems (databases, some kernels), deadlock handling via timeout or wait-die schemes is used.
