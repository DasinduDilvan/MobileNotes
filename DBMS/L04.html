Example COMPANY Database

Requirements of the Company (oversimplified for illustrative purposes)
  • The company is organized into departments. Each department has a unique name, a unique number, and a particular employee who manages the department. The start date when the manager began is tracked. A department may have several locations.
  • A department controls a number of projects, each with a unique name, a unique number, and a single location.
  • Each employee's name, identity card number, address, salary, sex(gender), and birth date are stored. An employee is assigned to one department but may work on several projects. Hours per week and direct supervisor are also tracked.
  • Each employee's dependents are tracked for insurance — dependent's first name, sex(gender), birth date, and relationship to the employee.

E-R Model Concepts

Entities Specific objects or things in the miniworld with an independent existence represented in the database.

Ex: student, car, a company, a job, or a university course
Attributes Each entity has attributes — the particular properties that describe it.

Ex: an employee may be described by name, age, address, salary, and job

Types of Attributes

Type Description Example
Simple (Atomic) Not divisible into further components Sex / Gender
Composite Composed of several sub-components; may form a hierarchy Address (Street, City, State, Zip); Name (First, Middle, Last)
Single Valued Has a single value for a particular entity National Identity Card No
Multi Valued Can have a set of values for the same entity Color of a CAR; Degrees of a STUDENT
Stored Value cannot be determined from other attributes; must be stored directly Date of Birth
Derived Value can be determined from the values of other attributes Age (derived from Date of Birth)

Null Values

In some cases, a particular entity may not have an applicable value for an attribute.
1. Not Applicable The attribute simply does not apply to this entity.

Ex: Apartment Number — only applies to apartment buildings, not single-family homes
Ex: College Degree — only applies to people who have degrees
2. Unknown The value may exist but is missing, or it is not known whether the value exists.

Ex: Height of a person — value exists but is missing
Ex: Home Phone — not known whether value exists

Complex Attributes

Composite and multi-valued attributes may be nested arbitrarily to any number of levels, although this is rare.
Example: contact_details

contact_details(
    { phone_no },
    { address (
        street_address(no, street, apartment_no),
        city, state, zip
    ) }
)

Entity Types and Entity Sets

Entity Type (Schema / Intension) Defines a collection (or set) of entities that have the same attributes. Each entity type is described by its name and attributes.
Entity Set (Extension) The collection of all entities of a particular entity type in the database at any point in time. Usually referred to using the same name as the entity type.
Example:
Entity Type: EMPLOYEE (Name, Age, Salary) → Entity Set: (John Smith, 55, 80k), (Fred Brown, 40, 30K), (Judy Clark, 25, 20K)...

Entity Type: COMPANY (Name, Headquarters, President) → Entity Set: (Sunco Oil, Houston, John Smith), (Fast Computer, Dallas, Bob King)...

Key Attributes

  • An entity type usually has one or more attributes whose values are distinct for each individual entity in the entity set
  • Its values can be used to identify each entity uniquely
  • Sometimes several attributes together form a key, meaning that the combination of the attribute values must be distinct for each entity

Value Sets (Domain) of Attributes

  • Also called Domain of Values
  • Specifies the set of values that may be assigned to that attribute for each individual entity
  • The value set provides all possible values
  • Represents data from the current state of the miniworld
Example: Age — valid values are integers in the range 16–70

Preliminary Entity Designs — COMPANY Database

DEPARTMENT Attributes: Name, Number, Manager, Manager_start_date, {Locations}
PROJECT Attributes: Name, Number, Location, Controlling_department
EMPLOYEE Attributes: Ssn, Name (Fname, Minit, Lname), Sex, Salary, Address, Birth_date, Department, Supervisor, Works_on (Project, Hours)
DEPENDENT Attributes: Dependent_name, Sex, Birth_date, Relationship, Employee

Relationships and Relationship Types

A relationship relates two or more distinct entities with a specific meaning. Relationships of the same type are grouped into a relationship type.
  • Ex: EMPLOYEE John Smith WORKS_ON the ProductX PROJECT
  • Ex: EMPLOYEE Franklin Wong MANAGES the Research DEPARTMENT
Degree of a Relationship Type: The number of participating entity types.
Both MANAGES and WORKS_ON are binary relationships (degree 2).
Note: More than one relationship type can exist with the same participating entity types — e.g., MANAGES and WORKS_FOR are distinct relationships between EMPLOYEE and DEPARTMENT, but with different meanings.

Six Relationships in the COMPANY Database

All six are binary relationships (degree 2).
WORKS_FOR — between EMPLOYEE and DEPARTMENT
MANAGES — also between EMPLOYEE and DEPARTMENT (different meaning)
CONTROLS — between DEPARTMENT and PROJECT
WORKS_ON — between EMPLOYEE and PROJECT
SUPERVISION — between EMPLOYEE (as subordinate) and EMPLOYEE (as supervisor)
DEPENDENTS_OF — between EMPLOYEE and DEPENDENT

Constraints of Binary Relationship Types

Cardinality Ratios (Maximum Cardinality) Possible ratios for binary relationship types:

1:1 — One to One
1:N — One to Many
N:1 — Many to One
M:N — Many to Many
Participation Constraints (Minimum Cardinality) Two types:

Total Participation — every entity must participate (existence dependency)
Partial Participation — some entities may not participate

Cardinality & Participation Analysis

Relationship Cardinality Participation
MANAGES 1:1 (EMPLOYEE ↔ DEPARTMENT) EMPLOYEE: Partial | DEPARTMENT: Total (must always have a manager)
WORKS_FOR 1:N (DEPARTMENT ↔ EMPLOYEE) Both: Total
CONTROLS 1:N (DEPARTMENT ↔ PROJECT) PROJECT: Total | DEPARTMENT: Partial (some may control no projects)
SUPERVISION 1:N (EMPLOYEE supervisor ↔ EMPLOYEE supervisee) Both: Partial (not every employee is a supervisor or has a supervisor)
WORKS_ON M:N (EMPLOYEE ↔ PROJECT) Both: Total
DEPENDENTS_OF 1:N (EMPLOYEE ↔ DEPENDENT) EMPLOYEE: Partial | DEPENDENT: Total

Attributes of Relationship Types

  • Relationship types can also have attributes, similar to those of entity types
  • Attributes of 1:1 or 1:N relationship types can be migrated to one of the participating entity types
  • For M:N relationships, some attributes may be determined by the combination of participating entities — not by any single entity. Such attributes must be specified as relationship attributes
Example: In WORKS_ON (M:N), the attribute Hours depends on both the employee and the project together — it cannot belong to either entity alone.

Weak Entity Types

An entity type that does not have a key attribute of its own. Its entities are identified by being related to specific entities from another entity type, combined with some of their own attribute values.
Identification uses the combination of:
  • A partial key of the weak entity type
  • The particular entity they are related to in the identifying entity type
Example: DEPENDENT is a weak entity type. A DEPENDENT entity is identified by the dependent's first name and birthdate and the specific EMPLOYEE it is related to. EMPLOYEE is the identifying entity type, and DEPENDENT_OF is the identifying relationship.

Alternative (min, max) Notation for Relationship Constraints

  • Specified on each participation of an entity type E in a relationship type R
  • Specifies that each entity e in E participates in at least min and at most max relationship instances in R
  • Default (no constraint): min=0, max=n (no limit)
  • Must have: min ≤ max, min ≥ 0, max ≥ 1
MANAGES Example A department has exactly one manager; an employee can manage at most one department.

EMPLOYEE in MANAGES: (0, 1)
DEPARTMENT in MANAGES: (1, 1)
WORKS_FOR Example An employee works for exactly one department; a department can have any number of employees.

EMPLOYEE in WORKS_FOR: (1, 1)
DEPARTMENT in WORKS_FOR: (1, N)

Limitation of ER Notation

Problem: The entity relationship model in its original form did not support the specialization/generalization abstractions. This limitation led to the development of the Enhanced ER (EER) model.

Key Takeaways

Remember:
  • Entities are real-world objects; attributes describe their properties
  • Attributes can be simple, composite, single-valued, multi-valued, stored, or derived
  • Null values arise when an attribute is not applicable or its value is unknown
  • A key attribute uniquely identifies each entity in an entity set
  • Relationships link entities; their degree is the number of participating entity types
  • Cardinality ratios (1:1, 1:N, M:N) and participation constraints (total/partial) define structural rules
  • A weak entity has no key of its own and relies on an identifying relationship
  • The (min, max) notation precisely expresses both minimum and maximum cardinality
  • The COMPANY database has six binary relationships: WORKS_FOR, MANAGES, CONTROLS, WORKS_ON, SUPERVISION, DEPENDENTS_OF
