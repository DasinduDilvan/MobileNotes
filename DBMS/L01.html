Data vs Information

Data Data refers to raw facts, figures and symbols that may be processed and turned into information.
Information Information means processed data that have been shaped into a form that is meaningful and useful to human beings.
Data → Processing → Information

What is a Database?

A database is a collection of related data.
Key Characteristics:
  • A database is designed, built, and populated with data for a specific purpose
  • It has an intended group of users
  • It has some preconceived applications in which these users are interested

What is Database Management System (DBMS)?

A database management system (DBMS) is a collection of programs that enables users to create and maintain a database.
DBMS is a general-purpose software system that facilitates:
Defining Specifying data types, structures, and constraints
Constructing Storing data on storage medium
Manipulating Querying, updating, and generating reports
Sharing Multiple users and applications accessing data
Additional Functions: DBMS includes protecting the database and maintaining it over a long period of time.

What is a Relational Database Management System (RDBMS)?

A relational database management system (RDBMS) is a program used to create, update, and manage relational databases.
Relational Database Characteristics:
  • ✓ Organize data in the form of tables (relations)
  • ✓ Present the data to the user as relations in tabular form
  • ✓ Provide relational operators to manipulate the data in tabular form

Why Not Traditional File Systems?

Problems with File Systems:
  • Redundancy in defining and storing data
    • Same data stored in multiple files
    • Inconsistencies when data is updated
  • Wasted storage space
    • Duplicate data occupies unnecessary space
  • Redundant efforts to maintain common data up-to-date
    • Must update the same data in multiple locations
    • Time-consuming and error-prone

Characteristics of the Database Approach

1. Self-Describing Nature Database system contains complete definition and description
2. Insulation & Abstraction Programs independent of data structure
3. Multiple Views Different perspectives for different users
4. Data Sharing Multiuser transaction processing

1. Self-Describing Nature of a Database System

  • Database system contains not only the database itself but also a complete definition or description of the database structure and constraints
  • This description is stored in the system catalog (metadata)
  • The DBMS software must work equally well with any number of database applications
  • The DBMS uses the catalog to understand the database structure

2. Insulation between Programs and Data, and Data Abstraction

Program-Data Independence Application programs are independent of the physical structure of the data stored in the database.
Program-Operation Independence The implementation of the operation is specified separately and can be changed without affecting the program.
Data and Operation Abstraction Hiding the complex details of how data is stored and managed, and showing users only what they need to see.
Benefit: Changes to data structure don't require changes to application programs

3. Support of Multiple Views of the Data

  • A database typically has many users, each of whom may require a different perspective or view of the database
  • A view may be:
    • A subset of the database
    • Virtual data derived from database files but not explicitly stored
  • A multiuser DBMS must provide facilities for defining multiple views
Example: An accounting department may see financial data, while HR sees employee data from the same database.

4. Sharing of Data and Multiuser Transaction Processing

Concurrent Access: Allow multiple users to access the database at the same time
  • Online Transaction Processing (OLTP)
    • Multiple users performing transactions simultaneously
    • Examples: Banking systems, airline reservations
  • Concurrency Control
    • Ensures multiple users can access and modify database simultaneously
    • Prevents data errors or inconsistencies
    • Maintains data integrity

Actors on the Scene

People whose jobs involve the day-to-day use of a large database

Database Administrators (DBAs)

Role: Administering the database and the DBMS
Responsibilities:
  • ✓ Authorizing access to the database
  • ✓ Coordinating and monitoring the use of database
  • ✓ Acquiring software and hardware resources as needed
  • ✓ Accountable for problems such as:
    • Breach of security
    • Poor system response time

Database Designers

Role: Responsible for identifying the data to be stored and choosing appropriate structures
Responsibilities:
  • ✓ Communicate with all prospective database users to understand their requirements
  • ✓ Identifying the data to be stored in the database
  • ✓ Choosing appropriate structures to represent and store this data
  • ✓ Develop a view of the database that meets the data and processing requirements

End Users

People whose jobs require access to the database for querying, updating, and generating reports
Casual End Users • Occasionally access the database
• May need different information each time
• Use sophisticated query language
• Typically middle/high-level managers or occasional browsers
Naive/Parametric End Users • Make up sizable portion of end users
• Main job function revolves around constantly querying and updating
• Use pre-programmed transactions
Sophisticated End Users • Engineers, scientists, business analysts
• Thoroughly familiar with DBMS facilities
• Implement applications for complex requirements
Stand-alone Users • Maintain personal databases
• Use ready-made program packages
• Example: Tax software users

System Analysts and Application Programmers

System Analysts • Determine requirements of end users
• Focus on naive and parametric users
• Develop specifications for canned transactions
• Bridge between users and programmers
Application Programmers • Implement specifications as programs
• Test, debug, document, and maintain canned transactions
• Should be familiar with full range of DBMS capabilities
Also known as: Software Engineers, Software Developers

Workers Behind the Scene

Associated with the design, development, and operation of the DBMS software and system environment
DBMS System Designers & Implementers Design and implement the DBMS modules and interfaces as a software package
Tool Developers Design and implement tools for database design, performance monitoring, and optimization
Operators & Maintenance Personnel Responsible for actual running and maintenance of hardware and software environment

Advantages of Using a DBMS

Controlling Redundancy Minimize data duplication
Restricting Unauthorized Access Security and access control
Persistent Storage For program objects and data structures
Storage Structures Indexes for efficient query processing
Multiple User Interfaces Different interfaces for different users
Complex Relationships Representing relationships among data
Integrity Constraints Enforcing data rules and validations
Backup and Recovery Protection against data loss

Implications of the Database Approach

  • Potential for enforcing standards
    • Standardization of data formats, naming conventions, and procedures
  • Reduced application development time
    • DBMS provides many standard functions
  • Flexibility
    • Easy to change database structure
  • Availability of up-to-date information
    • Changes immediately available to all users
  • Economies of scale
    • Cost savings through consolidation
  • Enhanced security and access control
    • Centralized control over data security
