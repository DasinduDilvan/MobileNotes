LECTURE 01

Introduction to Databases & Database Users

Foundations, DBMS concepts, actors, and advantages

Data vs Information

📊 Data

Raw facts, figures and symbols that may be processed. By itself, data has no meaning.

💡 Information

Processed data shaped into a meaningful, useful form for human beings.

Data → Processing → Information

Key Definitions

Database
A collection of related data, designed for a specific purpose, with an intended group of users.
DBMS
A collection of programs that enables users to create and maintain a database. Facilitates Defining, Constructing, Manipulating, and Sharing data.
RDBMS
Organizes data as tables (relations), presents data in tabular form, and provides relational operators to manipulate data.
Database System
The DBMS software together with the data itself (sometimes including applications).

RDBMS Examples

Open Source

MySQLPostgreSQLSQLite

Enterprise

OracleMS SQL ServerIBM DB2

Why Not File Systems?

  • Redundancy: Same data stored in multiple files → wasted space, inconsistency on update
  • Update anomalies: Must update same data in multiple locations — error-prone
  • No sharing: Multiple users cannot access concurrently
  • No security: Limited access control mechanisms

4 Characteristics of the Database Approach

1️⃣ Self-Describing

System catalog (metadata) stores complete description of the database structure and constraints. DBMS uses catalog to understand any database it manages.

2️⃣ Program-Data Independence

Application programs are independent of physical data structure. Changes to data structure don't require changing programs.

3️⃣ Multiple Views

Different users see different subsets or perspectives of the same database (e.g., HR sees employee data, Accounting sees financial data).

4️⃣ Data Sharing

Multiple users access simultaneously. OLTP (Online Transaction Processing) with concurrency control ensures data integrity.

Advantages of DBMS

✅ Redundancy Control

Minimizes data duplication, saves storage, prevents inconsistency.

🔒 Access Control

Restricts unauthorized access; centralized security management.

💾 Persistent Storage

Data persists across program executions; objects and data structures maintained.

⚡ Query Processing

Storage structures and indexes for efficient data retrieval.

🔗 Complex Relationships

Represents and enforces complex relationships between data entities.

🛡️ Backup & Recovery

Protection against hardware/software failures; transaction recovery.

📝 Exam Tip

Remember the 4 characteristics of database approach and be able to explain each. The difference between a DBMS and a file system is a common exam question. Know all actor types and their roles.

LECTURE 02

Database System Concepts & Architectures

Data models, schemas, three-schema architecture, DBMS languages

Data Models

A data model is a set of concepts to describe the structure of a database, the operations for manipulating these structures, and the constraints that must be obeyed.

CategoryAlso CalledDescriptionExamples
Conceptual / High-LevelEntity-based, Object-basedClose to user perception; entities, attributes, relationshipsER Model, EER Model
Physical / Low-Level—How data is stored on disk; for computer specialistsRecord formats, orderings, access paths
Implementation / RepresentationalRecord-basedUnderstood by users; hides storage details; directly implementableRelational Model
Self-Describing—Data description mixed with data valuesXML, Key-Value, NoSQL

Schema vs State

📐 Schema (Intension)
  • The design/structure of the database
  • Changes very infrequently
  • Defined at design time
  • Analogy: House blueprint
📋 State (Extension)
  • The actual data at a point in time
  • Changes every time DB is updated
  • Also called: snapshot, instance
  • Analogy: Furniture in the house
Empty State → Initial State → Current State (all must be Valid States)

Three-Schema Architecture

Purpose

Separate user applications from the physical database. Supports program-data independence and multiple views.

LevelSchemaWhat It DescribesWho Uses It
ExternalExternal / View SchemaIndividual user views — subset of database each user group needsEnd users, application programs
ConceptualConceptual SchemaLogical structure of entire database — entities, attributes, relationships, constraintsDBA, DB designers
InternalInternal SchemaPhysical storage structures, file organizations, indexes, access pathsDBMS internals, storage experts

Data Independence

🔄 Logical Data Independence

Change the conceptual schema without changing external schemas or application programs.

e.g., Adding a new attribute to a table without breaking existing views.

💿 Physical Data Independence

Change the internal schema (storage) without changing the conceptual schema.

e.g., Changing file organization or adding an index without affecting logical design.

DML Types

High-Level (Non-Procedural)
  • SQL — specifies WHAT not HOW
  • Set-oriented: retrieves many records at once
  • Can be standalone or embedded
Low-Level (Procedural)
  • One record at a time
  • Must be embedded in a host language (C, Java, COBOL)
  • More control but more complex

Client-Server Architectures

Two-Tier
  • Client (UI + App Logic) ↔ Database Server
  • Uses ODBC / JDBC interfaces
  • Simple but less secure
Three-Tier
  • Client (UI) ↔ App Server (Business Logic) ↔ DB Server
  • Common for web apps
  • Enhanced security — clients cannot directly access DB

LECTURE 03

Database Development Life Cycle

DDLC phases, desirable properties, schema levels, testing

5 Desirable Properties of a Database

✅ Completeness

Users can access all the data they need, including ad hoc queries not explicitly anticipated.

🔗 Integrity

Data is consistent (no contradictions) and correct (no invalid values). Users trust the database.

🔄 Flexibility

Database can evolve without excessive effort as requirements change.

⚡ Efficiency

Users do not have unduly long response times when accessing data.

👤 Usability

Data can be accessed and manipulated in ways matching user requirements.

Database Development Life Cycle

1
Requirements Gathering → Data Requirements Document (DRD)
Consult all users; identify data items, attributes, constraints, relationships. Says what data is needed, not how to process it.
↓
2
Data Analysis → Conceptual Data Model
Focuses on meaning and structure of data. Not used by any DBMS. Answers "What is required?" not "How is it achieved?"
↓
3
Database Design → Logical Schema
Determine DB type (relational, OO, etc.). Define tables, constraints, primary/foreign keys. Choose design that best satisfies prioritized properties.
↓
4
Implementation → Storage schema + Security + External schema
Construct DB per logical schema. Heavily influenced by specific DBMS capabilities. Activities: Release design, Populate DB, Support users, Data management strategies.
↓
5
Testing → Acceptance or Bug List
Uncover errors in design and implementation. Validation (right database?) + Verification (built correctly?).
↓
6
Maintenance
Operational monitoring, porting/implementation changes, requirements changes, bug fixing.

Three Schema Levels

Schema LevelFocusPropertiesAudience
ConceptualBusiness world viewSuppresses non-critical details; only significant entities with business meaning; many-to-many relationships acceptableBusiness analysts, all users
LogicalMathematical/relational theoryFully normalized entities; conforms to relational theory; may include logical domains (not necessarily physical tables)DB designers, developers
PhysicalSpecific DBMS productSingle logical model instantiated in specific DBMS; includes implementation details, configuration choicesDBAs, system engineers

DRD — Data Requirements Document

Key Characteristics
  • Describes what data is needed — not how it is processed
  • States: what data items exist, their attributes, constraints, and relationships
  • Used to agree requirements with users
  • Should be easily understood — not overly formal
  • Concise summary of all users' requirements (not just individuals')

Testing: Validation vs Verification

✔️ Validation

"Are we building the RIGHT database?"

  • Data stored and retrieved correctly
  • Queries return correct, meaningful results
  • Transactions behave correctly (commit/rollback)
🔧 Verification

"Are we building the database CORRECTLY?"

  • Schema matches requirements
  • Correct data types and attribute names
  • Primary keys and foreign keys defined properly

Maintenance Types

TypeDescription
OperationalMonitoring database performance continuously
Porting & ImplementationDBMS, user processes, or computer system changes that require DB revision
Requirements ChangeOriginal requirements specification changes — revisits entire DDLC
📝 Exam Tip

Know all 5 desirable properties with explanations. Be clear on what a DRD contains. The three assumptions behind mapping to database development are important. Distinguish between validation and verification.

LECTURE 04

Data Modeling — Entity-Relationship Model

Entity types, attributes, keys, relationships, constraints, weak entities

Core ER Concepts

🧱 Entity

A specific object or thing in the miniworld with independent existence. E.g., student, car, department, employee.

📌 Attribute

Properties that describe an entity. E.g., an employee has name, age, address, salary.

Types of Attributes

TypeDescriptionExample
Simple (Atomic)Cannot be divided furtherSex, Age
CompositeMade of sub-components; forms a hierarchyAddress (Street, City, State, Zip)
Single-ValuedOne value per entityNational ID Number
Multi-ValuedSet of values for same entityColors of a car, Degrees held

NULL Values

Not Applicable

Attribute doesn't apply to this entity. E.g., Apartment Number for a house.

Unknown

Value may exist but is missing, or we don't know if it exists. E.g., Home Phone.

Entity Types vs Entity Sets

Entity Type
The schema (intension) — defines a collection of entities with same attributes. E.g., EMPLOYEE (Name, Age, Salary).
Entity Set
The extension — all entity instances at a point in time. E.g., {(John Smith, 55, 80k), (Fred Brown, 40, 30k), ...}.
Key Attribute
Attribute(s) whose values uniquely identify each entity in the set. Shown with underline in ER diagrams.
Value Set (Domain)
All possible values an attribute can hold. E.g., Age: integers 16–70.

Relationships & Degree

A relationship connects two or more distinct entities with a specific meaning. The degree = number of participating entity types.

  • Binary (degree 2): Most common — WORKS_FOR, MANAGES, CONTROLS, WORKS_ON, SUPERVISION, DEPENDENTS_OF
  • Ternary (degree 3): SUPPLY (SUPPLIER, PROJECT, PART)

Relationship Constraints

Cardinality Ratios (Maximum)

1:1

One-to-One — each entity in A relates to at most one in B and vice versa. E.g., MANAGES.

1:N

One-to-Many — one entity in A relates to many in B. E.g., WORKS_FOR (Dept → Employees).

M:N

Many-to-Many — many in A to many in B. E.g., WORKS_ON (Employee ↔ Project).

Participation Constraints (Minimum)

Total Participation (double line)

Every entity MUST participate. E.g., every EMPLOYEE must work FOR a DEPARTMENT.

Partial Participation (single line)

Some entities may not participate. E.g., not every EMPLOYEE manages a DEPARTMENT.

Weak Entity Types

Definition

An entity type with no key attribute of its own. Identified by being related to an owner entity + its own partial key.

  • Shown as double rectangle
  • Identifying relationship shown as double diamond
  • Example: DEPENDENT — identified by (Dependent_name + Birthdate) + linked EMPLOYEE (Essn)
  • EMPLOYEE is the identifying entity type; DEPENDENTS_OF is the identifying relationship

(min, max) Notation

Specifies that each entity e in E participates in at least min and at most max relationship instances.

RelationshipEntity(min, max)Meaning

MANAGESDEPARTMENT(1, 1)Every department has exactly 1 manager
WORKS_FOREMPLOYEE(1, 1)Every employee works for exactly 1 department

LECTURE 05

Enhanced Entity-Relationship (EER) Model

Subclasses, superclasses, specialization, generalization, categories, inheritance

Superclasses & Subclasses

EER adds meaningful subgroupings of entities. A subclass is a subset of entities from a superclass, each in a distinct specific role.

IS-A Rule
  • A subclass member is the same entity as in the superclass — just in a specific role
  • An entity cannot exist in the DB merely as a subclass member — must also be in superclass
  • A superclass member may or may not belong to any subclass
  • One entity can belong to multiple subclasses simultaneously

Example (EMPLOYEE subclasses):

SECRETARYENGINEERTECHNICIAN — by job type

MANAGER — by management role

HOURLY_EMPLOYEESALARIED_EMPLOYEE — by pay method

Attribute Inheritance

Inheritance Rule

A subclass member inherits ALL attributes and ALL relationship participations of the superclass.

Example: SECRETARY inherits Name, SSN, Birth_date, Address, etc. from EMPLOYEE, plus has its own Typing_speed attribute.

Specialization vs Generalization

🔽 Specialization (Top-Down)

Start with a superclass, define subclasses by distinguishing characteristics.

  • Conceptual refinement process
  • Specific attributes added to subclasses
  • Specific relationships for subclasses only
🔼 Generalization (Bottom-Up)

Start with multiple classes having common features, create a superclass.

  • Conceptual synthesis process
  • Common attributes moved to superclass
  • Specific attributes remain in subclasses
Note

Both approaches produce the same end-result structure. In practice, a combination is used.

Constraints on Specialization

1. Membership Condition

TypeDescriptionHow to Identify
Predicate-DefinedMembership determined by condition on superclass attributeCondition written next to line in EER diagram
Attribute-DefinedAll subclasses use the same defining attribute for membershipjob_type = 'SECRETARY' / 'ENGINEER' etc.
User-DefinedNo condition; membership specified individually for each entityNo condition shown in diagram

2. Disjointness Constraint

Disjoint "d"

Entity belongs to at most one subclass. E.g., cannot be both SECRETARY and ENGINEER.

Overlapping "o"

Entity can belong to more than one subclass. E.g., MANAGER and ENGINEER simultaneously.

3. Completeness Constraint

Total (double line)

Every superclass entity must belong to at least one subclass.

Partial (single line)

Superclass entity is allowed not to belong to any subclass.

4 Combinations

CombinationMeaning
Disjoint + TotalEach entity belongs to exactly ONE subclass
Disjoint + PartialEach entity belongs to at most one subclass (or none)
Overlapping + TotalEach entity belongs to at least one subclass (possibly many)
Overlapping + PartialEach entity may belong to multiple or no subclasses

Hierarchies vs Lattices

Specialization Hierarchy
  • Each subclass has only one superclass
  • Single inheritance — tree structure
  • Inherits from one path only
Specialization Lattice
  • Subclass can have more than one superclass
  • Multiple inheritance — allows shared subclasses
  • Shared Subclass: subclass with multiple superclasses

Categories (UNION Types)

Definition

A category is a subclass representing a subset of the UNION of multiple superclasses from different entity types.

Example: OWNER = subset of (PERSON ∪ BANK ∪ COMPANY) — a vehicle owner can be any of these entity types.

Shown with "U" in EER diagram.

Category vs Shared Subclass

Category (UNION Type)Shared Subclass

SuperclassesCan be different entity typesMust share same key
InheritanceFrom one superclass at a timeMultiple inheritance

📝 Exam Tip

Know the 4 constraint combinations for specialization. Be able to distinguish category (UNION type "U") from shared subclass (multiple inheritance). The IS-A relationship rules and attribute inheritance are commonly tested.

LECTURE 06

ER & EER to Relational Mapping

9-step algorithm for converting conceptual schemas to relational tables

Mapping Algorithm Overview

StepConstructAction
1Regular (strong) entityCreate relation R with all simple attributes; choose primary key
2Weak entityCreate relation R; add owner PK as FK; PK = owner PK + partial key
3Binary 1:1 relationshipUse Foreign Key, Merged, or Cross-Reference approach
4Binary 1:N relationshipAdd 1-side PK as FK in N-side relation; add relationship attrs
5Binary M:N relationshipCreate new relationship relation; PK = combination of both FKs
6Multivalued attributeCreate new relation; PK = attribute value + owner PK (FK)
7N-ary relationship (n > 2)Create new relation; all participating PKs as FKs
8Specialization / GeneralizationChoose Option 8A, 8B, 8C, or 8D
9Category (Union type)Create category relation; use surrogate key if superclasses have different keys

Steps 1–7 Details

Step 3 — Binary 1:1 (Three approaches)

A: Foreign Key (Preferred)

Add PK of one relation as FK in the other. Choose the total participation side as the FK holder. Add relationship attributes here too.

B: Merged Relation

Merge both entity types and relationship into one. Only suitable when both participations are total.

C: Cross-Reference

Create a 3rd relation with both PKs as FKs. PK = one FK, other FK = unique. Requires extra JOIN — least preferred.

Step 8 — Specialization Mapping Options

OptionRelations CreatedWorks ForKey Trade-off
8ASuperclass + one per subclassAny (total/partial, disjoint/overlapping)Most general; requires JOIN for full entity data
8BOne per subclass only (includes superclass attrs)Total + Disjoint ONLYNo superclass table; duplication if overlapping
8CSingle relation + ONE type attribute (discriminator)Disjoint ONLYSimple; many NULLs for subclass-specific attrs
8DSingle relation + multiple Boolean flagsOverlapping (also disjoint)One Boolean per subclass; NULLs for specific attrs

Step 9 — Category (Union Type)

Surrogate Key

When category superclasses have different keys, create a new artificial surrogate key as the PK for the category relation. Add this surrogate key to each superclass relation as a FK reference to the category.

Example: OWNER category: PERSON(Ssn), BANK(Bname), COMPANY(Cname) all have different keys → create Owner_id as surrogate key.

LECTURE 07

Relational Algebra

Unary, set theory, binary operations — the formal query language

Overview

Relational algebra is a procedural formal query language. Operations on relations produce new relations (the algebra is closed). It is the theoretical foundation for SQL.

Unary Operations

SELECT — σ (sigma)

Syntax
σ <condition>(R)
  • Filters rows; result has same schema as R
  • Result tuple count ≤ R tuple count
  • Commutative: σc1(σc2(R)) = σc2(σc1(R))
  • Cascade → single SELECT with AND: σc1(σc2(σc3(R))) = σc1 AND c2 AND c3(R)

Examples: σ DNO=4(EMPLOYEE)    σ SALARY>30000(EMPLOYEE)

PROJECT — π (pi)

Syntax
π <attribute list>(R)
  • Keeps specified columns; removes duplicates (result is a set)
  • NOT commutative
  • πL1(πL2(R)) = πL1(R) if L2 ⊇ L1

Example: π LNAME, FNAME, SALARY(EMPLOYEE)

RENAME — ρ (rho)

FormEffect
ρS(B1,...,Bn)(R)Rename relation to S AND attributes to B1...Bn
ρS(R)Rename relation name only to S
ρ(B1,...,Bn)(R)Rename attribute names only

Set Theory Operations

⚠️ Type Compatibility Required

UNION, INTERSECTION, and SET DIFFERENCE require union-compatible (same number of attributes, compatible domains). Result attribute names taken from first operand.

OperationSymbolResultProperties
UNIONR ∪ SAll tuples in R OR S (no duplicates)Commutative, Associative
INTERSECTIONR ∩ SOnly tuples in BOTH R and SCommutative, Associative
DIFFERENCER − STuples in R but NOT in SNOT commutative
CARTESIAN PRODUCTR × SAll combinations (nR × nS tuples); degree n+mNo type compat. needed; rarely useful alone

Binary Join Operations

THETA JOIN (⋈)

R ⋈<condition> S = σ<condition>(R × S)

Result degree = n + m. Only related tuples appear. Generally result has < nR × nS tuples.

EQUIJOIN

Theta join where the only comparison is equality (=). Most common in practice. The COMPANY example: DEPARTMENT ⋈ MGRSSN=SSN EMPLOYEE

NATURAL JOIN (*)

EQUIJOIN that eliminates the duplicate join attribute. Automatically joins on attributes with the same name.

Q ← R(A,B,C,D) * S(C,D,E) → implicit: R.C=S.C AND R.D=S.D → result Q(A,B,C,D,E)

DIVISION (÷)

Definition

R(Z) ÷ S(X) where X ⊆ Z. Let Y = Z − X. Result T(Y) contains tuple t if t appears in R combined with every tuple in S.

Use case: "Find employees who work on ALL projects in S."

Aggregate Functions — ℱ

FunctionPurposeExample
SUMSum of numeric valuesℱ SUM Salary(EMPLOYEE)
AVGAverage valueℱ AVG Salary(EMPLOYEE)
MAXHighest valueℱ MAX Salary(EMPLOYEE)
MINLowest valueℱ MIN Salary(EMPLOYEE)
COUNTCount tuples (no dedup)ℱ COUNT SSN(EMPLOYEE)

Grouping syntax: DNO ℱ COUNT SSN, AVG Salary(EMPLOYEE) — groups by DNO then applies aggregate per group.

Outer Joins

TypeSymbolKeeps
LEFT OUTER JOINR ⟕ SAll tuples in R; unmatched S attributes → NULL
RIGHT OUTER JOINR ⟖ SAll tuples in S; unmatched R attributes → NULL
FULL OUTER JOINR ⟗ SAll tuples in both; unmatched → NULL on either side

Complete Set of Operations

The Complete Set

σ, π, ∪, −, ρ, × — any other relational algebra expression can be derived from these six operations.

R ∩ S = (R ∪ S) − ((R − S) ∪ (S − R))

R ⋈<cond> S = σ<cond>(R × S)

📝 Exam Tip

Know all operation symbols (σ, π, ρ, ∪, ∩, −, ×, ⋈, *, ÷). Understand which operations require type compatibility. CARTESIAN PRODUCT alone is meaningless — combine with SELECT. The complete set of 6 operations is commonly tested.

LECTURE 08

Relational Calculus

Tuple calculus, domain calculus, QBE overview — declarative query formalism

Algebra vs Calculus

⚙️ Relational Algebra

Procedural — specifies a sequence of operations describing how to retrieve results.

📝 Relational Calculus

Nonprocedural / Declarative — specifies what the result should contain, not how to find it.

Equivalence

Relational Algebra, Tuple Calculus, and Domain Calculus are all expressively equivalent. Any query in one can be expressed in the others.

Tuple Relational Calculus (TRC)

General Form
{ t | COND(t) }

t = tuple variable. Returns all tuples t satisfying COND(t). Range relation specified as EMPLOYEE(t) etc.

Simple Example — employees with salary > 50,000:

{ t.FNAME, t.LNAME | EMPLOYEE(t) AND t.SALARY > 50000 }

Quantifiers

∃ — Existential ("there exists")

(∃t)(F) is true if F is true for at least one tuple assigned to t.

Use: "there exists a department named Research"

∀ — Universal ("for all")

(∀t)(F) is true if F is true for every tuple in the universe assigned to t.

Use: "works on ALL projects in dept 5"

Bound variable
A tuple variable that appears in a (∀t) or (∃t) quantifier clause.
Free variable
A tuple variable NOT quantified. Only free variables should appear to the left of the bar ( | ).

Example Queries

Research dept employees (using ∃):

{ t.FNAME, t.LNAME, t.ADDRESS |
  EMPLOYEE(t) AND
  (∃d)(DEPARTMENT(d) AND d.DNAME = 'Research' AND d.DNUMBER = t.DNO) }

Employees with no dependents (∃ and ∀ equivalents):

-- Using ∃:
{ e.Fname, e.Lname | EMPLOYEE(e) AND
  (NOT(∃d)(DEPENDENT(d) AND e.Ssn = d.Essn)) }

-- Using ∀ (equivalent):
{ e.Fname, e.Lname | EMPLOYEE(e) AND
  ((∀d)(NOT(DEPENDENT(d)) OR NOT(e.Ssn = d.Essn))) }

Strategy for ∀ Queries

1
Exclude non-relation tuples: NOT(PROJECT(x)) → true for all non-project tuples
2
Exclude irrelevant relation tuples: NOT(x.DNUM = 5) → true for projects not in dept 5
3
Specify condition on remaining tuples: Employee must work on each remaining project (∃w)(WORKS_ON(w) ...)

Domain Relational Calculus (DRC)

Key Difference from TRC

Variables range over single attribute values (columns) rather than entire tuples (rows). Need n domain variables for a relation of degree n.

{ x1, x2, ..., xn | COND(x1, ..., xn, ..., xn+m) }

Example — Birthdate and address of 'John B. Smith':

{ u, v | (∃q)(∃r)(∃s)(∃t)(∃w)(∃x)(∃y)(∃z)
  (EMPLOYEE(q,r,s,t,u,v,w,x,y,z) AND
   q = 'John' AND r = 'B' AND s = 'Smith') }

10 variables for 10 EMPLOYEE attributes. u = BDATE, v = ADDRESS (free). All others bound by ∃.

Condition Types in DRC

Selection Condition

Relates domain variable to a constant. e.g., l = 'Research'

Join Condition

Relates two domain variables from different relations. e.g., m = z (DEPT.Dnumber = EMP.Dno)

QBE — Query By Example

Magnetic Disk Structure

🔵 Track

Circular ring on platter surface. Divided into sectors.

📐 Sector

Smallest unit of data read/written on disk.

🏢 Cylinder i

All i-th tracks across all platters (same arm position).

🎮 Disk Controller

Interface between computer and disk. Computes checksums, verifies writes, remaps bad sectors.

Performance Measures

Access Time
= Seek Time + Rotational Latency. Time from request issued to transfer start.
Seek Time
Time for disk arm to move to correct track.
Rotational Latency
Time for desired sector to rotate under the read-write head.
Transfer Rate
Rate at which data is read from or written to disk.
MTTF
Mean Time to Failure — typically 3–5 years. Failure probability increases with age.

RAID Levels

LevelNameDescriptionUsed?
RAID 0Striping (No Fault Tolerance)Data striped across disks; no redundancyOnly when data safety unimportant
RAID 1Mirroring & DuplexingEach disk has exact copy (mirror)✅ Widely used; high availability
RAID 2ECCMemory-style error-correcting coding across disks❌ Never used (subsumed by RAID 3)
RAID 3Bit-Interleaved ParityParity on dedicated disk❌ Not used anymore
RAID 4Dedicated Parity DriveBlock-interleaved parity on one disk❌ Never used (subsumed by RAID 5)
RAID 5Distributed ParityParity distributed across all disks✅ Most common; good balance
RAID 6Double Parity (P+Q)Two independent parity blocks; survives 2 disk failures⚠️ Rarely used
RAID Key Concepts
  • Latent Failures: Previously written data silently damaged — can cause data loss even when only one disk fails
  • Data Scrubbing: Continuously scans for latent failures
  • Hot Swapping: Replace failed disk without powering down
  • Software RAID = no special hardware; Hardware RAID = uses NV-RAM to log writes

File Organization

Fixed-Length Records

Simple; when a record is deleted:

  • Compact: Move all subsequent records up — expensive
  • Move last record: Copy last record into deleted slot — faster
  • Free List (best): Linked list of free slots in header — new records inserted at first free slot

Variable-Length Records

Arise when: multiple record types in one file, VARCHAR fields, or repeating fields. Represented by (offset, length) pairs + null-value bitmap.

Slotted Page Structure

Header contains: number of records, end of free space, location+size of each record. Records can be moved within page to remove gaps; header updated. Pointers → header entry, not directly to record.

File Organization Types

Heap Organization

Records placed anywhere there's free space. No ordering. Simple but slow for search.

Sequential Organization

Ordered by search key. Efficient for range queries and sorted output.

Hashing Organization

Hash function on attribute determines block placement. Fast point queries.

Blocking & Buffer Management

Block
Contiguous sectors from one track; unit of storage allocation and transfer. Typical size: 4–16 KB. Smaller = more transfers; larger = more wasted space.
Blocking Factor (bfr)
Number of records per block.
Spanned Records
Records larger than block size that span multiple blocks.
Buffer
Portion of main memory storing copies of disk blocks. Reduces disk accesses.
Buffer Manager
Subsystem allocating buffer space; aims to minimize disk block transfers.

Data Dictionary (System Catalog)

Stores metadata — stored as a set of relations itself. Contains:

  • Relation metadata: names, attributes, types, storage
  • Index metadata: index names, types, attributes
  • View metadata: view names and definitions
  • User metadata: accounts, encrypted passwords, permissions
  • Physical organization info
  • Statistical data (tuple counts, etc.)
📝 Exam Tip

Know RAID 0, 1, 5 well — which are commonly used and why. The three performance measures formula (Access Time = Seek + Rotational Latency). Understand free list for fixed records and slotted pages for variable records. Know what the data dictionary stores.

LECTURE 11

Introduction to Normalization

Functional dependencies, 1NF → 2NF → 3NF → BCNF → 4NF → 5NF

4 Informal Design Guidelines

G1: Clear Semantics

Each tuple represents one entity or relationship instance. Don't mix attributes of different entities. Use FKs to reference other entities.

G2: Avoid Update Anomalies

Avoid insertion, deletion, and modification anomalies caused by redundancy. Document unavoidable anomalies.

G3: Minimize NULLs

NULLs are ambiguous (not applicable? unknown? known but absent?). Place frequently NULL attributes in separate relations.

G4: No Spurious Tuples

Design for lossless join property. Avoid decompositions that generate invalid tuples when joined back.

Update Anomalies

AnomalyDescriptionExample (EMP_DEPT)
InsertionCannot insert without providing unrequired data, or cannot add entity without anotherCan't insert new department without an employee; must supply dept data for new employee
DeletionDeleting a record causes unintended loss of other dataDeleting last employee in a dept → lose dept's info
ModificationMust update multiple tuples for one logical change → inconsistency riskChanging dept manager SSN requires updating many employee rows

Functional Dependencies (FDs)

Definition: X → Y

For any two tuples t1, t2 in R: if t1[X] = t2[X], then t1[Y] = t2[Y].
"X functionally determines Y" — values of X uniquely identify values of Y.

Example FDs in EMP_PROJ(Ssn, Pnumber, Hours, Ename, Pname, Plocation)

Ssn → Ename (FD2)
Pnumber → {Pname, Plocation} (FD3)
{Ssn, Pnumber} → Hours (FD1)

Armstrong's Inference Rules

RuleNameStatement
IR1ReflexiveIf Y ⊆ X, then X → Y
IR2AugmentationIf X → Y, then XZ → YZ
IR3TransitiveIf X → Y and Y → Z, then X → Z

Derived rules: Decomposition (X→YZ ⟹ X→Y, X→Z), Union (X→Y & X→Z ⟹ X→YZ), Pseudo-Transitivity.

Key Definitions

Superkey
Set of attributes S where no two tuples have the same combination of S values.
Key (Candidate Key)
Minimal superkey — removing any attribute causes it to no longer be a superkey.
Primary Key
One designated candidate key.
Prime Attribute
Member of at least one candidate key.
Nonprime Attribute
Not a member of any candidate key.

Normal Forms

1️⃣ First Normal Form (1NF)

Requirements: All attributes are atomic (indivisible). No composite or multi-valued attributes. No nested relations.

Fix: Split multi-valued attributes into separate rows; propagate primary key to nested relations.

2️⃣ Second Normal Form (2NF)

Requirements: In 1NF + every nonprime attribute is fully functionally dependent on the primary key (no partial dependencies).

Partial Dependency

A nonprime attribute depends on ONLY PART of a composite primary key.

Example violation in EMP_PROJ({Ssn, Pnumber}, Hours, Ename, Pname, Plocation):

  • Ssn → Ename — partial dependency (Ename depends only on Ssn, not full key)
  • Pnumber → {Pname, Plocation} — partial dependency

2NF Decomposition: EP1(Ssn, Pnumber, Hours) | EP2(Ssn, Ename) | EP3(Pnumber, Pname, Plocation)

3️⃣ Third Normal Form (3NF)

Requirements: In 2NF + no nonprime attribute is transitively dependent on the primary key.

Transitive Dependency

X → Z via an intermediate nonprime attribute Y: X → Y and Y → Z (Y is nonprime).

Example violation in EMP_DEPT(Ssn, Ename, Bdate, Address, Dnumber, Dname, Dmgr_ssn):

  • Ssn → Dnumber and Dnumber → Dmgr_ssn → so Ssn → Dmgr_ssn is transitive

3NF Decomposition: ED1(Ename, Ssn, Bdate, Address, Dnumber) | ED2(Dnumber, Dname, Dmgr_ssn)

✦ BCNF — Boyce-Codd Normal Form

Requirements: For EVERY non-trivial FD X → A in R, X must be a superkey. Stronger than 3NF — a relation can be in 3NF but NOT BCNF.

Example violation — TEACH(Student, Course, Instructor):

  • fd1: {Student, Course} → Instructor ✓
  • fd2: Instructor → Course ✗ (Instructor is NOT a superkey)

BCNF Decompositions: {Student, Instructor} & {Student, Course} | or {Course, Instructor} & {Course, Student}

4️⃣ Fourth Normal Form (4NF)

Requirements: In BCNF + no Multi-Valued Dependencies (MVDs).

MVD: X →→ Y

One value of X is associated with multiple values of Y, where Y and Z are independent. Requires at least 3 columns.

Example: STUDENT(reg_no, course, club) — courses and clubs are independent MVDs.

Decompose: STUDENT_COURSE(reg_no, course) | STUDENT_CLUB(reg_no, club)

5️⃣ Fifth Normal Form (5NF)

Requirements: In 4NF + no Join Dependencies (JDs). A generalization of MVD.

Join Dependency

Table can be losslessly decomposed into multiple tables (each with a subset of attributes) and recreated via JOIN.

Example: SUPPLY(Sname, Part_name, Proj_name) → decomposes into R1(Sname, Part_name), R2(Sname, Proj_name), R3(Part_name, Proj_name).

Quick Reference: Normal Forms

Hierarchy

📝 Exam Tip

Be able to identify all 3 anomaly types with examples. Write and verify FDs using Armstrong's rules. For normalization: identify the primary key, check for partial (2NF) and transitive (3NF) dependencies, and decompose accordingly. Know that BCNF is stricter than 3NF (3NF can have non-superkey LHS if RHS is prime). Practice decomposing relations step by step.
