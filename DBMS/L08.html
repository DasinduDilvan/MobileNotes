Example: COMPANY Database

  • The company is organized into departments. Each department has a unique name, a unique number, and a managing employee. The department manager's start date is tracked. A department may have several locations.

1. Relational Calculus — Overview

Definition: A relational calculus expression creates a new relation, which is specified in terms of variables that range over rows of the stored database relations (in tuple calculus) or over columns of the stored relations (in domain calculus).
Key Characteristic

In a calculus expression, there is no order of operations to specify how to retrieve the query result. A calculus expression specifies only what information the result should contain. This is the main distinguishing feature between relational algebra and relational calculus.

Relational Algebra Procedural

You must write a sequence of operations to specify a retrieval request. Describes how to get the result.
Relational Calculus Nonprocedural / Declarative

You specify what information the result should contain. No step-by-step operation order needed.
Two Types of Relational Calculus
Tuple Relational Calculus (TRC) Variables range over tuples (rows) of relations.
Domain Relational Calculus (DRC) Variables range over single values from the domains (columns) of attributes.

2. Tuple Relational Calculus (TRC)

  • Based on specifying a number of tuple variables.
  • Each tuple variable usually ranges over a particular database relation — meaning it may take as its value any individual tuple from that relation.
General Form
{ t | COND(t) }
  • t is a tuple variable.
  • COND(t) is a conditional expression involving t.
  • The result is the set of all tuples t that satisfy COND(t).
Simple Example

Find the first and last names of all employees whose salary is above $50,000:

{ t.FNAME, t.LNAME | EMPLOYEE(t) AND t.SALARY > 50000 }
  • EMPLOYEE(t) specifies that the range relation of tuple variable t is EMPLOYEE.
  • t.FNAME, t.LNAME — corresponds to PROJECTION (πFNAME, LNAME).
  • t.SALARY > 50000 — corresponds to SELECTION (σSALARY > 50000).

3. The Existential and Universal Quantifiers

Two special symbols called quantifiers can appear in formulas:

∀ — Universal Quantifier ("for all") (∀ t)(F) is true if the formula F evaluates to true for every tuple in the universe assigned to free occurrences of t in F. Otherwise it is false.

Every tuple must make F true.
∃ — Existential Quantifier ("there exists") (∃ t)(F) is true if the formula F evaluates to true for at least one tuple assigned to free occurrences of t in F. Otherwise it is false.

At least one tuple must make F true.
Bound vs Free Variables
A tuple variable t is bound if it is quantified — meaning it appears in a (∀ t) or (∃ t) clause. Otherwise, it is free. Only free variables should appear to the left of the bar ( | ) in a relational calculus expression.

4. Example Query Using Existential Quantifier (∃)

Retrieve the name and address of all employees who work for the 'Research' department:

{ t.FNAME, t.LNAME, t.ADDRESS | EMPLOYEE(t) AND (∃ d)(DEPARTMENT(d) AND d.DNAME = 'Research' AND d.DNUMBER = t.DNO) }

5. Example Query Using Universal Quantifier (∀)

Find the names of employees who work on all the projects controlled by department number 5:

{ e.LNAME, e.FNAME | EMPLOYEE(e) AND ( (∀x)(NOT(PROJECT(x)) OR NOT(x.DNUM = 5) OR ( (∃w)(WORKS_ON(w) AND w.ESSN = e.SSN AND x.PNUMBER = w.PNO) ) ) ) }
Strategy for Using ∀

When using universal quantification, exclude all tuples you are not interested in by making the condition evaluate to true for those tuples automatically:

  • Step 1 — Exclude non-relation tuples: NOT(PROJECT(x)) evaluates to true for all tuples x that are not in the PROJECT relation.
  • Step 2 — Exclude irrelevant tuples from the relation: NOT(x.DNUM = 5) evaluates to true for all project tuples not controlled by department 5.
  • Step 3 — Specify condition on remaining tuples: (∃ w)(WORKS_ON(w) AND w.ESSN = e.SSN AND x.PNUMBER = w.PNO) — the employee must work on each remaining project.

6. Using Both Quantifiers — Transformation

List the names of employees who have no dependents:

Version 1: Using Existential Quantifier (∃)
{ e.Fname, e.Lname | EMPLOYEE(e) AND (NOT (∃d)(DEPENDENT(d) AND e.Ssn = d.Essn)) }
Version 2: Same Query Using Universal Quantifier (∀)
{ e.Fname, e.Lname | EMPLOYEE(e) AND ((∀d)(NOT(DEPENDENT(d)) OR NOT(e.Ssn = d.Essn))) }
General Transformation Rule: The two versions are logically equivalent. The existential form checks that there does NOT exist a dependent linked to the employee. The universal form checks that for ALL tuples d, either d is not a dependent at all, or d is not linked to this employee.

7. Languages Based on Tuple Relational Calculus — SQL

SQL is based on tuple relational calculus. It uses a basic block structure to express queries:
SELECT <list of attributes> FROM <list of relations> WHERE <conditions>

SQL syntax is expanded further to accommodate other operations beyond this basic block structure.

8. Domain Relational Calculus (DRC)

Definition: Domain relational calculus is another variation of relational calculus that is equivalent to both tuple calculus and relational algebra. Variables range over single values from domains of attributes (columns), rather than over entire tuples (rows).
  • The language QBE (Query-By-Example), related to domain calculus, was developed almost concurrently with SQL at IBM Research, Yorktown Heights, New York.
  • Domain calculus was developed as a way to explain what QBE does.
  • To form a relation of degree n, we need n domain variables — one for each attribute.
Key Difference: Tuple Calculus vs Domain Calculus
Tuple Calculus Variables range over entire tuples (rows) of a relation.
e.g., t.SALARY accesses an attribute of a tuple.
Domain Calculus Variables range over single values from the domains (columns) of attributes.
e.g., separate variable for each attribute column.
General Form of Domain Calculus Expression
{ x₁, x₂, ..., xₙ | COND(x₁, x₂, ..., xₙ, xₙ₊₁, xₙ₊₂, ..., xₙ₊ₘ) }
  • x₁, x₂, …, xₙ, xₙ₊₁, …, xₙ₊ₘ are domain variables that range over domains (of attributes).
  • COND is a condition or formula of the domain relational calculus.

9. Example Queries Using Domain Calculus

Example 1: Retrieve birthdate and address of 'John B. Smith'
{ u, v | (∃q)(∃r)(∃s)(∃t)(∃w)(∃x)(∃y)(∃z) (EMPLOYEE(q,r,s,t,u,v,w,x,y,z) AND q = 'John' AND r = 'B' AND s = 'Smith') }

Explanation:

  • Ten variables (q through z) are needed for the EMPLOYEE relation — one for each attribute in order: Fname, Minit, Lname, Ssn, Bdate, Address, Sex, Salary, Super_ssn, Dno.
  • Only u (for BDATE) and v (for ADDRESS) are free — they appear on the left of the bar.
  • All other variables (q, r, s, t, w, x, y, z) are bound by the ∃ quantifier.
  • The conditions q = 'John', r = 'B', s = 'Smith' select the specific employee tuple.
Abbreviated notation: EMPLOYEE(qrstuvwxyz) — variables listed without separating commas, in attribute order.
Example 2: Retrieve name and address of employees in the 'Research' department
{ q, s, v | (∃z)(∃l)(∃m) ( EMPLOYEE(q,r,s,t,u,v,w,x,y,z) AND DEPARTMENT(l,m,n,o) AND l = 'Research' AND m = z ) }

Explanation:

  • Free variables: q (Fname), s (Lname), v (Address) — the requested output attributes.
  • l = 'Research' is a selection condition (relates a domain variable to a constant).
  • m = z is a join condition (relates two domain variables from two different relations — DEPARTMENT.Dnumber = EMPLOYEE.Dno).
Selection Condition Relates a domain variable to a constant.
Example: l = 'Research'
Join Condition Relates two domain variables from two different relations.
Example: m = z

All three — Relational Algebra, Tuple Calculus, and Domain Calculus — are expressively equivalent. Any query expressible in one can be expressed in the others.

11. Overview of the QBE Language (Query-By-Example)

  • QBE is a query language related to domain relational calculus.
  • Developed almost concurrently with SQL at IBM Research, Yorktown Heights, New York.
  • Domain calculus was considered a formal way to explain what QBE does.
  • QBE provides a visual, table-based interface for formulating queries — users fill in example values in table templates to express queries.
QBE is a practical implementation of domain calculus concepts. It is particularly user-friendly because it uses a graphical/tabular form rather than a mathematical formula syntax.
