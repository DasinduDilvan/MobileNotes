Data Definition, Constraints & Schema Changes

DDL (Data Definition Language) is used to CREATE, DROP, and ALTER the descriptions (schema) of tables in a database.

CREATE TABLE

Specifies a new base relation by giving it a name and defining its attributes with data types and optional constraints.

Common SQL Data Types: INTEGER, FLOAT, DECIMAL(i,j), CHAR(n), VARCHAR(n), DATE

The NOT NULL constraint can be placed on an attribute to prevent null values.

Basic CREATE TABLE (with NOT NULL):
CREATE TABLE DEPARTMENT (
  DNAME         VARCHAR(10) NOT NULL,
  DNUMBER      INTEGER    NOT NULL,
  MGRSSN       CHAR(9),
  MGRSTARTDATE CHAR(9)
);

Keys & Foreign Keys in CREATE TABLE

SQL allows specifying primary keys, secondary keys (UNIQUE), and foreign keys (referential integrity) inside CREATE TABLE.

  • PRIMARY KEY (attr) — designates the primary key attribute(s)
  • UNIQUE (attr) — designates a secondary/candidate key
  • FOREIGN KEY (attr) REFERENCES table — referential integrity constraint
CREATE TABLE with keys and foreign key:
CREATE TABLE DEPT (
  DNAME        VARCHAR(10) NOT NULL,
  DNUMBER      INTEGER    NOT NULL,
  MGRSSN       CHAR(9),
  MGRSTARTDATE CHAR(9),
  PRIMARY KEY (DNUMBER),
  UNIQUE (DNAME),
  FOREIGN KEY (MGRSSN) REFERENCES EMP
);

Referential Integrity Options

On a FOREIGN KEY declaration, we can specify what happens when the referenced tuple is deleted or updated:

RESTRICTPrevents the delete/update if a referencing tuple exists.
CASCADEAutomatically propagates the delete/update to referencing tuples.
SET NULLSets the foreign key attribute to NULL in referencing tuples.
SET DEFAULTSets the foreign key attribute to its default value.
DEPT table — ON DELETE SET DEFAULT, ON UPDATE CASCADE:
CREATE TABLE DEPT (
  DNAME        VARCHAR(10) NOT NULL,
  DNUMBER      INTEGER    NOT NULL,
  MGRSSN       CHAR(9),
  MGRSTARTDATE CHAR(9),
  PRIMARY KEY (DNUMBER),
  UNIQUE (DNAME),
  FOREIGN KEY (MGRSSN) REFERENCES EMP
    ON DELETE SET DEFAULT ON UPDATE CASCADE
);
EMP table — multiple foreign keys with different actions:
CREATE TABLE EMP (
  ENAME    VARCHAR(30) NOT NULL,
  ESSN     CHAR(9),
  BDATE    DATE,
  DNO      INTEGER DEFAULT 1,
  SUPERSSN CHAR(9),
  PRIMARY KEY (ESSN),
  FOREIGN KEY (DNO) REFERENCES DEPT
    ON DELETE SET DEFAULT ON UPDATE CASCADE,
  FOREIGN KEY (SUPERSSN) REFERENCES EMP
    ON DELETE SET NULL ON UPDATE CASCADE
);

DROP TABLE & ALTER TABLE

DROP TABLE

  • Removes a relation (base table) and its definition entirely.
  • The relation can no longer be used in queries, updates, or any other commands.
DROP TABLE DEPENDENT;

ALTER TABLE

  • Used to add a new attribute to an existing base relation.
  • The new attribute will have NULL in all existing tuples — therefore NOT NULL is not allowed for the new attribute.
  • Values for the new attribute must be entered via UPDATE.
ALTER TABLE EMPLOYEE ADD JOB VARCHAR(12);

Retrieval Queries – SELECT-FROM-WHERE Block

The basic form of a SQL query is called a mapping or SELECT-FROM-WHERE block:

SELECT <attribute list>
FROM <table list>
WHERE <condition>
SELECT clauseLists the attribute names whose values are to be retrieved (projection attributes).
FROM clauseLists the relation names required to process the query.
WHERE clauseA Boolean expression (condition) that identifies tuples to be retrieved (selection condition).
Basic SQL queries correspond to using the SELECT, PROJECT, and JOIN operations of relational algebra. However, the result may contain duplicate tuples — unlike pure relational algebra which treats relations as sets.

Example Queries

Q0 – Simple query on one relation (SELECT-PROJECT):

Retrieve the birthdate and address of the employee whose name is 'John B. Smith'.

SELECT BDATE, ADDRESS
FROM EMPLOYEE
WHERE FNAME='John' AND MINIT='B' AND LNAME='Smith'
Q1 – Query on multiple relations with join (SELECT-PROJECT-JOIN):

Retrieve the name and address of all employees who work for the 'Research' department.

SELECT FNAME, LNAME, ADDRESS
FROM EMPLOYEE, DEPARTMENT
WHERE DNAME='Research' AND DNUMBER=DNO
  • DNAME='Research' is a selection condition (SELECT operation in relational algebra).
  • DNUMBER=DNO is a join condition (JOIN operation in relational algebra).
Q2 – Two join conditions:

For every project in 'Stafford', list project number, controlling dept number, and the manager's last name, address, and birthdate.

SELECT PNUMBER, DNUM, LNAME, BDATE, ADDRESS
FROM PROJECT, DEPARTMENT, EMPLOYEE
WHERE DNUM=DNUMBER AND MGRSSN=SSN AND PLOCATION='Stafford'
  • DNUM=DNUMBER relates a project to its controlling department.
  • MGRSSN=SSN relates the controlling department to the employee who manages it.

Aliases, SELECT *, DISTINCT & Unspecified WHERE

Qualifying Attribute Names

When two or more relations in a query have attributes with the same name, the attribute must be qualified by prefixing the relation name:

EMPLOYEE.DNAME, DEPARTMENT.DNAME

Aliases (Tuple Variables)

When a query needs to reference the same relation twice, aliases (also called tuple variables) are assigned to the relation name. The keyword AS can also be used.

Q8 – Employee and their immediate supervisor (self-join):
SELECT E.FNAME, E.LNAME, S.FNAME, S.LNAME
FROM EMPLOYEE AS E, EMPLOYEE AS S
WHERE E.SUPERSSN = S.SSN

E represents employees in the role of supervisees; S represents employees in the role of supervisors.

SELECT * (All Attributes)

A * in the SELECT clause retrieves all attribute values of the selected tuples.

Q1C – All employees in dept 5:
SELECT *
FROM EMPLOYEE
WHERE DNO=5
Q1D – All Research dept employees:
SELECT *
FROM EMPLOYEE, DEPARTMENT
WHERE DNAME='Research'
  AND DNO=DNUMBER

DISTINCT – Eliminating Duplicates

SQL does not treat a relation as a set — duplicate tuples can appear. Use DISTINCT to eliminate them.

Q11 – May have duplicates:
SELECT SALARY
FROM EMPLOYEE
Q11A – No duplicates:
SELECT DISTINCT SALARY
FROM EMPLOYEE

Unspecified WHERE Clause

A missing WHERE clause means no condition — equivalent to WHERE TRUE. All tuples from the FROM clause are selected.

Q9 – Retrieve all employee SSNs:
SELECT SSN FROM EMPLOYEE
⚠ Warning: If more than one relation is in the FROM clause and there is no join condition, the result is the CARTESIAN PRODUCT of all tuples — potentially an enormous and incorrect result. Always specify all necessary join conditions.

Set Operations

SQL incorporates set operations that work on union-compatible relations (same attributes in the same order). Duplicates are automatically eliminated from the result.

UNIONCombines results of two queries, returning all distinct tuples from either.
INTERSECTReturns only tuples that appear in both query results. (Available in some SQL versions.)
MINUS / EXCEPTReturns tuples in the first query that are not in the second. (Some SQL versions.)
Q4 – All projects involving an employee named 'Smith' (as worker OR as department manager):
(SELECT PNAME
FROM PROJECT, DEPARTMENT, EMPLOYEE
WHERE DNUM=DNUMBER AND MGRSSN=SSN AND LNAME='Smith')
UNION
(SELECT PNAME
FROM PROJECT, WORKS_ON, EMPLOYEE
WHERE PNUMBER=PNO AND ESSN=SSN AND LNAME='Smith')

Nesting of Queries (Subqueries)

A complete SELECT query — called a nested query — can be specified inside the WHERE clause of another query (the outer query).

  • The IN operator compares a value v with a set of values V, evaluating to TRUE if v is one of the elements in V.
  • Multiple levels of nesting are possible.
  • An unqualified attribute refers to the relation declared in the innermost nested query.
Q1 (nested version) – Employees in the 'Research' department:
SELECT FNAME, LNAME, ADDRESS
FROM EMPLOYEE
WHERE DNO IN (SELECT DNUMBER
                FROM DEPARTMENT
                WHERE DNAME='Research')

Correlated Nested Queries

A nested query is correlated when its WHERE clause references an attribute of a relation declared in the outer query — meaning it is re-evaluated for each tuple of the outer query.

Q12 – Employees who have a dependent with the same first name:
SELECT E.FNAME, E.LNAME
FROM EMPLOYEE AS E
WHERE E.SSN IN (SELECT ESSN
                  FROM DEPENDENT
                  WHERE ESSN=E.SSN AND E.FNAME=DEPENDENT_NAME)
A correlated nested query using = or IN can always be rewritten as a single-block query using joins. For example, Q12 is equivalent to:
Q12A – Single-block equivalent:
SELECT E.FNAME, E.LNAME
FROM EMPLOYEE E, DEPENDENT D
WHERE E.SSN=D.ESSN AND E.FNAME=D.DEPENDENT_NAME

CONTAINS Operator (not widely supported)

The CONTAINS operator compares two sets of values and returns TRUE if one set contains all values in the other (reminiscent of the division operation in relational algebra). Most SQL implementations do not support this operator.

Q3 – Employees who work on ALL projects controlled by dept 5:
SELECT FNAME, LNAME
FROM EMPLOYEE
WHERE (SELECT PNO FROM WORKS_ON WHERE SSN=ESSN)
       CONTAINS
       (SELECT PNUMBER FROM PROJECT WHERE DNUM=5)

NULLs in SQL Queries

NULL means a value is missing, undefined, or not applicable. SQL uses IS NULL / IS NOT NULL rather than = to check for NULLs, because each NULL value is considered distinct from every other NULL value — so equality comparison is not appropriate.

Q14 – Employees without a supervisor:
SELECT FNAME, LNAME
FROM EMPLOYEE
WHERE SUPERSSN IS NULL
Note: If a join condition is specified, tuples with NULL values for the join attributes are not included in the result.

Joined Relations Feature in SQL2

SQL2 introduced a cleaner syntax to express joins directly in the FROM clause using the JOIN ... ON construct, including outer joins.

INNER JOIN ... ONReturns only matching tuples from both relations.
LEFT OUTER JOIN ... ONIncludes all tuples from the left relation; unmatched right tuples get NULL.
NATURAL JOINAutomatically joins on attributes with the same name in both relations.
Q8 rewritten using LEFT OUTER JOIN:
SELECT E.FNAME, E.LNAME, S.FNAME, S.LNAME
FROM (EMPLOYEE E LEFT OUTER JOIN EMPLOYEE S
        ON E.SUPERSSN = S.SSN)
Q1 using JOIN ... ON:
SELECT FNAME, LNAME, ADDRESS
FROM (EMPLOYEE JOIN DEPARTMENT ON DNUMBER=DNO)
WHERE DNAME='Research'
Q2 using multiple joins (chained):
SELECT PNUMBER, DNUM, LNAME, BDATE, ADDRESS
FROM ((PROJECT JOIN DEPARTMENT ON DNUM=DNUMBER)
       JOIN EMPLOYEE ON MGRSSN=SSN)
WHERE PLOCATION='Stafford'

Aggregate Functions

SQL provides built-in aggregate functions that compute a single value from a set of tuples:

COUNT(*)Counts the total number of tuples.
SUM(attr)Computes the sum of attribute values.
MAX(attr)Finds the maximum value.
MIN(attr)Finds the minimum value.
AVG(attr)Computes the average value.
Some SQL implementations may not allow more than one aggregate function in the SELECT clause.
Q15 – Max, min, and average salary for all employees:
SELECT MAX(SALARY), MIN(SALARY), AVG(SALARY)
FROM EMPLOYEE
Q16 – Max, min, and average salary in the 'Research' department:
SELECT MAX(SALARY), MIN(SALARY), AVG(SALARY)
FROM EMPLOYEE, DEPARTMENT
WHERE DNO=DNUMBER AND DNAME='Research'
Q17 / Q18 – COUNT examples:
Q17 – Total employees:
SELECT COUNT(*)
FROM EMPLOYEE
Q18 – Research dept count:
SELECT COUNT(*)
FROM EMPLOYEE, DEPARTMENT
WHERE DNO=DNUMBER
  AND DNAME='Research'

Grouping: GROUP BY & HAVING

GROUP BY

Used to apply aggregate functions to subgroups of tuples. Each subgroup consists of tuples that share the same value for the grouping attribute(s). The grouping attributes must also appear in the SELECT clause.

Q20 – Per department: number of employees and average salary:
SELECT DNO, COUNT(*), AVG(SALARY)
FROM EMPLOYEE
GROUP BY DNO

EMPLOYEE tuples are divided into groups — each group shares the same DNO value. COUNT and AVG are then applied to each group independently.

Q21 – Per project: project number, name, and employee count (join + group):
SELECT PNUMBER, PNAME, COUNT(*)
FROM PROJECT, WORKS_ON
WHERE PNUMBER=PNO
GROUP BY PNUMBER, PNAME

Grouping and functions are applied after the join of the two relations.

HAVING

The HAVING clause specifies a selection condition on groups (rather than on individual tuples). It is used together with GROUP BY to filter out groups that do not meet a condition.

Q22 – Projects with more than 2 employees working on them:
SELECT PNUMBER, PNAME, COUNT(*)
FROM PROJECT, WORKS_ON
WHERE PNUMBER=PNO
GROUP BY PNUMBER, PNAME
HAVING COUNT(*) > 2

Substring Comparison – LIKE Operator

The LIKE operator is used for partial string matching. Two wildcard characters are used:

%Replaces an arbitrary number of characters (zero or more). Also written as * in some implementations.
_Replaces exactly one arbitrary character.
Q25 – Employees whose address contains 'Houston,TX':
SELECT FNAME, LNAME
FROM EMPLOYEE
WHERE ADDRESS LIKE '%Houston,TX%'
Q26 – Employees born in the 1950s (8th character of BDATE is '5'):
SELECT FNAME, LNAME
FROM EMPLOYEE
WHERE BDATE LIKE '_______5_'
The LIKE operator allows treating character string values as non-atomic, enabling partial matching inside strings.

Arithmetic Operations & ORDER BY

Arithmetic Operations

Standard arithmetic operators +, -, *, / can be applied to numeric attribute values in a query result.

Q27 – Show effect of a 10% salary raise for ProductX project employees:
SELECT FNAME, LNAME, 1.1 * SALARY
FROM EMPLOYEE, WORKS_ON, PROJECT
WHERE SSN=ESSN AND PNO=PNUMBER AND PNAME='ProductX'

ORDER BY

Sorts the tuples in a query result based on the values of one or more attributes. Default order is ascending (ASC). Use DESC for descending order.

Q28 – Employees and their projects, ordered by dept name, then last name:
SELECT DNAME, LNAME, FNAME, PNAME
FROM DEPARTMENT, EMPLOYEE, WORKS_ON, PROJECT
WHERE DNUMBER=DNO AND SSN=ESSN AND PNO=PNUMBER
ORDER BY DNAME, LNAME

Summary of SQL Query Structure

A SQL query can consist of up to six clauses. Only SELECT and FROM are mandatory.

SELECT <attribute list>
FROM <table list>
[WHERE <condition>]
[GROUP BY <grouping attribute(s)>]
[HAVING <group condition>]
[ORDER BY <attribute list>]
SELECTLists attributes or aggregate functions to be retrieved.
FROMSpecifies all relations (or aliases) needed in the query (not nested queries).
WHERESelection and join conditions on individual tuples.
GROUP BYSpecifies the grouping attribute(s).
HAVINGCondition for selecting among groups.
ORDER BYSorting order for the result.
Evaluation Order: A query is evaluated by first applying the WHERE clause, then GROUP BY and HAVING, and finally the SELECT clause.

Specifying Updates – INSERT

INSERT adds one or more tuples to a relation. Attribute values should be listed in the same order they were defined in CREATE TABLE.

Full INSERTAll attribute values provided in order.
Partial INSERTAttribute names specified explicitly; omitted attributes receive NULL.
Bulk INSERTInsert multiple tuples from a SELECT query result.
U1 – Full insert:
INSERT INTO EMPLOYEE
VALUES ('Richard','K','Marini','653298653','1962-12-30',
       '98 Oak Forest,Katy,TX','M',37000,'987654321',4);
U1A – Partial insert (only FNAME, LNAME, SSN):
INSERT INTO EMPLOYEE (FNAME, LNAME, SSN)
VALUES ('Richard', 'Marini', '653298653');
U3A + U3B – Create and populate a summary table using bulk INSERT:
CREATE TABLE DEPTS_INFO (
  DEPT_NAME VARCHAR(10),
  NO_OF_EMPS INTEGER,
  TOTAL_SAL  INTEGER
);

INSERT INTO DEPTS_INFO (DEPT_NAME, NO_OF_EMPS, TOTAL_SAL)
  SELECT DNAME, COUNT(*), SUM(SALARY)
  FROM DEPARTMENT, EMPLOYEE
  WHERE DNUMBER=DNO
  GROUP BY DNAME;
Note: DEPTS_INFO will not stay up-to-date automatically if DEPARTMENT or EMPLOYEE are later modified. A view is needed to keep such a derived table current.
Important: Only constraints specified in DDL commands are automatically enforced by the DBMS when updates are applied.

DELETE

Removes tuples from a relation. Key behaviors:

  • A WHERE clause selects tuples to delete.
  • Tuples are deleted from only one table at a time (unless CASCADE is specified on a referential integrity constraint).
  • A missing WHERE clause deletes all tuples — the table becomes empty.
  • The number of deleted tuples depends on how many satisfy the WHERE condition.
  • Referential integrity should be enforced.
Delete by last name:
DELETE FROM EMPLOYEE WHERE LNAME='Brown'
Delete by SSN:
DELETE FROM EMPLOYEE WHERE SSN='123456789'
Delete using nested query (delete all Research dept employees):
DELETE FROM EMPLOYEE
WHERE DNO IN (SELECT DNUMBER
               FROM DEPARTMENT
               WHERE DNAME='Research')
Delete all tuples (empty the table):
DELETE FROM EMPLOYEE

UPDATE

Modifies attribute values of selected tuples in a relation. Key behaviors:

  • A WHERE clause selects the tuples to be modified.
  • A SET clause specifies the attributes to modify and their new values.
  • Each UPDATE command modifies tuples in the same relation.
  • Referential integrity should be enforced.
U5 – Change location and dept number of project 10:
UPDATE PROJECT
SET PLOCATION = 'Bellaire', DNUM = 5
WHERE PNUMBER = 10
U6 – Give all Research dept employees a 10% salary raise:
UPDATE EMPLOYEE
SET SALARY = SALARY * 1.1
WHERE DNO IN (SELECT DNUMBER
                FROM DEPARTMENT
                WHERE DNAME='Research')
In the SET clause, the SALARY attribute on the right of = refers to the old value before modification; the one on the left refers to the new value after modification.
