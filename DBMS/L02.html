📘 Database Management Systems

Lecture 02 — Database System Concepts and Architectures

Basic Definitions - Review

Database: A collection of related data.
Data: Known facts that can be recorded and have an implicit meaning.
Mini-world: Some part of the real world about which data is stored in a database.
Example: Student grades and transcripts at a university.
Database Management System (DBMS): A software package/system to facilitate the creation and maintenance of a computerized database.
Database System: The DBMS software together with the data itself. Sometimes, the applications are also included.
RDBMS: A program used to create, update, and manage relational databases.

Typical DBMS Functionality

Define a Database In terms of data types, structures and constraints
Construct or Load Database on a secondary storage medium
Manipulate the Database Querying, reports, insertions, deletions, modifications
Concurrent Processing Multiple users and programs, keeping data valid and consistent
Protection/Security Prevent unauthorized access
Presentation & Visualization Display data in meaningful formats

Example of a Database - UNIVERSITY

Mini-world: Part of a UNIVERSITY environment
Mini-world Entities:
STUDENTS Student records
COURSES Course offerings
SUBJECTS Course subjects
DEPARTMENTS Academic departments
LECTURERS Teaching staff
Mini-world Relationships:
  • SUBJECTS are of specific COURSES
  • STUDENTS take SUBJECTS
  • LECTURERS teach SUBJECTS
  • COURSES are offered by DEPARTMENTS
  • STUDENTS major in DEPARTMENTS

Data Models

Data Model: A set of concepts to describe the structure of a database, the operations for manipulating these structures, and certain constraints that the database should obey.
Data Model Components:
Structure Elements, data types, groups of elements (entities, records, tables)
Operations Retrievals, updates, manipulations
Constraints Restrictions on valid data

Data Model Structure and Constraints

  • Constructs are used to define the database structure
  • Constructs typically include:
    • Elements and their data types
    • Groups of elements (e.g., entity, record, table)
    • Relationships among such groups
  • Constraints specify restrictions on valid data
  • These constraints must be enforced at all times

Data Model Operations

  • Operations are used for specifying database retrievals and updates
  • Refer to the constructs of the data model
  • Types of operations:
    • Basic model operations: Generic insert, delete, update
    • User-defined operations: Custom operations for specific needs
Examples of User-Defined Operations:
Calculate Final Marks:
10% Quizzes + 20% Assessments + 70% Final Exam
Calculate GPA:
Sum(GPV of the Result × No. of Credits) / Sum(Credits)

Categories of Data Models

Conceptual or High-Level Data Models • Provide concepts such as entities, attributes, and relationships • Close to the way many users perceive data
• Also called entity-based or object-based data models
• Used during database design phase

Physical or Low-Level Data Models • Describe details of how data is stored in the computer
• Concepts for computer specialists, not for end users
• Describe how data is stored as files
• Include information such as record formats, record orderings, and access paths

Implementation or Representational Data Models • Represent data by using record structures (record-based data models)
• Concepts easily understood by end users
• Hide many details of data storage
• Can be implemented on computer system directly
• Include the widely used relational data models

Self-Describing Data Models • Combine the description of data with the data values • Examples: XML, key-value stores, some NOSQL systems
• Created for managing big data

Database Schema

Database Schema: The description of a database. Includes descriptions of the database structure, data types, and the constraints on the database.
Schema Diagram: An illustrative display of (most aspects of) a database schema.
Schema Construct: A component of the schema or an object within the schema.
Examples: STUDENT, COURSE
Key Point: The schema is the blueprint or design of the database.

Database State

Database State (Snapshot): The actual data stored in a database at a particular moment in time.
  • Includes the collection of all the data in the database
  • Also called the current set of occurrences or instances in the database
  • The term instance is also applied to individual database components
    • Examples: record instance, table instance, entity instance
  • Every time we insert/delete/change a data item, we change one state into another state

Types of Database States

Empty State Database when it is first created (before loading any data)
Initial Database State Database when it is initially loaded into the system
Current State The state at the present moment
Valid State A state that satisfies the structure and constraints of the database
State Progression

Empty State → Initial State → Current State
(All states must be Valid States)

Database Schema vs. Database State

Schema (Intension) • The structure/design
• Changes very infrequently • Defines what data can be stored
• Like a blueprint
State (Extension) • The actual data
• Changes every time database is updated • The current content
• Like actual data in tables
Analogy: Schema is like a house blueprint (rarely changes), while State is like the furniture and people inside (changes frequently).

Three-Schema Architecture

Purpose: Proposed to support DBMS characteristics of:
  • Program-data independence
  • Support of multiple views of the data
Goal: To separate the user applications from the physical database.
Note: Not explicitly used in commercial DBMS products, but has been useful in explaining database system organization.

The Three Schema Levels

Internal Schema (Internal Level) • Describes physical storage structures and access paths (e.g., indexes)
• Typically uses a physical data model • Closest to physical storage
• Concerned with how data is stored
Conceptual Schema (Conceptual Level) • Describes the structure and constraints for the whole database • For a community of users • Uses a conceptual or implementation data model
• Hides details of physical storage
• Describes what data is stored and relationships
External Schemas (External Level) • Describes the various user views • Each external schema describes the part of database that a particular user group is interested in
• Hides the rest of the database from that user group
• Usually uses the same data model as conceptual schema
• Multiple external schemas possible
Three-Schema Architecture Hierarchy

External Level (User Views)
↕ Mapping
Conceptual Level (Logical Structure)
↕ Mapping
Internal Level (Physical Storage)

Mappings in Three-Schema Architecture

Mappings: The processes of transforming requests and results between schema levels.
  • Programs refer to an external schema
  • DBMS maps the request to the internal schema for execution
  • Data extracted from internal DBMS level is reformatted to match the user's external view
Example: Formatting the results of an SQL query for display in a Web page

Data Independence

Data Independence: Capacity to change the schema at one level of a database system without having to change the schema at the next higher level.
Logical Data Independence • Capacity to change the conceptual schema • Without changing external schemas and application programs • Allows changes to logical structure without affecting user views
Physical Data Independence • Capacity to change the internal schema • Without changing the conceptual schema • Allows changes to storage without affecting logical structure
Benefit: Data independence provides flexibility and reduces maintenance costs.

DBMS Languages

Data Definition Language (DDL) Defines database structure and constraints
Data Manipulation Language (DML) Manipulates data (query, insert, update, delete)
DML Types:
High-Level DML Non-procedural languages (e.g., SQL)
Low-Level DML Procedural languages

Data Definition Language (DDL)

  • Used by the DBA and database designers
  • Specify the conceptual schema of a database
  • In many DBMSs, DDL is also used to define internal and external schemas (views)
  • Some DBMSs use separate languages:
    • Storage Definition Language (SDL): Define internal schemas
    • View Definition Language (VDL): Specify user views and their mappings to conceptual schema
Note: SDL is typically realized via DBMS commands provided to the DBA and database designers.

Data Manipulation Language (DML)

  • Used to specify database manipulations:
    • Retrieval
    • Insertion
    • Deletion
    • Modification of data
  • DML commands (data sublanguage) can be embedded in a general-purpose programming language (host language)
    • Examples: COBOL, C, C++, Java
  • A library of functions can also be provided to access the DBMS
  • Alternatively, stand-alone DML commands can be applied directly (called a query language)

Types of DML

High Level or Non-Procedural Language • Include the relational language SQL • May be used in a standalone way or embedded in a programming language
• Can specify and retrieve many records in a single DML statement • Called set-at-a-time or set-oriented DMLs • Specifies which data to retrieve rather than how to retrieve it
Low Level or Procedural Language • Retrieve data one record-at-a-time • Must be embedded in a general-purpose programming language • Retrieves individual records or objects from the database
• Processes each separately
• More complex to use but offers more control

DBMS Interfaces

Stand-alone Query Language Entering SQL queries at DBMS interactive interface (e.g., SQL*Plus in ORACLE)
Programmer Interfaces For embedding DML in programming languages
User-Friendly Interfaces Menu-based, forms-based, graphics-based
Mobile Interfaces Interfaces for performing transactions using mobile apps

DBMS Programming Language Interfaces

Embedded Approach Examples: Embedded SQL (for C, C++, etc.), SQLJ (for Java)
Procedure Call Approach Examples: JDBC for Java, ODBC (Open Database Connectivity) for other programming languages as APIs (Application Programming Interfaces)
Database Programming Language Approach Example: ORACLE has PL/SQL, a programming language based on SQL; language incorporates SQL and its data types as integral components
Scripting Languages PHP (client-side scripting) and Python (server-side scripting) are used to write database programs

User-Friendly DBMS Interfaces

  • Menu-based (Web-based): Popular for browsing on the web
  • Forms-based: Designed for naïve users used to filling in entries on a form
  • Graphics-based:
    • Point and Click, Drag and Drop
    • Specifying a query on a schema diagram
  • Natural language: Requests in written English
  • Combinations: Menus and forms used extensively in Web database interfaces

Other DBMS Interfaces

Natural Language Free text as a query
Speech Input query and output response
Web Browser With keyword search
Parametric Interfaces Example: Bank tellers using function keys for routine transactions
DBA Interfaces Creating accounts, granting authorizations, setting parameters, changing schemas

Database System Utilities

Functions performed by utilities:
Loading Loading data stored in files into a database (includes data conversion tools)
Backup Backing up the database periodically on tape or cloud
Reorganization Reorganizing database file structures
Performance Monitoring Performance monitoring utilities
Report Generation Report generation utilities
Other Functions Sorting, user monitoring, data compression, etc.

Centralized DBMS Architecture

Centralized DBMS: Combines everything into single system including DBMS software, hardware, application programs, and user interface processing software.
  • All processing is done at centralized site
  • User can still connect through a remote terminal
  • All data stored in one location
  • Simpler architecture but single point of failure

Client-Server Architecture

Specialized Servers with Specialized Functions:
Print Server Handles printing tasks
File Server Manages file storage
DBMS Server Database management
Web Server Web content delivery
Email Server Email services
Key Point: Clients can access the specialized servers as they need.

Clients in Client-Server Architecture

  • Provide appropriate interfaces through a client software module to access and utilize various server resources
  • Clients may be:
    • Diskless machines
    • PCs or Workstations with disks with only the client software installed
  • Connected to servers via some form of a network
    • LAN: Local Area Network
    • Wireless network
    • Internet

DBMS Server

  • Provides database query and transaction services to the clients
  • Relational DBMS servers are often called:
    • SQL servers
    • Query servers
    • Transaction servers
  • Applications running on clients utilize an Application Program Interface (API) to access server databases via standard interfaces:
    • ODBC: Open Database Connectivity standard
    • JDBC: For Java programming access

Two-Tier Client-Server Architecture

  • Client and server must install appropriate client module and server module software for ODBC or JDBC
  • A client program may connect to several DBMSs, sometimes called the data sources
  • In general, data sources can be:
    • Files
    • Other non-DBMS software that manages data
Two-Tier Architecture

Client (User Interface + Application Logic)
↕
Server (Database Server)

Three-Tier Client-Server Architecture

Common for Web Applications
Intermediate Layer - Application Server or Web Server • Stores the web connectivity software • Stores the business logic part of the application • Used to access the corresponding data from the database server
Enhanced Security Benefits:
  • Database server only accessible via middle tier
  • Clients cannot directly access database server
  • Clients contain user interfaces and Web browsers
  • The client is typically a PC or mobile device connected to the Web
Three-Tier Architecture

Tier 1: Client (User Interface)
↕
Tier 2: Application Server (Business Logic)
↕
Tier 3: Database Server (Data Management)

Classification of DBMSs

Based on the Data Model Used:
Legacy Network, Hierarchical
Currently Used Relational, Object-oriented, Object-relational
Recent Technologies Key-value storage, NOSQL systems
NOSQL Systems: Document-based, Column-based, Graph-based, Key-value based
Other Classifications:
Single-user vs Multi-user • Single-user: Typically used with personal computers
• Multi-user: Most DBMSs
Centralized vs Distributed • Centralized: Single computer with one database
• Distributed: Multiple computers with multiple databases

History of Data Models - Network Model

Self-Study Topic
  • First network DBMS implemented by Honeywell in 1964-65 (IDS System)
  • Adopted heavily due to support by CODASYL (Conference on Data Systems Languages) - CODASYL-DBTG report of 1971
  • Implemented in various systems:
    • IDMS (Cullinet - now Computer Associates)
    • DMS 1100 (Unisys)
    • IMAGE (H.P.)
    • VAX-DBMS (Digital Equipment Corp.)
Advantages • Model complex relationships
• Represents semantics of add/delete
• Handle most situations
• Optimal navigation through database
Disadvantages • Navigational and procedural nature
• Complex array of pointers
• Little scope for automated query optimization

History of Data Models - Hierarchical Model

  • Initially implemented in joint effort by IBM and North American Rockwell around 1965
  • Resulted in the IMS family of systems
  • IBM's IMS product had (and still has) a very large customer base worldwide
  • Other systems: System 2k (SAS inc.)
Advantages • Simple to construct and operate
• Corresponds to natural hierarchically organized domains (e.g., org chart)
• Simple language
Disadvantages • Navigational and procedural nature
• Database visualized as linear arrangement
• Little scope for query optimization

History of Data Models - Relational Model

  • Proposed in 1970 by E.F. Codd (IBM)
  • First commercial system in 1981-82
  • Available in several commercial products:
    • DB2, ORACLE, MS SQL Server, SYBASE, INFORMIX
  • Several free open-source implementations:
    • MySQL, PostgreSQL
  • Currently most dominant for developing database applications
  • SQL relational standards:
    • SQL-89 (SQL1), SQL-92 (SQL2), SQL-99, SQL3, ...

History of Data Models - Object-Oriented

  • Several models proposed for implementing in a database system
  • Persistent O-O Programming Languages:
    • C++ (e.g., in OBJECTSTORE or VERSANT)
    • Smalltalk (e.g., in GEMSTONE)
  • Other systems: O2, ORION (at MCC - then ITASCA), IRIS (at H.P. - used in Open OODB)
  • Object Database Standards:
    • ODMG-93, ODMG-version 2.0, ODMG-version 3.0

History of Data Models - Object-Relational

  • Trend to mix object models with relational started with Informix Universal Server
  • Relational systems incorporated concepts from object databases leading to object-relational
  • Exemplified in versions of Oracle, DB2, and SQL Server
  • Current trend: Relational DBMS vendors extend relational DBMSs with capability to process:
    • XML
    • Text
    • Other data types
  • The term "Object-relational" is receding in the marketplace

Key Takeaways

Remember:
  • Data models provide concepts to describe database structure, operations, and constraints
  • Schema is the database design (changes rarely), State is the current data (changes frequently)
  • Three-schema architecture provides data independence through internal, conceptual, and external levels
  • DDL defines structure, DML manipulates data
  • Two-tier architecture is simple, Three-tier adds security and business logic layer
  • Database models evolved from Hierarchical/Network → Relational → Object-Oriented → Object-Relational → NoSQL
