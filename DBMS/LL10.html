🗄️ Disk Storage & File Structures

ICT1212 – Database Management Systems | Lecture 10
📀 Classification of Physical Storage Media

Key criteria: speed, cost per unit, reliability (data loss on power failure / physical failure).

⚡ Volatile Storage
  • Loses contents when power is switched off.
  • Examples: cache, main memory.
🛡️ Non-Volatile Storage
  • Persists even after power loss.
  • Secondary & Tertiary storage, battery-backed RAM.
📈 Storage Hierarchy (fast→slow / costly→cheap):
Primary (cache, main memory – volatile, fastest) → Secondary (flash, magnetic disk – non‑volatile, online) → Tertiary (tape, optical – offline, slow).
📦 Physical Storage Media – In Depth
🧠 Cache
Fastest, costly, volatile, hardware-managed.
💾 Main Memory
Fast, volatile, too small for full DB, general-purpose.
⚡ Flash Memory
Non‑volatile, reads ~RAM speed, writes slower, erase slowest. Used in cameras, phones. NAND flash cheaper than NOR.
💽 Magnetic Disk
Primary long-term storage; direct access; non‑volatile; large capacity; data moved to RAM for processing.
💿 Optical Storage
CD/DVD/Blu-ray, non‑volatile, slower than magnetic disks, archival (R/RW).
📼 Tape Storage
Sequential access, very high capacity, cheap per byte but expensive drives, used for backup/archival.
🧲 Magnetic Disks – Architecture & Performance

Key components: Read‑write head (near platter surface), platters divided into tracks → sectors (smallest read/write unit). Cylinder = same track on all platters.
Disk controller: computes checksums, verifies writes, remaps bad sectors.

⏱️ Performance measures
  • Access time = seek time + rotational latency.
  • Seek time: reposition arm over correct track.
  • Rotational latency: wait for sector under head.
  • Data-transfer rate: speed of reading/writing.
  • MTTF (mean time to failure): 3–5 years typical.
🔁 Block & Scheduling
  • Block = contiguous sectors (4–16 KB typical).
  • Disk-arm scheduling minimizes movement.
  • File organization: place related blocks on same cylinder.
  • Defragmentation to reduce scattered blocks.
💡 Optimization: Nonvolatile write buffers (NV‑RAM) & Log disk (sequential log) reorder writes to minimize seeks.

Disk interfaces: ATA (PATA, SATA), SCSI, SAS. Connection types: DAS (direct attach), NAS (network), SAN (storage area networks).

🔁 RAID: Redundant Arrays of Independent Disks

Purpose: combine multiple disks → single logical disk with high capacity, parallelism (speed), and redundancy (reliability). Originally “inexpensive”, now “independent”.

RAID 0
Striping, no fault tolerance. Max performance, unsafe.
RAID 1
Mirroring (duplexing). High reliability, 2x disks.
RAID 5
Block-interleaved distributed parity. Popular, good balance.
RAID 6
Double parity, tolerates 2 disk failures.

Levels 2, 3, 4 are rarely used today (subsumed, obsolete). Level 3 (bit-interleaved parity) no longer common. RAID 0 used only when data safety not critical.

🔧 Hardware vs Software RAID:
• Software RAID: implemented in OS, no special hardware.
• Hardware RAID: dedicated controller + non‑volatile RAM. Features: latent failure detection, data scrubbing, hot swapping, spare disks.
📌 RAID ensures that a single point of failure does not stop system functioning; spare disks can be swapped online.
📂 File Organization & Record Structures

Database = collection of files → each file = sequence of records → each record = sequence of fields.
Two main approaches: fixed-length records & variable-length records.

📏 Fixed-Length Records
Simpler, each record same size. Deletion strategies:
1) compact (shift records)
2) move last record into deleted slot (non‑sequential order).
📐 Variable-Length Records
From: multiple record types, VARCHAR fields, repeating fields. Represented via (offset, length) pairs, null‑value bitmap. Slotted page structure: header tracks #entries, free space, record locations → records can move internally while pointers reference header entry.
🗂️ Slotted Page Structure – header contains number of records, end of free space, and array of (location, size). Prevents fragmentation inside block; record pointers point to header slots, not direct offsets.

Organization methods inside files:

  • Heap (unordered): any free space, simple insertion.
  • Sequential: sorted by search key, good for range scans.
  • Hashing: hash function on attribute determines block placement.
📚 Data Dictionary Storage (System Catalog)

Stores metadata about the database:

  • Relation names & attributes (names, types, lengths)
  • View definitions & integrity constraints
  • User accounts, passwords, privileges
  • Statistical data (cardinalities, tuple counts)
  • Physical organization (heap/hash/sequential)
  • Physical file location & index metadata
📌 Represented relationally inside the DBMS – queried like regular tables.
⛁ Storage Access & Buffer Management

Block: fixed-length storage unit (file partition). Blocks are transfer units between disk & memory. Goal: minimize block transfers.

Buffer Manager
Subsystem that allocates main memory space (buffer pool) to keep copies of disk blocks, reducing I/O.
Blocking Factor (bfr)
Number of records per block. Integral fit may leave unused space. Spanned records: when record > block size, crosses multiple blocks.
✔️ Keeping frequently used blocks in buffer dramatically improves performance; replacement policies (LRU, etc.) used.
🧮 Exercise Concepts (Disk Calculations)

Classic storage exercises help understand capacity, rotational latency, transfer rates, and file placement.

Track capacity
(sectors per track) × (sector size). Surface = tracks × track capacity. Disk capacity = surfaces × surface capacity.
Cylinders
Number of tracks per surface (same as cylinders).
Rotational delay
Max = 1 / (rpm/60) seconds. Transfer rate: bytes per revolution × revs per sec.
Blocking calculations
Block size ÷ record size (floor) = records per block. Sequential read: total blocks × (seek+rotational+transfer). Random access: each block incurs full access time.

Example (from lecture): disk 4KB sectors, 1000 tracks/surface, 7200 rpm, file with 150k records of 100B → 40 records per block → 3750 blocks → cylinders needed depend on tracks/cylinder & blocks per track.

⚡ Flash Storage & Advanced Optimization

NAND flash dominates storage (SSDs). Erase is slow, but reads fast. Solid-state disks use multiple flash devices to reach 100–200 MB/sec transfer rates.

Log disk approach: dedicated disk used as sequential log of block updates – avoids seeks entirely. No special hardware needed, improves write performance. File systems commonly reorder writes to minimize arm movement.

Nonvolatile write buffers: battery-backed RAM or flash memory – immediately absorbs writes, later reorders writes to disk. Speeds up transaction commits.

📌 Tertiary storage (tape/optical jukeboxes) used for massive off‑line archives, high capacity but slow sequential access.
📌 Core Summary & Principles
  • Storage media trade-off: speed vs cost vs volatility vs reliability.
  • Magnetic disks: direct access, cylinder/track/sector, block transfers, MTTF, scheduling to reduce seek time.
  • RAID levels 0,1,5,6 – trade performance, capacity, redundancy; RAID 5 most common for safety.
  • File organization – heap, sequential, hashing; each suited for different access patterns.
  • Record representation – fixed length vs variable (slotted pages essential for flexibility).
  • Buffer manager reduces I/O by caching blocks; critical for DB performance.
  • System catalog stores all metadata required for query processing and storage management.
💡 Modern DBMS combine SSDs, disk arrays with RAID, careful file organization to balance speed and durability.
🛠️ File Fragmentation & Defragmentation

Files become fragmented when blocks scattered across disk due to inserts/deletes. Sequential access to fragmented files causes extra disk arm movement → utilities defragment by rearranging blocks contiguously.

Disk-arm-scheduling algorithms: reorder pending accesses to minimize arm travel (e.g., elevator algorithm).

Head crash
Earlier disks were susceptible if head touched platter. Modern drives more robust, but still risk.
Hot swapping
Replace failed disk while system online, often with spare disks to rebuild immediately.
🗃️ Metadata Storage in Practice

Information about relations, indices, and physical locations is stored as relational tables inside the database (system tables). The DBMS reads the catalog at runtime to parse queries, enforce constraints, and optimize access paths.

Example catalog tables: sys_tables, sys_columns, sys_indexes. They use the same storage manager as user data.

Maintaining accurate statistical data (cardinalities) helps the query optimizer choose efficient execution plans.
